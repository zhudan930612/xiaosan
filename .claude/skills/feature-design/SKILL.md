# 功能设计方案输出

## 触发条件

用户说：我要做XX功能、优化XX功能、设计XX功能、功能设计。

## 执行流程

### 步骤 1：加载上下文包（包 B 或 C）

自动读取：
- `01_产品主文档/*产品上下文.md`
- `01_产品主文档/*产品规划.md`
- `02_已上线功能描述/*模块.md`（对应模块）

若涉及备案业务，额外读取 `06_附件资料/备案资料清单.md`。

向用户报告已读取的参考文档。

### 步骤 2：自主产出设计方案

基于模板输出完整方案（业务流程、规则说明、页面清单）。
- 信息不足时标注 `⚠️ 待确认`，不中断执行
- 若处于方案比选阶段，优先输出方案对比和推荐意见

### 步骤 3：自动执行自检清单

对照 `00_项目说明/AI自检清单.md` 中的【清单一：功能设计文档自检】检查。

### 步骤 4：交付

输出完整文档，汇报自检结果、待确认项和关键决策。

## 如需输出原型

用户明确要求或方案已确认时：
1. 读取原型设计标准文档 + 对应功能设计文档
2. 确认原型类型（若用户未指定，主动提供选项）
3. 自主产出 HTML 原型，存放到 `04_原型设计文件/`
4. 执行【清单三：原型设计文档自检】
5. 交付原型，汇报自检结果

## 功能设计文档约束

1. 文件开头标注 `🤖 AI草稿` 或 `✅ 人工确认`
2. 每项设计结论注明信息来源（截图/产品上下文/访谈）
3. 无法溯源的内容标注 `⚠️ 待确认`
4. 文档末尾附"待确认事项"章节
5. 须经 PM 人工审核，标注 `✅ 已确认，可开发` 方可转交开发
6. 发现待解决问题同步写入产品规划对应模块的"待解决问题"章节
7. 页面设计和原型设计不写入正文，单独生成 HTML 存于 `04_原型设计文件/`
8. 必须包含决策记录章节
9. 原型无标注时，功能设计必须兜底补充完整规则

## 格式规范

- 结构性内容（信息架构、方案对比、流程图）：Markdown
- 规则性内容（需求说明、业务规则、字段规则、校验逻辑）：纯文本
- 判断标准：是否需要被复制粘贴给开发？是则用纯文本

## 问答式备选

若用户说"我想讨论一下..."或信息严重不足，退回问答式。
