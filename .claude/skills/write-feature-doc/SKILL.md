# 功能描述文档编写技能

## 触发词

- 编写功能文档
- 功能描述
- 记录功能
- 文档化功能
- 写功能文档
- write feature doc

## 输入来源

运行前需要以下输入：

1. **模板文件**：使用Glob搜索 `*功能描述文档模板.md` 定位模板文件
2. **产品上下文**（可选）：使用Glob搜索 `*产品上下文.md` 定位产品上下文文档
3. **用户提供的内容**：
   - 截图（页面截图、流程图等）
   - 需求说明文字
   - 已有的功能描述片段

## 核心原则

### 截图内容记录规则（必须严格遵守）

记录截图内容时：

1. **严格按照截图内容记录**：只记录截图中实际显示的字段、文本、按钮等元素，不进行任何推测或补充
2. **禁止基于预期推测**：不要基于功能名称、常见模式或经验来推测可能存在的字段或功能
3. **逐字逐句核对**：记录前必须仔细查看截图，逐个字段进行核对，确保每个记录的内容都能在截图中找到对应
4. **明确标注缺失信息**：如果某些信息在截图中不可见或不清晰，应明确标注"待补充"或"截图中不可见"
5. **避免套用模式**：不要因为看到类似的表单结构就套用其他页面的字段或逻辑，每个截图都应独立分析

---

## 工作模式

采用**交互式流程收集**模式:逐步收集详细流程信息,每收集一个功能点立即更新到文档中。

**核心优势**:
- 实时反馈:每个功能点写入后用户能立即看到效果
- 可中断:随时可以暂停,已写入的内容不会丢失
- 可修改:发现问题可以立即修改已写入的部分
- 降低风险:不会因为一次性输入太多导致遗漏或错误

**工作节奏**:
- 逐步收集信息,不要一次性要求用户提供所有信息
- 每完成一个功能点的记录,立即更新文档
- 发现问题立即反馈,不要等到最后

**确认机制**:
- 记录完一个页面或功能点后,询问用户是否准确
- 遇到截图中不清晰的内容,主动询问
- 完成一个完整流程后,与用户确认是否有遗漏

## 执行步骤

#### Step 1：确认基本信息

1. 使用Glob搜索 `*功能描述文档模板.md` 定位并读取模板文件
2. 使用Glob搜索 `*产品上下文.md` 定位并读取产品上下文(可选)
3. 与用户确认:
   - 模块名称(如果命令行未提供)
   - 涉及的用户角色
   - 文档保存路径(默认: `模块功能描述/[模块名称]模块.md`)

#### Step 2：逐步收集详细流程

**核心原则**: 每收集一个功能点,立即整理并写入文档

**收集流程**:

1. **提问当前功能点**:
   ```
   请告诉我第一个功能点的信息:
   - 页面名称是什么?
   - 如何触发进入这个页面?
   - 页面有哪些功能区域?
   - 有哪些字段?(请说明必填/选填、字段类型)
   - 有哪些操作按钮?
   - 业务规则是什么?
   - 有待补充的内容吗?
   ```

2. **用户回答后**:
   - 对不清楚或有歧义的内容追问
   - 整理成模板格式
   - **立即写入文档**(使用Edit工具追加到"详细流程步骤"章节)

3. **确认是否继续**:
   ```
   ✓ 已记录: [功能点名称]

   是否继续下一个功能点?
   - 回复"是"或"继续" → 继续收集
   - 回复"否"或"完成" → 进入Step 3生成其他章节
   ```

4. 如果继续,重复步骤1-3

**写入格式**:
```markdown
#### [功能点名称]

**入口路径**:
- [入口描述，如：App → 模块 → 功能]

或

**触发操作**:
- [触发条件描述]

**页面名称**："[页面标题]"页面

**页面功能**:

**1. [功能区域1]**:
- **[字段名]**([必填/选填],[类型])
  - [字段说明]

**2. 操作按钮**:
- [按钮名称]:[按钮行为]

**业务规则**:
- [规则1]
- [规则2]

**待补充内容**:
1. [待补充项1]

---
```

#### Step 3：AI自动生成其他章节

根据已收集的详细流程,参照功能描述文档模板自动生成:

1. **模块概述**:
   - 业务场景(从流程中提取业务背景)
   - 核心价值(针对不同角色总结价值点)
   - 主要功能(总结流程中的功能模块)
   - 适用范围
   - 受众角色(角色定义、使用场景、典型用户、核心诉求)
   - 业务价值

2. **通用规则说明**:
   - 文件上传规则(如适用)
   - 字段校验规则(从流程中提取共性规则)
   - 页面交互规则(从流程中提取交互模式)
   - 业务规则类别(如审核流程规则、状态流转规则等)
   - 动态表单规则(如有动态表单)
   - 特殊要求类别(如适用)

3. **完整业务流程**:
   - 流程概述(总结完整业务流程,标注关键分支和决策点)
   - 流程图(根据详细流程绘制ASCII流程图,标注步骤编号和分支条件)

**注意**: 所有章节的结构和格式必须严格遵循模板文件中的定义。

#### Step 4：展示完整文档并确认

1. 展示生成的完整文档结构
2. 询问是否需要修改
3. 如需修改,定位到具体章节进行调整

## 输出规范

### 语言风格
- 使用中文
- 产品经理视角，面向开发人员
- 清晰、准确、无歧义
- 避免技术实现细节

### 格式规范
- 使用 Markdown 格式
- 标题层级清晰（# ## ### ####）
- 使用列表、表格、代码块等格式增强可读性
- 流程图使用 ASCII 字符（```包裹）

### 内容规范
- 字段描述格式：**字段名**（必填/选填，字段类型，说明）
- 业务规则使用列表形式
- 待补充内容明确标注
- 截图内容严格按照实际内容记录

### 特殊标注
- 待补充内容：**待补充内容**：1. xxx
- 待确认问题：**待确认问题**：1. xxx
- 未来规划：**未来规划**：- xxx

## 文档维护

### 更新文档（用户说"更新文档"时触发）

1. 读取现有文档
2. 根据用户提供的新信息（截图、文字说明）更新对应章节
3. 保持文档结构不变
4. 在更新的章节添加更新日期标注（可选）

### 补充待定内容（用户说"补充待定内容"时触发）

1. 读取现有文档
2. 找到所有"待补充"、"待确认"标注
3. 根据用户提供的信息逐项补充
4. 移除已补充项的"待补充"标注

---

## 质量检查清单

输出文档前，检查以下项目：

- [ ] 文档结构完整（头部、基础信息、详细功能、特殊场景）
- [ ] 目录链接正确
- [ ] 所有截图内容已严格按照实际内容记录
- [ ] 没有推测或补充截图中不存在的内容
- [ ] 字段描述格式统一
- [ ] 业务规则清晰明确
- [ ] 待补充内容已明确标注
- [ ] 流程图清晰易懂
- [ ] 表格格式正确

```
---
✓ 功能描述文档已生成完成

文档路径: 模块功能描述/[模块名称]模块.md

后续操作:
- 说"更新文档" → 根据新信息更新文档
- 说"补充待定内容" → 补充标注为"待补充"的内容
- 说"继续收集" → 继续添加新的功能点
- 说"检查文档" → 对照质量检查清单进行检查
- 说"同步文档"或使用 /sync → 更新产品上下文和产品规划
```

---

## 注意事项

1. **严格遵守截图记录规则**：这是最重要的原则，任何违反都会导致文档不准确
2. **保持文档结构一致性**：所有功能描述文档都应遵循相同的模板结构
3. **明确标注不确定内容**：宁可标注"待补充"，也不要猜测
4. **及时与用户确认**：遇到不清楚的地方，主动询问用户
5. **保持文档可维护性**：使用清晰的章节结构和锚点链接
