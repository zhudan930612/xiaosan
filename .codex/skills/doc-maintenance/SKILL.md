# 文档维护与格式规范

## 功能描述文档约束（02_已上线功能描述/）

1. **只记录已上线功能**：不记录规划中的功能
2. **严格基于截图**：所有描述必须基于截图或实际产品，不能推测
3. **不写完整 PRD**：记录 AI 设计功能时真正需要参考的内容
4. **保持结构一致性**：遵循相同模板结构

## 功能设计文档约束（03_需求设计文件/）

1. 文件开头标注 `🤖 AI草稿` 或 `✅ 人工确认`
2. 每项设计结论注明信息来源；无法溯源的标注 `⚠️ 待确认`
3. 禁止将推断写为事实，未经确认的内容标注 `⚠️ 推测，待确认`
4. 文档末尾必须有"待确认事项"章节
5. 须经 PM 人工审核，标注 `✅ 已确认，可开发` 方可转交开发
6. 发现待解决问题同步写入产品规划对应模块的"待解决问题"章节
7. 页面设计和原型设计不写入正文，单独生成 HTML 存于 `04_原型设计文件/`
8. 必须包含决策记录章节
9. 原型无标注时，功能设计必须兜底补充完整规则

## 决策文档约束（05_决策记录文件/）

1. 保留原始沟通结论，不因后续整理而删除
2. 统一放在同一个目录中，通过文件名体现功能主题
3. 命名格式：`YYYY-MM-DD-[功能主题]决策记录.md`
4. 已确认结论和待确认事项必须分开记录
5. 不直接替代产品规划、功能设计和功能描述

## 文件命名编号规范

- `03/` 和 `04/` 下新文件使用两位数字前缀 `NN_`，按最大编号 + 1 递增
- 功能设计和原型设计编号必须一致
- 其他目录创建文件前，先检查该目录已有文件的命名习惯

## 功能设计与原型的文件关联规则

每个功能主题对应一对文件，编号必须一致：
- 功能设计：`03_需求设计文件/NN_[功能名]功能设计.md`
- 原型设计：`04_原型设计文件/NN_[功能名]原型设计.html`

修改时必须同时处理对应的一对文件。

## 格式选择优先级

1. 政策性/合同性正式文本 → 纯文本（无 Markdown 标记）
2. 功能设计文档 → Markdown + 纯文本混合
3. 方案比选/规划建议 → Markdown 为主
4. 用户明确指定 → 以用户要求为准

## 功能设计文档混合输出细则

- Markdown 适合：信息架构、方案对比、流程图、模块拆分、状态说明、待确认事项、设计依据、决策记录
- 纯文本必须：需求说明、业务规则、字段规则、校验逻辑、状态流转规则、接口约束、权限判断、异常处理、给开发的实现口径
- 推荐：先用 Markdown 搭章节结构，再在规则章节内用纯文本段落
