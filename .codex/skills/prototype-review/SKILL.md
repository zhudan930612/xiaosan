# 原型审查技能

## 触发词

当用户说以下关键词时触发：
- "审查原型"
- "检查原型"
- "原型审查"
- "review prototype"
- "/prototype-review"
- "帮我检查下原型"
- "看看原型有没有问题"

## 工作流程

### 步骤1: 确定审查目标

**方式1: 用户指定文件**
- 用户直接提供文件路径 → 审查该文件

**方式2: 自动发现**
- 使用 Glob 搜索 `**/*原型*.html`
- 如果找到多个，列出供用户选择
- 如果没找到，提示用户指定文件

### 步骤2: 读取并分析原型

读取 HTML 文件后，按照以下维度进行检查：

#### 2.1 可访问性检查

| 检查项 | 检查内容 | 问题等级 |
|-------|---------|---------|
| 颜色对比度 | 文字颜色与背景对比度是否 ≥ 4.5:1 | 严重 |
| 表单标签 | input 是否正确关联 label | 严重 |
| 焦点状态 | 可交互元素是否有可见的焦点样式 | 警告 |
| 图片替代文本 | 功能性图片是否有 alt 属性 | 警告 |
| 语义化标签 | 是否使用 header/main/footer 等语义标签 | 建议 |

#### 2.2 一致性检查

| 检查项 | 检查内容 | 问题等级 |
|-------|---------|---------|
| 按钮样式 | 同类按钮是否使用相同样式 | 警告 |
| 间距统一 | 卡片间距、内边距是否一致 | 建议 |
| 字体层级 | 标题/正文/提示文字大小是否统一 | 建议 |
| 圆角一致 | 同类元素圆角是否统一 | 建议 |
| 颜色使用 | 是否使用项目主色调 | 建议 |

#### 2.3 规范检查（基于原型设计标准）

| 检查项 | 检查内容 | 问题等级 |
|-------|---------|---------|
| 手机框架尺寸 | 是否为 375px 宽度 | 严重 |
| 高度自适应 | `.phone-frame` 是否使用 `height: auto` | 严重 |
| 状态栏规范 | 是否与导航栏同色，高度 44px | 警告 |
| 最小高度 | 内容区是否有 `min-height: 667px` | 警告 |
| 底部定位 | 是否使用相对定位而非 fixed | 严重 |
| 弹窗隐藏 | 弹窗是否默认隐藏 | 警告 |
| 占位元素 | 是否存在占位 div 撑开高度 | 警告 |
| 刘海装饰 | 是否存在 phone-notch 装饰 | 警告 |
| 标注布局 | 标注是否使用 flex 而非 absolute | 警告 |
| 可编辑性 | 可编辑元素是否正确设置 contenteditable | 建议 |

### 步骤3: 输出审查报告

以 Markdown 格式输出审查报告，包含：

1. **摘要统计** - 严重/警告/建议 问题数量
2. **详细问题清单** - 按问题等级分组
3. **修复建议** - 针对每个问题给出具体修复代码
4. **通过项** - 列出符合规范的项目

## 颜色对比度计算方法

使用相对亮度公式计算对比度：

```javascript
// 相对亮度计算
function getLuminance(r, g, b) {
    const [rs, gs, bs] = [r, g, b].map(c => {
        c = c / 255;
        return c <= 0.03928 ? c / 12.92 : Math.pow((c + 0.055) / 1.055, 2.4);
    });
    return 0.2126 * rs + 0.7152 * gs + 0.0722 * bs;
}

// 对比度计算
function getContrastRatio(lum1, lum2) {
    const lighter = Math.max(lum1, lum2);
    const darker = Math.min(lum1, lum2);
    return (lighter + 0.05) / (darker + 0.05);
}
```

WCAG 标准：
- AA 级：正常文字 ≥ 4.5:1，大号文字 ≥ 3:1
- AAA 级：正常文字 ≥ 7:1，大号文字 ≥ 4.5:1

## 评分标准

- **100分**: 无严重问题，无警告问题
- **80-99分**: 无严重问题，有 ≤ 3 个警告
- **60-79分**: 有严重问题但已修复，或有 ≤ 5 个警告
- **< 60分**: 存在未修复的严重问题

## 注意事项

1. **优先修复严重问题** - 确保基本可用性
2. **区分必须和建议** - 不要把建议当成必须修复
3. **给出具体代码** - 修复建议要有可操作性
4. **考虑项目实际** - 如果是故意的设计选择，可标记为"设计意图"
5. **检查前确认** - 如果找到多个原型文件，让用户选择审查哪个
