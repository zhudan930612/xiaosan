# AI 自检清单

## 使用说明

本清单用于 AI 在产出各类文档后**自动执行自查**，无需用户逐条确认。

**执行时机**：
1. 输出任何文档后，AI 自动对照本清单检查
2. 若发现不符合项，AI 必须在交付前自动修正
3. 修正后再次检查，直至全部通过
4. 最终向用户汇报："自检通过，共检查 X 项，全部符合"

**自检不通过的处理**：
- 可直接修正的 → 自动修正，不打扰用户
- 需要用户提供信息才能修正的 → 在"待确认事项"中标注，交付时提示用户补充

---

## 清单一：功能设计文档自检（03_需求设计文件/）

### 文档状态
- [ ] 文件开头标注了 `🤖 AI草稿，待人工确认` 或 `✅ 人工确认，可开发`
- [ ] 若方案未确认，没有标注 `✅ 已确认，可开发`

### 信息来源
- [ ] 每个设计结论都注明了信息来源（如：基于截图、基于产品上下文、基于访谈记录）
- [ ] 从截图推断的业务规则标注了 `⚠️ 推测，待确认`
- [ ] 无法溯源的内容标注了 `⚠️ 待确认`
- [ ] 没有将推断写为事实的情况

### 内容完整性
- [ ] 包含"功能概述"章节
- [ ] 包含"业务流程"章节
- [ ] 包含"功能需求描述"或"业务规则"章节
- [ ] 规则性内容（需求说明、字段规则、校验逻辑）使用纯文本格式
- [ ] 结构性内容（流程、模块拆分、决策记录）使用 Markdown 格式

### 待确认事项
- [ ] 文档末尾有"待确认事项"章节
- [ ] 每条待确认项包含：问题描述、影响范围、建议负责人
- [ ] 没有遗漏的 `⚠️ 待确认` 内容未归入待确认事项

### 决策记录
- [ ] 包含"决策记录"章节
- [ ] 记录了关键决策：决策内容、选择方案、决策依据

### 原型配对
- [ ] 若功能涉及页面设计，检查 `04_原型设计文件/` 下是否有对应的原型文件
- [ ] 若原型选择"无标注"，功能设计文档的"功能需求描述"或"业务规则"章节已逐页补充完整的页面状态规则、数据规则、字段规则、交互分支规则、边界情况规则

---

## 清单二：功能描述文档自检（02_已上线功能描述/）

### 内容约束
- [ ] 只记录已上线功能，不包含规划中的功能
- [ ] 所有功能描述基于截图或实际产品，没有推测内容
- [ ] 没有写超出"AI设计功能时真正需要参考的内容"的冗余内容

### 结构一致性
- [ ] 遵循与同级模块相同的文档结构
- [ ] 模块边界清晰，没有与其他模块的功能重叠

---

## 清单三：原型设计文档自检（04_原型设计文件/）

### 前置检查
- [ ] 已读取 `01_产品主文档/小散原型设计标准文档.md`
- [ ] 原型类型已确认（可编辑/不可编辑、有标注/无标注）

### 文件规范
- [ ] 文件名格式：`[功能名]原型设计.html`
- [ ] HTML 包含基础交互能力（点击跳转、状态切换等）

### 标注完整性（仅"有标注"类型）
- [ ] 每个页面标注包含：页面状态规则
- [ ] 每个页面标注包含：数据规则
- [ ] 每个页面标注包含：字段规则
- [ ] 每个页面标注包含：交互分支规则
- [ ] 每个页面标注包含：边界情况规则
- [ ] 标注内容与对应功能设计文档的规则描述一致，没有矛盾

---

## 清单四：决策记录文档自检（05_决策记录文件/）

### 格式规范
- [ ] 文件名包含日期和主题：`YYYY-MM-DD-[功能主题]决策记录.md`
- [ ] 已确认结论与待确认事项分开记录
- [ ] 待确认事项标注了影响范围和下一步行动

---

## 清单五：文档间一致性自检

### 跨文档检查
- [ ] 功能设计文档中的规则描述与已上线功能描述不矛盾
- [ ] 决策记录中的已确认结论已同步到对应的功能设计文档或产品规划
- [ ] 产品规划中标记为"设计中"的功能，在 `03_需求设计文件/` 中有对应文档
- [ ] `03_需求设计文件/*.md` 与 `04_原型设计文件/*.html` 的命名前缀一一对应

---

## 清单六：格式规范自检

### 政策性文档（服务协议、隐私政策等）
- [ ] 不使用 Markdown 标记符号（无 #、##、-、**、--- 等）
- [ ] 章节用中文编号区分（一、二、1.1、（1）等）
- [ ] 列表项直接换行，不加符号前缀

### 功能设计文档
- [ ] 需求说明、业务规则、字段规则、校验逻辑、状态流转规则使用纯文本
- [ ] 信息架构、方案对比、流程图、待确认事项、决策记录使用 Markdown
