# 任务式协作指南

## 理念转变

**从问答式到任务式**

| 维度 | 问答式（旧） | 任务式（新） |
|-----|-----------|-----------|
| 用户输入 | "帮我写一下巡查频次配置的功能设计" | "任务：完成巡查频次配置功能设计文档" |
| AI 行为 | 逐步询问、逐段输出、等待确认 | 自主规划、自动读取、自动检查、一次性交付 |
| 交互频率 | 高（每段内容都要确认） | 低（只在关键决策点确认） |
| 用户角色 | 参与者（每一步都要指导 AI） | 审阅者（只看结果，做关键决策） |
| AI 角色 | 执行者（按指令行事） | 代理者（自主完成任务） |

**核心原则**：用户给目标，AI 自主完成。用户只在关键决策点介入。

---

## 任务式工作流

### 阶段一：接收任务（用户）

用户用以下格式给出任务：

```
任务：[目标]
[可选补充信息]
```

示例：
- `任务：完成巡查频次配置功能设计文档`
- `任务：编写安全巡检模块的已上线功能描述`
- `任务：输出备案分派流程的HTML原型`
- `任务：给出下一步功能优先级建议`

**用户可补充的信息**：
- 已有参考资料（截图、会议纪要等）
- 特殊约束（时间紧、需要简化等）
- 需要重点关注的内容

---

### 阶段二：解析任务（AI）

AI 收到任务后，自动完成以下步骤：

1. **识别任务类型**：匹配【任务上下文包】中的触发条件
2. **加载上下文包**：自动读取该任务类型对应的所有参考文档
3. **向用户报告**："已识别任务类型：XX，已自动读取以下参考文档：..."
4. **自主规划执行步骤**：基于上下文包中的执行步骤，规划具体行动

---

### 阶段三：自主执行（AI）

AI 按规划步骤自主执行，**不需要用户逐步确认**。

执行过程中 AI 遵循以下规则：

**规则 1：信息充分时自主决策**
- 若参考文档中已有足够信息，AI 直接基于文档内容产出
- 不需要询问用户"这个功能是做什么的"

**规则 2：信息不足时标注待确认**
- 若参考文档中没有相关信息，AI 在文档中标注 `⚠️ 待确认`，而不是停下来问用户
- 所有待确认项集中放在"待确认事项"章节

**规则 3：关键决策点主动汇报**
- 以下情况需要暂停执行，向用户汇报并征求意见：
  - 发现多个可选方案，且各有优劣
  - 发现与现有文档存在矛盾
  - 需要用户确认业务规则
  - 超出任务范围的额外需求

**规则 3 汇报格式**：
```
【需要决策】
问题：[简要描述]
选项 A：[方案及利弊]
选项 B：[方案及利弊]
建议：[AI 的推荐及理由]
请确认选择哪个方案，或提供其他思路。
```

---

### 阶段四：自检交付（AI）

产出完成后，AI 自动执行对应的【AI自检清单】：

1. 对照清单逐条检查
2. 可直接修正的问题 → 自动修正
3. 需要用户补充的问题 → 归入"待确认事项"
4. 向用户汇报："自检通过，共检查 X 项，全部符合"或"发现 Y 项待确认，已归入待确认事项"

**交付格式**：
```
任务完成：[任务目标]

交付物：
- 文件路径：[路径]
- 文档性质：[🤖 AI草稿 / ✅ 人工确认]

自检结果：
- 检查项：X 项
- 通过：X 项
- 待确认：Y 项（已归入文档"待确认事项"章节）

关键决策记录：
- [决策 1：内容、选择、依据]
- [决策 2：内容、选择、依据]

请审阅文档，确认后我将继续下一步（如需）。
```

---

### 阶段五：用户审阅（用户）

用户审阅交付物，给出反馈：

- **确认**："没问题，继续下一步"或"确认，归档"
- **修改**："第 X 节需要调整..."或"补充..."
- **终止**："这个方向不对，重新来"

## 任务式 Prompt 示例

### 示例 1：功能设计

```
任务：完成巡查频次配置功能设计文档

补充信息：
- 巡查频次目前是每个工程单独配置，新需求是支持按工程类型批量配置
- 需要支持 PC 管理端和 App 巡查端两种场景
- 优先级高，希望本周内进入开发
```

AI 的执行流程：
1. 识别为【包 B 或 C：功能设计】
2. 自动读取产品上下文、产品规划、已上线描述
3. 自主分析现有巡查频次功能现状
4. 输出功能设计文档（包含需求分析、业务流程、规则说明）
5. 自检清单一检查
6. 交付并提示审阅

---

### 示例 2：功能描述

```
任务：编写安全巡检模块的已上线功能描述

补充信息：
- 基于以下截图：[附截图]
- 模块包含：检查项管理、巡检任务、巡检记录、隐患上报
```

AI 的执行流程：
1. 识别为【包 A：编写功能描述文档】
2. 自动读取产品上下文、产品规划、现有安全巡检描述（如有）
3. 基于截图逐页分析功能
4. 输出功能描述文档
5. 自检清单二检查
6. 交付并提示审阅

---

### 示例 3：原型设计

```
任务：输出备案分派流程的 HTML 原型

补充信息：
- 对应功能设计文档已确认，路径：03_需求设计文件/备案分派流程功能设计.md
- 原型类型：不可编辑，有交互且有标注
```

AI 的执行流程：
1. 识别为【包 D：原型设计】
2. 自动读取原型设计标准文档 + 对应功能设计文档
3. 确认原型类型（用户已指定，无需询问）
4. 按页面清单逐页输出 HTML 原型
5. 每页补充 5 类规则标注
6. 自检清单三检查
7. 交付并提示审阅

## 何时退回问答式

以下场景不适合任务式，应退回问答式：

1. **探索性讨论**：用户说"我在想是不是可以..."、"你觉得 XX 怎么样"
2. **信息严重不足**：没有参考文档、没有截图、没有上下文
3. **需求频繁变更**：用户不断修改任务目标
4. **复杂方案比选**：需要在多个方向间深入讨论

判断方式：若 AI 发现需要 3 次以上交互才能明确任务目标，主动建议："当前信息不足以直接产出，建议先通过问答式梳理需求，明确后再进入任务式执行。"

## 渐进式采用建议

| 阶段 | 采用方式 | 说明 |
|-----|---------|------|
| 第 1 周 | 先用任务式做"功能描述文档" | 已有截图和模板，信息最充分 |
| 第 2 周 | 扩展至"原型设计" | 原型有明确的功能设计文档作为输入 |
| 第 3 周 | 扩展至"功能设计" | 此时已熟悉任务式节奏 |
| 第 4 周 | 全面采用 | 所有标准任务类型都用任务式 |

建议先在信息充分、流程明确的任务上试用，逐步建立信任后再扩展。
