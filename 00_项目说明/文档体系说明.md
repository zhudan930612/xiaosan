# 文档体系说明

## 一、目标

本项目采用“产品主文档 + 已上线功能描述 + 需求设计文件 + 原型设计文件 + 决策记录文件 + 附件资料”的文档结构，目标是让后续每次功能沟通、AI 设计、原型产出和历史回顾都有固定落点，同时避免目录层级过深、维护成本过高。

## 二、目录结构

## 三、各层文档职责

### 1. 产品主文档

- `01_产品主文档/小散业务产品上下文.md`
  记录业务背景、政策要求、角色定义、术语、模块边界等低频变更信息。
- `01_产品主文档/小散业务产品规划.md`
  记录模块拆分、子功能状态、设计中事项、待解决问题。
- `01_产品主文档/小散原型设计标准文档.md`
  记录全局原型风格和输出规范。

### 2. 已上线功能描述

- `02_已上线功能描述/`
  只记录系统中已经上线、可以通过截图或真实系统确认的内容。
- 这部分是 AI 判断“现状能力边界”的首要依据。

### 3. 需求设计文件

- `03_需求设计文件/`
  统一存放功能设计文档、方案研究文档、规则说明文档。
- 不再按主题建多层目录，依靠文件名区分主题。

### 4. 原型设计文件

- `04_原型设计文件/`
  统一存放 HTML 原型、流程图、系统框架图等可视化文档。

### 5. 决策记录文件

- `05_决策记录文件/`
  统一存放每次独立沟通形成的决策文档原件。
- 不再按主题建子目录，依靠文件名体现功能主题和时间。

### 6. 附件资料

- `06_附件资料/`
  统一存放支撑设计、回顾和对照所需的附件、规则、历史材料和整理稿。
- 附件资料统一进入这一个目录，不再额外拆分会议纪要、系统截图、临时素材等子层级。

## 四、功能设计文档规范

### 4.1 使用模板

- `00_项目说明/功能设计文档模板.md`
  提供标准化的功能设计文档结构，包含模块概述、基础信息、详细功能描述、数据权限规则、业务规则详解、原型设计说明等必备章节。

### 4.2 新建功能设计文档流程

1. **复制模板**：将 `功能设计文档模板.md` 复制到 `03_需求设计文件/`
2. **重命名**：按 `[功能名]功能设计.md` 格式命名
3. **填充内容**：根据模板中的占位符 `{}` 逐项填写
4. **标注状态**：文件开头标注 `🤖 AI草稿，待人工确认` 或 `✅ 人工确认，可开发`

### 4.3 模板结构说明

| 章节 | 用途 | 必填 |
|-----|------|------|
| 一、模块概述 | 说明业务场景、核心价值、受众角色 | 是 |
| 二、基础信息 | 定义入口、筛选器、状态 | 是 |
| 三、详细功能描述 | 逐个功能点描述（位置/内容/交互/业务规则） | 是 |
| 四、数据权限规则 | 权限控制逻辑、可见性判断 | 视功能需要 |
| 五、业务规则详解 | 计算公式、状态流转、刷新规则 | 是 |
| 六、原型设计说明 | 原型类型、页面清单、交互要求 | 是 |
| 七、待确认事项 | 列出待确认问题和负责人 | 建议填写 |
| 八、决策记录 | 记录关键决策和选择依据 | 建议填写 |

### 4.4 与原型设计的关系

- 功能设计文档先定稿（或至少核心规则确定）后，再输出原型
- 原型设计文件只存放 HTML 原型，需求规则写在功能设计文档中
- 原型中的标注只写交互说明，不写业务规则（业务规则在文档中详述）

## 六、决策文档工作流

### 1. 归档方式

- 决策文档统一放在 `05_决策记录文件/` 中。
- 通过文件名区分功能主题，不再额外创建主题子目录。

### 2. 命名方式

推荐格式：

```text
YYYY-MM-DD-[功能主题]决策记录.md
```

示例：

```text
05_决策记录文件/2026-04-09-线上投保决策记录.md
05_决策记录文件/2026-04-09-巡查统揽决策记录.md
```

### 3. 使用原则

- 决策文档保留原件，作为沟通过程和历史结论的追溯依据。
- 决策文档默认不强制拆分回各模块文档。
- 当某个结论已经稳定，并且用户明确要求更新正式文档时，再同步到产品规划、功能设计或已上线功能描述。
- 后续 AI 继续设计时，应优先读取 `05_决策记录文件/` 中与当前功能最相关、时间最新的决策文档，再结合产品主文档和正式设计文档继续推进。

## 七、更新优先级

当一个主题发生变化时，文档更新优先级如下：

1. 先更新 `05_决策记录文件/`，保留本次沟通原文。
2. 若已形成较稳定方案，再更新 `03_需求设计文件/`。
3. 若实施状态发生变化，再更新产品规划。
4. 若功能已经上线并可核实，再更新已上线功能描述。
5. 只有影响宏观业务边界时，才更新产品上下文。

## 八、特殊说明

- `AGENTS.md` 与 `CLAUDE.md` 保持在项目根目录，不迁入子目录。
- 原因：这两个文件本身是 AI 进入项目时读取的关键入口，保留在根目录更稳定。
- 若文档结构调整、工作流调整或命名规则调整，需同步更新这两个文件。
- 文件夹中的文档查看顺序统一按“修改时间倒序”或“创建时间倒序”查看，优先使用修改时间倒序。
- 不通过批量修改文件名来控制排序，现有文件名保持业务语义清晰即可。
