# 文档体系说明

## 一、目标

本项目采用“产品主文档 + 已上线功能描述 + 需求设计文件 + 原型设计文件 + 决策记录文件 + 附件资料”的文档结构，目标是让后续每次功能沟通、AI 设计、原型产出和历史回顾都有固定落点，同时避免目录层级过深、维护成本过高。

## 二、目录结构

## 三、各层文档职责

### 1. 产品主文档

- `01_产品主文档/小散业务产品上下文.md`
  记录业务背景、政策要求、角色定义、术语、模块边界等低频变更信息。
- `01_产品主文档/小散业务产品规划.md`
  记录模块拆分、子功能状态、设计中事项、待解决问题。
- `01_产品主文档/小散原型设计标准文档.md`
  记录全局原型风格和输出规范。

### 2. 已上线功能描述

- `02_已上线功能描述/`
  只记录系统中已经上线、可以通过截图或真实系统确认的内容。
- 这部分是 AI 判断“现状能力边界”的首要依据。

### 3. 需求设计文件

- `03_需求设计文件/`
  统一存放功能设计文档、方案研究文档、规则说明文档。
- 不再按主题建多层目录，依靠文件名区分主题。

### 4. 原型设计文件

- `04_原型设计文件/`
  统一存放 HTML 原型、原型设计说明文档、流程图、系统框架图等可视化文档。

### 5. 决策记录文件

- `05_决策记录文件/`
  统一存放每次独立沟通形成的决策文档原件。
- 不再按主题建子目录，依靠文件名体现功能主题和时间。

### 6. 附件资料

- `06_附件资料/`
  统一存放支撑设计、回顾和对照所需的附件、规则、历史材料和整理稿。
- 附件资料统一进入这一个目录，不再额外拆分会议纪要、系统截图、临时素材等子层级。

## 四、功能设计文档规范

### 4.1 使用模板

- `00_项目说明/功能设计文档模板.md`
  提供标准化的功能设计文档结构，采用“统一主骨架 + 可选扩展章节”的方式，避免小功能设计文档过重、过散。
- `00_项目说明/原型设计文档模板.md`
  提供标准化的原型设计文档结构，用于在输出高保真 HTML 原型前，先确认原型范围、页面结构、视觉方向和交互表现。

### 4.2 新建功能设计文档流程

1. **复制模板**：将 `功能设计文档模板.md` 复制到 `03_需求设计文件/`
2. **重命名**：按 `[功能名]功能设计.md` 格式命名
3. **填充内容**：根据模板中的占位符 `{}` 逐项填写
4. **标注状态**：文件开头标注 `🤖 AI草稿，待人工确认` 或 `✅ 人工确认，可开发`

### 4.3 模板结构说明

| 章节 | 用途 | 必填 |
|-----|------|------|
| 一、文档信息 | 说明文档目标、适用范围、信息来源、可选扩展章节规则 | 是 |
| 二、功能概述 | 说明业务场景、核心价值、功能边界 | 是 |
| 三、业务流程 | 用于描述主流程和关键分支，不展开页面细节 | 是 |
| 四、核心方案 | 说明方案摘要、涉及页面、能力范围、方案取舍 | 是 |
| 五、功能需求描述 | 按页面逐项描述入口、页面结构、页面流程和需求说明（标注），是文档主干 | 是 |
| 六、非功能性需求 | 说明性能、安全、兼容性、运维配置等要求 | 视功能需要 |
| 七、页面清单 | 罗列页面范围，供原型输出和范围确认使用 | 是 |
| 八、决策记录 | 记录关键决策和选择依据 | 建议填写 |
| 九、待确认事项 | 列出待确认问题和负责人 | 建议填写 |
| 十、附录 | 放原型输出说明、维护记录、相关文档 | 建议填写 |

默认主骨架固定，复杂功能如有需要，可额外追加“数据权限补充”“状态流转补充”“接口与数据映射”“原型输出说明”“历史方案对比”等可选扩展章节。

### 4.4 与原型设计的关系

- 功能设计文档先定稿（或至少核心规则确定）后，再决定是否输出原型
- 若用户未明确要求原型，可不输出原型设计文档和 HTML 原型文件
- 若用户明确要求原型，建议先创建“原型设计文档”，确认原型内容后再输出 HTML 原型
- 原型设计文档默认适用于中大型功能和高保真原型场景，小功能可不强制创建
- 原型应依据“功能需求描述”章节中的“页面流程”和“需求说明（标注）”输出
- 原型设计文档主要确认页面结构、视觉方向、交互表现和状态表现，不重复抄写完整业务规则
- 原型中的标注用于补充页面交互和逻辑说明，不替代功能设计文档中的业务规则

### 4.5 原型设计文档工作流

1. 先完成功能设计文档，明确业务流程、页面流程和页面规则。
2. 当确认需要输出原型时，先在 `04_原型设计文件/` 下创建 `[功能名]原型设计.md`。
3. 使用 `00_项目说明/原型设计文档模板.md` 填写原型目标、原型范围、视觉方向、页面原型说明和交互状态。
4. 用户确认 `原型设计.md` 后，再生成 `[功能名]原型设计.html`。
5. 若后续原型内容有调整，应优先更新 `原型设计.md`，再同步更新 HTML 原型。

## 六、决策文档工作流

### 1. 归档方式

- 决策文档统一放在 `05_决策记录文件/` 中。
- 通过文件名区分功能主题，不再额外创建主题子目录。

### 2. 命名方式

推荐格式：

```text
YYYY-MM-DD-[功能主题]决策记录.md
```

示例：

```text
05_决策记录文件/2026-04-09-线上投保决策记录.md
05_决策记录文件/2026-04-09-巡查统揽决策记录.md
```

### 3. 使用原则

- 决策文档保留原件，作为沟通过程和历史结论的追溯依据。
- 决策文档默认不强制拆分回各模块文档。
- 当某个结论已经稳定，并且用户明确要求更新正式文档时，再同步到产品规划、功能设计或已上线功能描述。
- 后续 AI 继续设计时，应优先读取 `05_决策记录文件/` 中与当前功能最相关、时间最新的决策文档，再结合产品主文档和正式设计文档继续推进。

## 七、更新优先级

当一个主题发生变化时，文档更新优先级如下：

1. 先更新 `05_决策记录文件/`，保留本次沟通原文。
2. 若已形成较稳定方案，再更新 `03_需求设计文件/`。
3. 若实施状态发生变化，再更新产品规划。
4. 若功能已经上线并可核实，再更新已上线功能描述。
5. 只有影响宏观业务边界时，才更新产品上下文。

## 八、特殊说明

- `AGENTS.md` 与 `CLAUDE.md` 保持在项目根目录，不迁入子目录。
- 原因：这两个文件本身是 AI 进入项目时读取的关键入口，保留在根目录更稳定。
- 若文档结构调整、工作流调整或命名规则调整，需同步更新这两个文件。
- 文件夹中的文档查看顺序统一按“修改时间倒序”或“创建时间倒序”查看，优先使用修改时间倒序。
- 不通过批量修改文件名来控制排序，现有文件名保持业务语义清晰即可。
