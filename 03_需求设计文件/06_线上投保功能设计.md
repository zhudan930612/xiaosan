## 设计范围

- 本文档仅讨论"备案申请 > 匹配保单"页面的前端交互改版。
- 本文档不确定保司最终接入方式，不写死外链、H5 内嵌、平台统一 API 的实现路径。
- 本文档不新增线下保单独立审核后台，不改变"保单随备案材料进入审核链路"的现状。

## 背景与目标

### 现状

- 当前备案申请的"匹配保单"页面只有线下上传窗口，用户对"上传保单"路径感知最强。
- 模块文档虽然已预留"线上投保/上传保单二选一"的能力描述，但实际关键交互流程未形成统一方案。
- 已有线上投保相关讨论主要集中在技术接入层，对"页面如何突出线上、弱化线下"缺少前端设计口径。

### 业务目标

来源：基于 [小散业务产品规划.md](../01_产品主文档/小散业务产品规划.md)、[小散业务产品上下文.md](../01_产品主文档/小散业务产品上下文.md)、[备案管理模块.md](../02_已上线功能描述/备案管理模块.md)、2026-04-01 阶段性工作会原文（文件待补充入库）

- 让用户在"匹配保单"页第一眼先看到线上投保，而不是先看到上传框。
- 在不堵死线下上传入口的前提下，提高线上路径的优先级和可见度。
- 明确表达"线上更便捷"，但避免形成"平台强制指定购买渠道"的风险。
- 保留线下上传作为备案环节中的合法可用路径，避免投诉和合规争议。

### 已确认口径

来源：基于本轮用户确认

- 线上投保不需要用户额外填写其他资料，只需要将备案已填写资料传给保司。
- 当前仍处于方案讨论阶段，待保司合作方式谈妥后才进入开发。
- 线下保单当前仍由街镇审核人员在备案审核资料时一并审核，未确认单独的平台审核机制。

## 功能概述

本次设计的核心不是新增一条技术链路，而是重构"匹配保单"页面的入口层级与文案引导。页面需要把"线上投保"从一个未落地能力，调整为当前页面中的主推荐路径；同时保留"已有保单，直接上传"的线下入口，确保在保司方案未最终确定、用户已在线下投保、或业务不适合强推线上时，备案流程仍可继续。

## 业务流程

### 流程图

```text
用户完成基础信息/工程信息/安全信息
  -> 进入"匹配保单"
  -> 查看保单匹配说明
  -> 平台优先展示线上投保
      -> 用户选择线上投保
          -> 进入选择保险公司页面
          -> 用户选择保险公司
          -> 进入投保确认页面（展示备案资料复用说明）
          -> 用户确认并完成支付
          -> 支付成功，保单信息自动同步
          -> 返回备案继续提交
      -> 用户选择已有保单
          -> 进入上传保单
          -> 上传保单文件
          -> 随备案材料进入审核链路
  -> 提交备案申请
```

### 业务规则

触发条件
用户完成前三个 Tab 后进入"匹配保单"页。

处理逻辑
1. 页面默认突出线上投保，但必须保留线下上传入口。
2. 线上投保说明以"复用备案资料、减少重复填写"为核心卖点。
3. 线下上传入口不能被隐藏到难以发现，也不能仅在异常情况下出现。
4. 线上和线下都属于当前页面允许的合法路径，提交时不对路径本身做价值判断。

边界情况
- ⚠️ 待确认：若后续保司只支持外链跳转，页面状态文案是否需要区分"跳转中"与"返回继续备案"。
- ⚠️ 待确认：若某些辖区暂时未开通线上投保，是否按配置显示"推荐线上"或降级为并列展示。

## 方案对比

### 方案 A：同屏主次分明

来源：基于现有备案页结构、用户对"线下不能堵死"的确认

#### 方案说明

页面同时展示"线上投保"和"上传保单"两条路径。
线上投保放在主卡位，作为主推荐入口；上传保单放在下方副卡位，弱化但始终可见。

#### 页面结构

- 顶部保留压缩版政策说明卡片。
- 政策说明下方直接展示"推荐线上投保"主卡。
- 主卡下方展示"已有有效保单"副卡。
- 底部主按钮按当前选择路径变化。

#### 优点

- 风险最低，容易被业务、合规和审核端接受。
- 用户一眼能看到两种方式，不会误解为平台禁止线下。
- 适合当前保司接入方式未定的阶段。

#### 不足

- 线下入口仍持续可见，容易让用户沿用旧习惯直接上传保单。
- 只能做到"线上更显眼"，不一定能形成明显导流。

### 方案 B：默认线上，线下可切换

来源：基于本轮讨论结论，结合会议中"要控制投保流程、突出平台承接"的方向

#### 方案说明

页面默认只展开线上投保主流程，同时保留明确的线下切换入口，如"已有保单？直接上传"。
用户点击后切换为上传保单态，且需保留"改为线上投保"的回切入口。

#### 页面结构

- 顶部保留压缩版政策说明卡片。
- 页面默认展示"线上投保"主流程卡。
- 主流程卡下方展示线下切换入口。
- 切换后展示上传区域，并保留返回线上入口。
- 底部主按钮仅服务当前路径。

#### 优点

- 比方案 A 更能体现"线上优先"的产品态度。
- 又没有把线下入口隐藏成异常兜底，风险可控。
- 与"线上复用备案资料、减少重复填写"的业务事实更匹配。

#### 不足

- 对入口文案、回切逻辑、默认态表达要求更高。
- 若线下入口过弱，容易引发"被引导得过头"的感受。

### 方案 C：线上吸引力强化版（同屏展示 + 多维引导）

来源：基于本轮用户对"突出线上投保可以使用各种手段"的要求

#### 方案说明

页面同屏展示"线上投保"和"上传保单"两条路径，线上投保区域通过视觉设计、文案包装、利益点展示、步骤引导等多种手段强化吸引力，让用户主动愿意选择线上投保；线下上传入口缩小但不隐藏，保持合规底线。

#### 页面结构

- 顶部保留压缩版政策说明卡片。
- 政策说明下方展示"线上投保"主推荐区（大卡片，品牌色突出）。
- 主推荐区内包含：标题区、利益点列表、步骤示意、主操作按钮。
- 主推荐区下方展示"已有有效保单"副卡（小卡片，弱化视觉）。
- 底部主按钮按当前选择路径变化。

#### 吸引力强化手段（方案 C 核心差异）

1. **视觉权重强化**
   - 线上投保卡片面积占页面主体约 60%~70%
   - 使用品牌主色（绿色系）作为卡片主色调
   - "推荐"角标或标签醒目展示
   - 卡片可带有轻微阴影或边框高亮

2. **利益点文案包装**
   - 主标题：突出行动价值，如"一键投保，免填资料"
   - 副标题：强调核心利益，如"备案信息自动同步保险公司，无需重复填写"
   - 利益点列表（3~4 条）：
     - 快捷出单：线上即时生成保单
     - 免审核：线上投保保单免人工核验
     - 信息复用：已填备案资料自动带入
     - 大平台保障：官方合作保险公司

3. **步骤简化示意**
   - 在卡片内以图标+文字形式展示简化流程：
     - ① 选择保险公司 → ② 确认信息 → ③ 完成支付
   - 让用户直观感知"只需 3 步"

4. **信任背书元素**
   - "平台官方推荐"标识
   - "已服务 XXXX 个工程"社会认同（如有数据）
   - 合作保险公司 logo 展示（可选）

5. **线下入口弱化处理**
   - 线下卡片面积小、颜色浅（灰色调）
   - 文案不带行动号召感，仅陈述事实："已有保单？直接上传"
   - 不展示利益点列表，仅一句话说明

#### 优点

- 线上路径的视觉吸引力和转化动机最强，不是" hiding 线下"而是"吸引选线上"。
- 两条路径始终同屏可见，合规风险低。
- 通过利益点包装和步骤简化，降低用户对"线上投保很复杂"的顾虑。
- 适合需要在合规边界内最大化线上转化的业务目标。

#### 不足

- 需要更精细的文案打磨和视觉设计投入。
- 利益点承诺需与实际产品能力严格对齐，避免过度承诺。
- 若后续保司接入方式变化（如只能外链），部分利益点可能需要调整文案。

### 方案 D：弱化政策说明，极致突出线上投保

来源：基于本轮用户对"在方案 C 基础上进一步强化线上投保吸引力"的要求

#### 方案说明

在方案 C 的基础上，进一步弱化政策说明类内容的视觉权重，将页面首屏全部留给线上投保核心内容，让线上投保成为用户进入页面后的唯一视觉焦点。政策说明不作为页面主体元素出现，而是以最小化方式提供（底部折叠或弹窗查看），避免政策文案分散用户对线上投保入口的注意力。

#### 页面结构

- 页面顶部直接展示"线上投保"超大推荐卡片（占页面主体 80% 以上）。
- 线上投保卡片内包含：标题区、核心利益点（4~5 条）、步骤示意、主操作按钮、信任背书标签。
- 线下入口极简处理：仅保留一行文字链接"已有保单？点击上传"，不做卡片、不带描述、不展示利益点。
- 政策说明压缩为页面底部一行小字，或折叠为"查看政策要求"可展开区域，或仅在点击问号图标时弹窗展示。
- 底部主按钮仅服务线上投保路径。

#### 吸引力强化手段（在方案 C 基础上进一步升级）

1. **政策说明极致弱化**
   - 政策说明不再以卡片形式占据页面顶部。
   - 改为页面底部一行灰色小字："点击了解安全生产责任保险政策要求"。
   - 或采用折叠面板形式，默认收起，不占用首屏空间。
   - 或仅在页面角落放置问号图标，点击后弹窗展示政策说明。

2. **线上投保区域最大化**
   - 线上投保卡片面积占页面主体约 80%~85%。
   - 卡片使用品牌主色 + 渐变色背景，视觉冲击力更强。
   - 核心卖点使用更大字号、更醒目的颜色突出。
   - 增加动态感：如保单号生成动画示意、"已有 XXX 个工程选择线上投保"实时计数（如有数据）。

3. **线下入口极简处理**
   - 线下入口从"小卡片"降级为"一行文字链接"。
   - 位置放在页面最底部，视觉层级最低。
   - 文案仅保留"已有保单？点击上传"，不做任何描述或引导。
   - 点击后仍然触发软卡点弹窗（与方案 C 一致）。

4. **首屏信息聚焦**
   - 用户进入页面后，首屏只看得到线上投保大卡片 + 底部操作按钮。
   - 政策说明和线下入口需要滚动或点击才能看到。
   - 减少用户在首屏的信息处理负担，直接导向线上投保行动。

#### 优点

- 线上投保路径的视觉占比和注意力集中度最高，转化动机最强。
- 首屏信息极度聚焦，用户几乎不存在"选哪条路径"的犹豫。
- 政策说明没有消失，只是后置/折叠，合规底线仍然保留。
- 与"平台需要控制投保流程、突出平台承接"的业务方向最匹配。

#### 不足

- 政策说明后置可能引起部分用户"找不到政策依据"的困惑。
- 线下入口降级为文字链接后，已有线下保单的用户需要滚动才能发现上传入口。
- 对政策合规审查的接受度要求更高，可能需要法务或业务部门额外确认。
- 若用户对政策要求有疑问，需要额外一步操作（点击/展开/弹窗）才能查看。

#### 与方案 C 的核心差异

| 维度 | 方案 C | 方案 D |
|------|--------|--------|
| 政策说明位置 | 页面顶部，卡片形式 | 页面底部/折叠/弹窗 |
| 政策说明视觉权重 | 中等（压缩为摘要卡片） | 极低（一行小字或图标） |
| 线上投保卡片占比 | 60%~70% | 80%~85% |
| 线下入口形式 | 小卡片，灰色调 | 一行文字链接 |
| 首屏焦点 | 线上投保 + 政策说明 | 仅线上投保 |
| 合规风险 | 低 | 中低（需确认政策说明后置是否可接受） |

## 推荐方案

来源：基于本轮讨论与用户确认

推荐采用方案 C，方案 D 作为次选（需确认政策说明后置的合规接受度），方案 B 作为备选，方案 A 作为保守备选。

推荐理由
1. 方案 C 在"吸引力强化"和"合规底线"之间取得了最佳平衡，不是通过隐藏线下来导流，而是通过强化线上的吸引力来引导用户主动选择。
2. 利益点包装（免填资料、快捷出单、免审核）与当前业务事实完全对齐，没有过度承诺。
3. 步骤简化示意能有效降低用户对"线上投保流程复杂"的心理门槛。
4. 同屏展示两条路径，用户在任何时候都能看到线下入口，不会产生"被强制"的感受。
5. 视觉权重的差异足以让线上路径成为用户的第一注意力焦点。

方案 D 适用场景
- 若业务侧和合规侧确认"政策说明可以后置/折叠"，方案 D 可作为首选。
- 方案 D 更适合在"导流转化率"优先级明显高于"政策透明度"的场景下使用。
- 建议先以方案 C 上线，收集用户行为数据后，再评估是否有必要升级到方案 D。

## 线下路径软卡点设计

来源：基于本轮用户新增要求、[备案管理模块.md](../02_已上线功能描述/备案管理模块.md) 中"上传保单需要审核人员人工核验保单的真实性和有效性"的现状描述、以及本轮关于合规与投诉风险的讨论结论

设计原则
1. 线下路径可以增加"软卡点"，但不能增加"硬拦截"。
2. 软卡点的目的应是再次提醒"线上更便捷"，而不是制造"线下不可走"的感受。
3. 所有软卡点尽量合并在一次弹窗内完成，不叠加多层弹窗。
4. 不写死"1~3 天"等具体时长，仅表述"可能影响备案审核时效"。
5. 不引入"需先尝试线上失败才能上传"的逻辑。

### 卡点 1：⚠️ 提示 + 二次确认

触发时机
用户点击"已有保单？直接上传"或"上传保单"入口时触发，不直接切换到上传态。

弹窗建议文案
线上投保
- 快捷出单
- 保单免审核
- 大平台有保障

上传保单
- 需人工核验真实性
- 可能影响备案时效

[ 仍要上传保单 ]
[ 改为线上投保 ]

效果说明
1. 能拦住一部分"习惯性直接上传"的用户，给线上路径一次明确回拉机会。
2. 能把"线上更便捷"的原因讲清楚，降低用户对平台弱化线下的敏感度。
3. 属于可解释的前端确认动作，合规风险低于隐藏入口或拖慢流程。

### 卡点 2：原因采集

触发时机
与二次确认弹窗合并呈现，用户决定继续走线下时顺手完成一次原因选择。

建议采集项
- 已线下购买保单
- 暂不选择线上投保
- 线下投保更优惠
- 其他

规则说明
原因采集仅保留单选，不增加长文本必填项。
默认不选中任何原因，用户需主动选择后才可继续上传保单。

效果说明
1. 为后续判断线下分流原因提供数据依据，便于和保司、业务继续讨论。
2. 帮助后续验证"是用户习惯问题，还是线上能力覆盖问题"。
3. 在不明显增加操作成本的前提下，形成一次用户路径留痕。

### 卡点 3：线上/线下路径轻对比

触发时机
不单独再加一层弹窗，而是与"⚠️ 提示 + 二次确认"合并在同一弹窗中展示。

展示原则
- 采用轻量说明式对比，不做营销海报式夸张对比。
- 线上强调"复用备案资料、无需重复填写、完成后继续备案"。
- 线下强调"适用于已完成投保、保单需人工核验"。
- 不出现"线下慢很多""线下不推荐"等过强措辞。

效果说明
1. 比单句提示更容易让用户理解两条路径的真实差异。
2. 可以增强"改为线上投保"按钮的转化力。
3. 通过"说明差异"而非"攻击线下"的方式导流，更适合当前合规边界。

### 软卡点整体结论

推荐在方案 C 中采用"单层弹窗合并三项信息"的方式实现软卡点。

建议实现顺序
1. 先上线"⚠️ 提示 + 二次确认"。
2. 同时加上轻量原因采集。
3. 将线上/线下差异说明融入同一弹窗，不再额外叠加第二层比较页。

不建议采用
- 隐藏线下入口
- 强制先尝试线上失败后才能上传
- 人为设置未确认依据的审核时长承诺
- 通过增加过多字段和材料要求来故意提高线下操作成本

## 功能点描述

### 匹配保单页改版

入口路径
施工端 -> 备案申请 -> 匹配保单

页面名称
"匹配保单"页面

页面功能

1. 政策说明摘要区
- 将现有大段政策文案压缩为摘要卡片。
- 保留政策必要性，但不再占据页面主体视觉中心。

2. 线上投保主入口（方案 C）
- 作为页面主推荐区出现，视觉面积占主体约 60%~70%。
- 卡片使用品牌主色调，带"推荐"角标或标签。
- 标题区突出行动价值："一键投保，免填资料"。
- 副标题说明核心利益：备案信息自动同步，无需重复填写。
- 利益点列表展示 3~4 条核心卖点（快捷出单、保单免审核、信息复用、大平台保障）。
- 步骤简化示意：图标+文字展示"①选公司 → ②确认 → ③支付"。
- 主操作按钮文案："去线上投保"。
- 点击后进入投保子流程第一步：选择保险公司页。

3. 线下上传入口
- 作为合法保底路径保留，视觉弱化（小卡片、灰色调）。
- 文案强调"已有保单，直接上传"，不带行动号召感。
- 用户点击线下入口时，先弹出单层软卡点弹窗，再决定是否进入上传态。
- 软卡点弹窗内同时承载风险提示、线上/线下差异说明和原因采集。

4. 投保子流程
- 选择承保公司页：横向 Tab 切换保险公司，下方联动展示该保险公司的保障详情（基础信息+保障详情）。基础信息从备案资料自动带入。用户确认后点击"确定投保"进入支付。
- 支付页：根据保司接入方式唤起支付，当前仅做占位设计。
- 支付成功页：展示成功状态，保单信息自动关联备案，提供返回备案入口。

5. 底部主按钮
- 按当前路径变化文案。
- 同一状态只保留一个主操作按钮。
- 线上投保完成后，返回匹配保单页展示保单摘要，按钮变为"提交备案审核"。

业务规则
- 不论方案 A/B/C，线下上传入口都必须可达。
- 不允许仅在"线上失败"后才出现线下入口。
- 页面引导重点是"线上更便捷"，不是"线下不可用"。
- 当前不新增线下保单平台审核角色。
- 线下软卡点仅允许采用"提示、对比、原因采集"这类轻确认方式，不允许形成事实上的禁止上传。
- 线下风险提示可使用"可能影响备案审核时效"，但当前不写死具体时长。
- 原因采集建议为单选，且与二次确认合并在同一弹窗中完成。
- 投保子流程中所有信息均从备案资料自动带入，不额外要求用户填写。
- 支付页的具体形态高度依赖保司接入方式，当前仅做占位设计。
- 保单信息同步能力取决于保司数据回传能力，若无法实时返回保单号，可仅展示投保成功状态。
