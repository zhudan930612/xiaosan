# 园区与街镇共管归属处理方案

**文档性质**：🤖 AI草稿，待人工确认
**设计日期**：2026-04-15
**目标模块文档**：备案管理、管理后台、巡查统揽/智慧大屏、权限管理

---

## 一、方案定位

### 1.1 业务背景

当前小型工程备案流程中，工程会通过定位解析到街道，用户再手动选择所属街道/社区。

基于现有产品设计，系统当前主要按“所属街道”承载以下能力：

- 备案审核归属
- 巡查数据归属
- 大屏街道统计
- 组织管辖区域过滤

但根据本次沟通确认，青浦工业园区、上海西虹桥商务开发有限公司对应的监管范围，并不等同于单一行政街道，而是与街镇存在共管、重叠、转派关系。  
信息来源：基于用户补充说明。

### 1.2 核心问题

当前系统把以下 4 个概念混在了同一个“街道归属”里：

- 行政归属：定位解析出的行政街道
- 申报归属：用户在前台选择的所属街道/社区
- 监管归属：实际负责审核、处理、巡查的监管组织
- 统计归属：大屏和权限过滤所使用的口径

这会导致园区/商务区共管场景下出现以下问题：

- 用户无法在前台直接选择“青浦工业园区”或“西虹桥商务区”
- 街镇与园区之间的“转办/改派”关系无法在系统中表达
- 大屏统计口径和实际监管口径不一致
- 权限只能套在街道上，无法表达园区与街镇重叠监管

信息来源：
- 基于用户补充说明
- 基于 [小散业务产品上下文.md](../01_产品主文档/小散业务产品上下文.md)
- 基于 [巡查统揽功能设计方案.md](./巡查统揽功能设计方案.md)

### 1.3 方案目标

- 不改变用户当前“按街道申报”的认知和操作习惯
- 能在系统中表达“行政街道”和“实际监管组织”不是同一层概念
- 支持街镇与园区之间的改派/转办
- 尽量控制对现网备案、大屏、权限逻辑的冲击范围

信息来源：基于用户补充说明与现有产品结构推导。

## 二、现状与约束

### 2.1 已知业务事实

1. 青浦工业园区与部分街镇共管，存在重叠范围。
2. 上海西虹桥商务开发有限公司对应的监管范围也不等同于单一街道。
3. 线下现状是：街镇先收口，认为不属于自身监管的工程，再转给园区/商务区处理。
4. 现有前台申报端没有“青浦工业园区”或“上海西虹桥商务开发有限公司”这类可选归属。

信息来源：基于用户补充说明。

### 2.2 现有系统约束

1. 备案端当前基于定位和街道/社区选择承载归属信息。  
信息来源：基于用户描述。

2. 产品上下文中，备案审核责任主体仍以“街镇”表达。  
信息来源：基于 [小散业务产品上下文.md](../01_产品主文档/小散业务产品上下文.md) 中“街镇审核材料”“街镇负责平台审核和属地监管”等描述。

3. 巡查统揽方案中，数据权限判断明确依赖“工程所属街道 in 用户管辖街道列表”。  
信息来源：基于 [巡查统揽功能设计方案.md](./巡查统揽功能设计方案.md) 第 4 章数据可见性设计。

4. 大屏、街道排名、街道详情等能力当前都按“街道”维度组织。  
信息来源：基于 [巡查统揽功能设计方案.md](./巡查统揽功能设计方案.md) 中街道排名区、街道详情页等设计。

### 2.3 风险点

- 如果直接把“工业园区/商务区”加到前台街道枚举，会把行政归属和监管归属混为一体
- 如果只在后台增加园区组织，但不新增归属字段，大屏和权限仍无法准确表达共管关系
- 如果一次性改成双归属前台填写，会显著增加用户理解和填报成本

信息来源：基于现有系统结构推导，⚠️ 推测，待确认。

## 三、方案对比

### 方案 A：把园区/商务区直接加入前台“所属街道”枚举

**方案说明**

用户申报时，前台直接把“青浦工业园区”“西虹桥商务区”作为街道选项展示，用户自行判断选择。

**优点**

- 表面上改动直观
- 审核归属可直接落到园区/商务区

**缺点**

- 行政街道与监管组织混成一层
- 地图、定位、街道统计、大屏筛选都会出现口径混乱
- 用户不一定知道自己属于街道还是园区，容易误填
- 历史数据与新增数据口径不一致

**结论**

不推荐。  
信息来源：基于用户补充说明与现有大屏/权限设计推导。

### 方案 B：保留前台按街道申报，新增“主责监管组织”

**方案说明**

用户前台仍按行政街道/社区申报。系统新增“主责监管组织”字段，审核端支持把工程从街镇改派给青浦工业园区、西虹桥商务区等监管组织。后续待办、审核、权限主要按“主责监管组织”生效。

**优点**

- 对用户最友好，前台几乎不增加理解成本
- 能表达街镇先收再转派的真实业务流程
- 大屏原有街道口径可以保留
- 可逐步扩展监管组织口径，而不是一次性推翻现有模型

**缺点**

- 后台、权限、大屏都要增加一层“监管归属”逻辑
- 需要定义转派规则、默认归属规则和历史数据补录策略

**结论**

推荐采用。  
信息来源：基于用户补充说明与现有系统约束综合判断。

### 方案 C：前台直接做双归属填写

**方案说明**

用户在前台同时填写：

- 所属行政街道/社区
- 所属监管组织（街镇/园区/商务区）

**优点**

- 数据表达最完整
- 理论上后续大屏和权限最灵活

**缺点**

- 用户填写成本高
- 容易填错
- 前台改动、校验逻辑、帮助文案和培训成本都更高
- 与现在线下“街镇先收、再转派”的做法不完全一致

**结论**

不建议作为当前阶段方案，可作为长期演进方向保留。  
信息来源：基于用户补充说明与现有用户操作模型推导。

## 四、推荐方案

### 4.1 推荐结论

推荐采用 **方案 B：保留前台按街道申报，新增“主责监管组织”**。

### 4.2 核心设计原则

1. 行政归属与监管归属分层，不再用一个“街道字段”承载全部含义。
2. 前台申报延续现有心智，不要求用户理解园区/商务区归属。
3. 审核归属、待办归属、权限归属统一改由“主责监管组织”承载。
4. 大屏保留“行政街道”口径，同时增加“监管组织”口径，避免一次性推翻原统计体系。

信息来源：基于方案对比后的推荐判断。

## 五、详细设计

### 5.1 核心概念定义

| 字段/概念 | 定义 | 主要用途 | 信息来源 |
|------|------|------|------|
| 行政街道/社区 | 用户定位与地址解析后所属的行政街道/社区 | 地图定位、街道统计、行政口径展示 | 基于现有系统 |
| 主责监管组织 | 实际负责审核、处理、巡查的组织，可为街镇、园区或商务区 | 待办归属、审核权限、监管视角统计 | 基于本次方案设计 |
| 是否共管 | 当前工程是否处于街镇与园区/商务区重叠监管范围 | 表达共管状态 | 基于用户补充说明 |
| 协同监管组织 | 除主责监管组织外的关联监管组织 | 共管协同、后续统计扩展 | 基于本次方案设计 |

### 5.2 数据模型调整

建议新增字段如下：

- `administrative_street`
  表示行政街道
- `administrative_community`
  表示行政社区
- `primary_supervision_org`
  表示主责监管组织
- `is_joint_supervision`
  表示是否共管
- `joint_supervision_orgs`
  表示协同监管组织列表
- `supervision_dispatch_from`
  记录转派来源组织
- `supervision_dispatch_time`
  记录转派时间

业务规则：

工程创建时，系统先写入行政街道/社区。主责监管组织默认等于该街道对应的街镇组织；若后续审核中判断属于工业园区或西虹桥商务区，则由有权限的审核人员执行改派，改派后主责监管组织更新为目标组织，同时保留原行政街道不变。

信息来源：
- 基于用户补充说明
- 基于现有备案、大屏、权限依赖关系推导

### 5.3 备案流程调整

规则说明

前台申报阶段不新增“工业园区/商务区”选项。用户仍然按现有方式选择所属街道/社区。

后台流转规则如下：

1. 用户提交备案后，工程先进入默认街镇组织待审核列表。
2. 街镇审核人员若判断该工程属于工业园区/西虹桥商务区主责范围，可执行“转派主责监管组织”。
3. 转派后：
原街镇组织从“主责待办”中移除该工程。
目标监管组织进入待审核状态。
系统保留转派日志。
4. 若工程属于共管场景，可标记“是否共管 = 是”，并挂载协同监管组织，但当前阶段只允许一个主责监管组织。

信息来源：
- 基于用户补充说明“街镇先转给工业区处理”
- ⚠️ 具体是否所有街镇都可转派、是否需要二次确认，待确认

### 5.4 管理后台调整

管理后台需补充以下能力：

1. 监管组织维护
支持配置街镇、园区、商务区等监管组织。

2. 行政区域与监管组织映射维护
支持配置：
某街道默认归属哪个街镇组织
某些区域是否存在园区/商务区共管
街镇与园区/商务区的转派关系

3. 主责监管组织改派
审核详情页支持改派主责监管组织。

4. 共管关系配置
支持配置某监管组织与哪些街镇存在重叠监管关系。

信息来源：基于本次方案设计，⚠️ 需结合后台实际能力确认。

### 5.5 权限规则调整

现有权限模型只按“管辖街道”过滤，需改成至少双维度判断：

- 行政区域维度
- 主责监管组织维度

建议规则：

1. 行政街道用于地图、大屏街道统计、地址展示。
2. 主责监管组织用于审核待办、审批操作、组织视角列表。
3. 条款权限和工程类型权限继续保留。
4. 组织可见范围判断优先以主责监管组织为准；若仅查看街道统计类数据，则按行政街道口径。

规则说明

IF 当前页面属于审核/处理/待办场景
THEN 按主责监管组织 + 条款权限判断

IF 当前页面属于街道统计/地图展示场景
THEN 按行政街道 + 工程类型判断

IF 当前页面属于监管组织统计场景
THEN 按主责监管组织 + 工程类型判断

信息来源：
- 基于 [巡查统揽功能设计方案.md](./巡查统揽功能设计方案.md) 现有权限逻辑
- 基于本次方案设计

### 5.6 大屏调整

当前大屏以街道为主要统计口径。建议改成双口径并存：

- 口径一：按行政街道统计
- 口径二：按监管组织统计

当前阶段建议：

1. 首页、地图、街道排名仍以行政街道为主。
2. 新增“统计口径”切换或“监管组织筛选”。
3. 街道详情页继续看行政归属视角。
4. 如需查看工业园区/西虹桥商务区表现，则切换到“监管组织口径”。

规则说明

不要把“青浦工业园区”“西虹桥商务区”直接塞进街道排名和街道筛选里，否则会破坏原有街道统计体系。

信息来源：
- 基于 [巡查统揽功能设计方案.md](./巡查统揽功能设计方案.md)
- 基于用户补充说明

### 5.7 历史数据处理建议

建议分阶段处理：

第一阶段：
只对新提交工程启用“主责监管组织”。

第二阶段：
如业务确有需要，再评估是否对存量工程批量回填主责监管组织。

规则说明

若直接对历史数据批量回填主责监管组织，会影响既有统计结果、权限结果和报表口径。当前更稳妥的方式是先增量生效，再视统计需求补历史。

信息来源：基于现有系统影响范围推导，⚠️ 推测，待确认。

## 七、会议建议拍板项

1. 是否确认青浦工业园区、西虹桥商务区在系统内定义为“监管组织”，而不是“街道”？
2. 是否确认前台申报端继续只选择行政街道/社区，不新增园区/商务区选项？
3. 是否确认审核端支持“转派/改派主责监管组织”？
4. 是否确认大屏采用双口径：行政街道口径 + 监管组织口径？
