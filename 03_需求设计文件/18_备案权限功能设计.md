# 备案权限体系重构功能设计

## 1. 背景与目标

### 1.1 当前问题

当前备案系统的权限模型存在以下核心问题：

1. **权限颗粒度过粗**：权限绑定在组织层面，组织内所有人共享同样的工程类型授权，无法按个人能力或职责分配。
2. **缺少街镇级统计视角**：当前按组织划分，镇长无法直接看到本街镇下所有科室的数据，区级也无法直接看到所有街镇的数据。
3. **不支持跨街镇分派**：备案一旦提交，所属街镇固定，无法流转到其他街镇处理。
4. **虚拟园区支持不足**：青浦工业园区、西虹桥商务区等虚拟街镇需要与实体街镇平级的管理方式。

### 1.2 设计目标

1. 建立四级权限体系（区级-街镇级-科室级-人员级），支持按街镇维度统计。
2. 权限从组织级下沉到人员级，同时通过"组织默认模板+继承"降低运营配置成本。
3. 支持跨街镇分派，分派粒度为街镇级别，由系统自动匹配目标街镇内有权限的人员。
4. 兼容后续6条业务改动（虚拟园区、按工程类型分权限、作业场所等）。

### 1.3 信息来源

- 基于产品上下文中的备案管理模块业务流程
- 基于06_附件资料/街镇条款级监管组织整理表.xlsx中的组织结构和授权数据
- 基于用户提供的后台组织管理截图
- 基于用户沟通的6条业务改动需求

## 2. 核心设计决策

### 决策1：沿用现有区域管理，不新增街镇表

**决策内容**：不新增独立的街镇数据表，沿用现有系统中"管辖区域"字段和区域管理模块。组织通过"管辖区域"字段关联到区域（街镇/园区）。

**选择方案**：复用现有区域管理，组织保留"管辖区域"字段。

**决策依据/原因**：
- 现有系统的区域管理模块已支持街镇/园区维度的数据管理，无需重复建设。
- 降低数据迁移成本和实施风险，避免两套区域数据冲突。
- 虚拟园区（青浦工业园区、西虹桥商务区）在现有区域管理中作为普通区域录入即可。

**注意**：跨街镇分派时的目标区域选择，数据来源仍为现有区域管理模块。

### 决策2：权限采用"组织默认模板 + 人员继承/微调"模式

**决策内容**：不采用逐人逐条配置权限的纯手工模式，而是组织层面配置默认授权模板，人员加入组织时自动继承，仅特殊情况下允许个人微调。

**选择方案**：组织配置默认模板，人员自动继承，支持个人级覆盖。

**决策依据/原因**：
- 同一科室的人员管理的工程类型高度一致（如规保办统一管理房屋建筑、交通工程等）。
- 如果逐人配置，11个街镇、上百号人的运营压力不可接受。
- 90%以上的场景下，人员权限与组织职责完全一致，不需要个性化配置。

### 决策3：跨街镇分派只需选择目标街镇

**决策内容**：跨街镇分派时，发起人只需选择目标街镇，不需要选择到具体人员。系统根据目标街镇 + 备案工程类型自动匹配有权限的人员，并发送确认待办。

**选择方案**：分派到街镇，系统自动匹配人员，被分派人确认后生效。

**决策依据/原因**：
- 发起人不需要知道目标街镇内部的人员分工，降低使用门槛。
- 系统已有条款级授权数据，可以自动匹配。
- 被分派人需要确认是否属于自己管辖范围，避免虚拟街道选错导致的分派错误。

## 4. 权限计算逻辑

### 4.1 人员最终权限计算

所有人员的权限都来源于组织关联（user_org_relation），不存在独立的全局角色。role_type 的值仅决定数据查询的过滤范围，不改变权限来源。

具体规则：

1. 可审核工程类型和作业场所权限：
   - 遍历该人员的所有启用状态的组织关联。
   - 对每个关联：
     - auth_mode=1（继承）→ 可审核工程类型 = organization.default_auth_types；小型工程作业场所 = organization.default_work_places_small；零星作业作业场所 = organization.default_work_places_scattered
     - auth_mode=2（自定义）→ 可审核工程类型 = user_org_relation.auth_types；小型工程作业场所 = user_org_relation.work_places_small；零星作业作业场所 = user_org_relation.work_places_scattered
   - 将所有关联的工程类型权限做并集。小型工程作业场所和零星作业作业场所分别做并集。

2. 数据可见范围：由 role_type 决定查询过滤条件。

示例：

张敏杰的配置：
- 关联1：白鹤镇-规保办，role_type=1（普通审核员），继承，组织模板={房屋建筑,交通工程,既有建筑修缮(非住宅)}

张敏杰的权限：
- 可审核工程类型：{房屋建筑, 交通工程, 既有建筑修缮(非住宅)}
- 可查看范围：白鹤镇-规保办组织内，且分配给自己或组织内未分配的

王镇长的配置：
- 关联1：白鹤镇政府，role_type=3（街道领导），继承，组织模板={全部工程类型}

王镇长的权限：
- 可审核工程类型：{全部工程类型}
- 可查看范围：白鹤镇管辖区域内所有组织的全部备案（不参与具体审核，仅统计查看）

### 4.2 备案列表查询过滤

根据登录人员的 role_type，备案列表的过滤条件如下：

场景1：普通审核员（role_type=1）
- 过滤条件（并集）：
  - project_record.assignee_id = 登录人员ID（分配给自己的）
  - 或（project_record.org_id IN (该人员所有关联的组织ID) AND project_record.project_type_name IN (该人员的最终可审核工程类型列表) AND project_record.assignee_id IS NULL）（组织内未分配的，且自己有工程类型权限的）
- 返回上述并集。

场景2：科室负责人（role_type=2）
- 过滤条件：project_record.org_id IN (该人员所有关联的组织ID)
- 返回这些组织的全部备案（不限制工程类型，因为科室负责人的职责是管理本科室全部备案）。

场景3：街道领导（role_type=3）
- 过滤条件：project_record.jurisdiction_area IN (该人员所有关联的组织的管辖区域列表)
- 返回这些管辖区域内的全部备案。
- 说明：街道领导挂靠在"XX镇政府"或"XX园区管委会"组织下，该组织的管辖区域即为其可见范围。

场景4：区级领导（role_type=4）
- 过滤条件：无限制（或 project_record.jurisdiction_area IN 全区所有区域）
- 返回全区所有备案。
- 说明：区级领导挂靠在"青浦区建管委"组织下，管辖区域为全区。

说明：
- 普通审核员和科室负责人的数据范围受 org_id 限制，只能看到挂靠组织名下的备案。
- 街道领导和区级领导的数据范围受 jurisdiction_area 限制，可以看到整个管辖区域内的全部备案，不受 org_id 限制。
- 所有角色的可审核工程类型和作业场所，都来源于组织模板继承或自定义配置。

### 4.3 备案自动分配逻辑

备案提交后，系统按以下逻辑自动分配：

步骤1：提取备案信息
- jurisdiction_area = 用户选择的街道社区中的街道/区域部分
- project_type_name = 备案时选择的工程类型中文名称（如"房屋建筑"）
- work_place = 备案时选择的作业场所中文名称（如有）

步骤2：匹配组织
- 查询 organization 表，找到 jurisdiction_area = 备案管辖区域 且 default_auth_types 包含 project_type_name 的组织列表。
- 如果找到多个组织，⚠️ 待确认：选择逻辑是什么？（建议：按组织名称排序取第一个，或给所有匹配组织都分配？）

步骤3：匹配人员
- 在匹配到的组织内，查询 user_org_relation，找到有该工程类型权限的审核员列表。
- 过滤条件：org_id = 匹配到的组织ID，status=1，role_type=1，且可审核工程类型包含 project_type_name。
- 如有作业场所字段，按备案的工程大类分别校验：
  - 小型工程/住宅装修：校验人员的work_places_small（或继承organization.default_work_places_small）是否包含备案的作业场所
  - 零星作业：校验人员的work_places_scattered（或继承organization.default_work_places_scattered）是否包含备案的作业场所
- 如果找到多个人员，⚠️ 待确认：分配逻辑是什么？（建议：轮流分配/负载均衡/分配给所有人？当前系统是怎么做的？）

步骤4：更新备案记录
- project_record.org_id = 匹配到的组织ID
- project_record.assignee_id = 匹配到的人员ID（或NULL如果未匹配到具体人员）

## 5. 组织管理后台页面设计

### 5.1 组织管理页面【改造】

页面路径：后台管理 → 组织管理（企业中心 → 监管单位）

改造内容：

1. 添加/编辑组织弹窗
   - 机构编码（必填）
   - 组织名称（必填）
   - 管辖区域（必填，下拉选择，数据源=现有区域管理模块中的区域列表）
   - 可审核的工程类型（可多选，共23类）
     - 小型工程（12类）：①房屋建筑 ~ ⑫人防工程，既有建筑修缮细分为住宅类/非住宅类
     - 住宅室内装饰装修工程（1类）
     - 零星施工作业（10类）：①空调安装等 ~ ⑩其他风险作业
     - 支持按大类全选/取消全选
   - 小型工程可管理作业场所（可多选，12项通用场所）
     - 全部场所 / 商业综合体 / 工业企业 / 医疗卫生机构 / 教育机构 / 文化场馆 / 体育场馆 / 党政机关 / 养老机构 / 区属国企 / 住宅小区 / 区管交通设施 / 公共绿地公园环卫
   - 零星作业可管理作业场所（可多选，12项通用场所，与小型工程独立配置）
     - 同上12项场所，组织可按实际管辖范围独立勾选

2. 组织列表
   - 表头：组织名称、机构编码、管辖区域、可审核工程类型、可管理作业场所、管理成员、创建时间、操作
   - 可审核工程类型列展示规则（方案B：固定截断+tooltip）：
     - 单元格最大高度显示2行（约52px），超出用"..."截断，鼠标悬停tooltip显示完整列表。
     - 全选时显示绿色"全部工程类型"标签，不展开。
     - 既有建筑修缮勾选子类时，显示"既有建筑修缮(住宅)"或"既有建筑修缮(非住宅)"。
     - 部分勾选且数量多（超过2行容量）时，按优先级展示前N个+"..."，tooltip展开全部。
   - 可管理作业场所列展示规则：
     - 显示小型工程场所与零星作业场所的并集（去重），tooltip中标注来源。
     - 两套配置完全相同时正常展开标签；两套都选"全部场所"时只显示一个绿色"全部场所"标签。

   - 操作列：编辑、同步权限给人员

3. 批量同步权限工具
   - 按钮位置：组织列表 → 操作列 → "同步权限给人员"
   - 点击后弹出确认框：
     - 提示：将把组织【白鹤镇-规保办】的默认授权模板（含工程类型、小型工程作业场所、零星作业作业场所）同步给该组织下所有人员，是否覆盖个人自定义权限？
     - 选项：【仅同步继承模式的人员】/【覆盖全部（含自定义）】

     - 选项2：更新所有人员，同时将auth_mode重置为1（继承）。同步内容同上。

### 5.2 人员管理页面【改造】

2. 添加/编辑人员

     - 权限模式：

## 6. 备案流转与跨街镇分派

### 6.1 正常备案流转

用户提交备案 → 系统提取jurisdiction_area和project_type_name → 自动匹配组织和人员 → 备案进入待审核状态 → 分配给assignee_id的人员收到待办。

匹配不到组织时的处理：
- 如果按管辖区域 + 工程类型未匹配到任何组织，备案状态设为"待分配"（新增状态码），系统管理员或区级领导可以看到并手动分配。
- ⚠️ 待确认：是否需要给系统管理员发送提醒？

### 6.2 跨街镇分派流程

触发条件：
- 备案处于待审核状态
- 操作人有分派权限（⚠️ 待确认：谁可以发起分派？仅原街镇人员？还是区级以上都可以？）

操作流程：

步骤1：发起分派
- 操作人在备案详情页点击"转派"按钮
- 弹出分派弹窗：
  - 原管辖区域（只读，显示当前jurisdiction_area）
  - 目标管辖区域（必填，下拉选择，数据来源=现有区域管理模块，排除原区域）
  - 分派原因（选填，文本输入，限200字）
- 点击"确认分派"

步骤2：系统校验
- 校验目标区域是否有组织配置了该备案的project_type_name权限。
- 如果无 → 提示"目标区域无可处理该工程类型的组织，请重新选择"
- 如果有 → 在project_assign_record中创建记录，assign_status=1（待确认）

步骤3：发送待办给目标区域
- 系统查询目标区域下有该project_type_name权限的所有人员。
- 给这些人员的待办列表推送一条"分派确认"待办。
- 待办内容显示：备案基本信息（工程名称、工程类型、原区域、分派原因）。

步骤4：目标区域人员确认
- 人员点击待办进入"分派确认"页面：
  - 展示备案基础信息（只读）
  - 展示原区域和分派原因
  - 操作按钮：【确认接收】/【拒绝接收】
- 点击【确认接收】：
  - 校验该人员是否仍有权限处理该工程类型（防止权限已变更）
  - 更新project_record：jurisdiction_area=目标区域，org_id=确认人所属组织，assignee_id=确认人ID
  - 更新project_assign_record：assign_status=2，confirm_time=当前时间，confirm_user_id=确认人ID
  - 给原区域发起人发送"分派已接收"通知
- 点击【拒绝接收】：
  - 弹出输入框，必填：拒绝原因
  - 更新project_assign_record：assign_status=3，reject_reason=输入内容
  - 给原区域发起人发送"分派被拒绝"通知，附带拒绝原因
  - 备案恢复到分派前的状态（jurisdiction_area=original_jurisdiction_area）

步骤5：多次分派
- 如果分派被拒绝，原街镇可以再次发起分派。
- 一个备案可以有多条分派记录，但只有最后一条生效的记录决定当前归属。

### 6.3 分派相关的备案状态

新增备案状态：
- 6=待分派确认：已发起跨街镇分派，等待目标街镇人员确认

状态流转：
- 待审核 → 发起分派 → 待分派确认 → 确认接收 → 待审核（归属新街镇）
- 待审核 → 发起分派 → 待分派确认 → 拒绝接收 → 待审核（恢复旧街镇）

## 7. 统计维度设计

### 7.1 区级统计视角

适用用户：区级领导（role_type=4）

统计指标：
1. 全区备案总量（按时间维度：今日/本周/本月/本年/累计）
2. 各区域备案数量分布（柱状图/表格）
   - 分组维度：管辖区域名称（街镇/园区）
   - 指标：备案数量、待审核数量、已通过数量、已驳回数量
3. 各工程类型分布（饼图）
   - 分组维度：工程类型中文名称（房屋建筑、交通工程、既有建筑修缮...）
   - 指标：备案数量
4. 各区域审核效率（表格）
   - 分组维度：管辖区域名称
   - 指标：平均审核时长、待审核积压数量、审核通过率

数据权限：可看全区所有备案。

### 7.2 街镇级统计视角

适用用户：街道领导（role_type=3）、科室负责人（role_type=2）、普通审核员（role_type=1）

统计指标：
1. 本区域备案总量（按时间维度）
2. 各科室/组织备案数量分布（柱状图/表格）
   - 分组维度：组织名称
   - 指标：备案数量、待审核数量、已通过数量、已驳回数量
3. 各工程类型分布（饼图）
   - 分组维度：工程类型中文名称
4. 各审核员工作量（表格，仅街镇级和科室负责人可见）
   - 分组维度：审核员姓名
   - 指标：审核数量、待审核积压、平均审核时长
5. 各作业场所分布（饼图/表格）
   - 分组维度：作业场所名称
   - 指标：备案数量

数据权限：
- 街道领导：可看本区域所有组织的备案。
- 科室负责人：可看本科室所有备案。
- 普通审核员：只能看分配给自己的备案，不参与统计视图。

### 7.3 按工程类型统计（改动3）

在以上统计视图中，均支持按工程类型（中文名称）筛选。

例如：区级人员可以筛选"只看房屋建筑"，查看各区域的房屋建筑备案分布。

### 7.4 按作业场所统计（改动6）

在以上统计视图中，增加按作业场所分组：
- 分组维度：work_place字段（中文名称）
- 指标：备案数量
- 适用场景：区级指导单位可按作业场所筛选查看对应数据。

## 8. 与6条业务改动的对应关系

### 改动1：增加虚拟园区

对应本章设计：

- 备案表单中的"街道社区"选择器，虚拟园区的社区列表为空或显示"不适用"。
- 备案流转时，虚拟园区与实体街镇使用同样的匹配逻辑。

### 改动2：备案指定分派给其他街镇

对应本章设计：
- 第6章"跨街镇分派流程"完整覆盖。
- 分派粒度为街镇级别，系统自动匹配目标街镇内有权限的人员。
- 被分派人需确认接收后方可审核。
- 分派后备案归属目标街镇，统计口径随之变更。

### 改动3：统计和分配权限按工程类型划分到对应审核员头上

对应本章设计：
- 第4章"权限计算逻辑"通过project_type_name（中文工程类型名称）实现工程类型级授权。
- 第5章"组织管理"通过default_auth_types（组织默认授权模板）和人员继承实现。
- 第7章"统计维度"支持按工程类型分组统计。
- 备案自动分配时，按project_type_name匹配有权限的人员。

### 改动4：小型工程和应急大部分不要居委会，只有小区区域内、农村自建房的才要居委会

对应本章设计：
- 此改动属于表单校验规则，与权限体系无直接关系。
- 在备案表单的基础信息Tab中，"社区"字段的必填性改为条件必填：
  - 小型工程 + 工程类型="农民自建或翻建低层住宅" → 社区必填
  - 小型工程 + 工程地址判定为"小区区域内" → 社区必填
  - 其他小型工程 → 社区选填/隐藏
- ⚠️ 待确认："小区区域内"的判定标准是什么？"应急"具体指什么？

### 改动5：零星作业不需要审核但对于特定工程需要发送对应的待办消息

对应本章设计：
- 此改动属于审核流程规则，与权限体系部分相关。
- 审核流程改造：
  - 零星作业（project_type=3）提交后，根据条件判断是否免审：
    - 免审条件：不满足"需要待办"的条件
    - 需要待办的条件：施工时间>24小时 AND 作业类型∈{前三个}
  - 免审的备案直接变为"已通过"状态，生成备案回执。
  - 需要待办的备案：状态变为"已通过"（免审），同时系统发送待办消息给对应街镇有权限的人员。
- 待办消息内容："XX街镇XX备案（作业类型XX）施工时间超过24小时，请线下实际查看"
- 待办消息推送方式：⚠️ 待确认（随申办站内信？短信？内部政务平台？）

### 改动6：零星作业新增作业场所

对应本章设计：
- project_record表新增work_place字段（中文名称）。
- 备案表单中，零星作业的基础信息Tab新增"作业场所"字段（选择器）。
- 条件显示：作业类别≠⑦（住宅区/公共场所绿化枝叶修剪）时才显示此字段。

- 统计报表中支持按作业场所分组（第7章）。
- 大屏统计：区级指导单位按作业场所筛选查看对应数据。

## 9. 数据迁移方案

### 9.1 组织表改造迁移

1. 保留"管辖区域"字段，继续使用现有区域管理模块。
2. 新增字段：default_auth_types（默认可审核工程类型，中文名称列表，覆盖23类）、default_work_places_small（小型工程默认可管理作业场所）、default_work_places_scattered（零星作业默认可管理作业场所）。
3. default_auth_types根据Excel组织汇总建议表或现有组织授权数据初始化，将条款编码映射为对应的中文工程类型名称。
4. default_work_places根据组织的实际管辖范围初始化，一般实体街镇组织可配置"全部场所"。

### 9.2 人员组织关联表扩展迁移

1. 新增字段：role_type、auth_mode、auth_types、work_places、is_primary。
2. 现有数据迁移：
   - role_type：全部设为1（普通审核员），后续人工调整为科室负责人。
   - auth_mode：全部设为1（继承组织模板）。
   - auth_types：全部设为NULL（因为是继承模式）。
   - work_places_small：全部设为NULL（因为是继承模式）。
   - work_places_scattered：全部设为NULL（因为是继承模式）。
   - is_primary：每人保留一条记录设为1，其他设为0（如果有多个组织的话）。

### 9.3 备案表改造迁移

1. 新增字段：original_jurisdiction_area、assignee_id、work_place。
2. 现有备案的jurisdiction_area保持不变（已根据用户选择的区域填充）。
3. original_jurisdiction_area = jurisdiction_area（历史数据默认原始区域=当前区域）。
4. assignee_id根据现有分配逻辑回填（⚠️ 待确认：当前系统是否已有assignee_id字段或类似字段？）。
5. work_place为空（历史数据无此字段）。

## 13. 实施建议

### 13.1 实施阶段划分

**第一阶段：数据模型改造（1-2周）**

- 改造project_record表（新增original_jurisdiction_area、assignee_id、work_place）
- 新建project_assign_record表
- 数据迁移

**第二阶段：后台管理页面改造（1-2周）**
- 改造组织管理页面（添加工程类型授权、作业场所授权、批量同步工具）
- 改造人员管理页面（添加4种角色选择、权限模式、自定义权限配置）

**第三阶段：业务逻辑改造（2-3周）**

- 改造备案列表查询（按新权限模型过滤）
- 实现各级统计视角
- 逐条实现6条业务改动

### 13.2 风险点

1. **数据迁移风险**：现有组织表中的"管辖区域"字段数据质量需要提前检查，确保区域名称与现有区域管理模块一致。
2. **权限切换风险**：新权限模型上线时，需要确保所有人员的权限计算正确，建议上线前做全面测试。
3. **性能风险**：备案列表查询增加了多表关联和权限计算，需要对查询做性能优化（建议加索引、做权限缓存）。
4. **用户习惯风险**：后台管理页面改造较大，需要给运营人员做培训和操作手册。
