# 市局小型工程系统对接差异处理方案（产品视角）

版本：V1.0
日期：2026-04-24
性质：🤖 AI草稿，待人工确认

---

## 一、背景与目标

### 1.1 背景

市局要求青浦区小型工程备案数据同步到市级系统。市局对接文档中定义了57个字段的接口规范，其中：
- 部分字段青浦系统已有，可直接同步
- 部分字段值域不同，需要做转换映射
- 部分字段青浦系统完全没有，需要新增采集

### 1.2 市局核心要求（原文）

> 必须按照文档里面的要求提供、没有的字段需要增加、值不同的需要区里面转换成我们这边的值或者区系统兼容转换成我们这边的值。

翻译为产品语言：
1. **接口必填字段** → 系统必须有，没有的要补
2. **值域不同的字段** → 系统要做转换规则（要么区系统存储市局值，要么同步时做映射）
3. **非必填但缺失的字段** → 产品需要判断是否需要新增采集

## 三、需要你决策/确认的六类核心问题

### 问题一：工程类型只有两类，我们12类怎么归？

**市局接口**：工程类型只有 1房屋建筑 / 2其他

**我们系统**：12种工程类型

**需要你决策**：

| 我们系统的工程类型 | 建议归入市局"房屋建筑" | 建议归入市局"其他" |
|------------------|---------------------|-------------------|
| ①新建/改建/扩建房屋建筑 | ✓ | |
| ②交通工程 | | ✓ |
| ③供水排水/水务工程 | | ✓ |
| ④园林绿化 | | ✓ |
| ⑤既有建筑修缮 | | ✓ |
| ⑥非住宅装饰装修 | | ✓ |
| ⑦加装电梯 | | ✓ |
| ⑧临时设施/临时建筑 | | ✓ |
| ⑨农民自建/翻建低层住宅 | ✓ | |
| ⑩农业设施 | | ✓ |
| ⑪农村基础设施 | | ✓ |
| ⑫改建/扩建人防工程 | ？ | ？ |

⚠️ **待你确认**：⑫人防工程归哪一类？建议找市局确认"人防工程"是否算"房屋建筑"。

---

### 问题二：建设性质我们没有这个字段，怎么补？

**市局接口**：建设性质必填，四选一：1装饰装修 / 2改建 / 3扩建 / 4新建

**我们系统**：没有这个字段，但部分工程类型名称里包含了类似信息

**三种处理方案，你选一种**：

| 方案 | 做法 | 优点 | 缺点 | 工作量 |
|-----|------|------|------|--------|
| A. 从工程类型自动推导 | ①→新建，②→？，③→新建，④→新建/改建，⑤→改建，⑥→装饰装修，⑦→改建，⑧→新建，⑨→新建，⑩→新建，⑪→新建，⑫→改建 | 用户无感知，不改备案流程 | ②交通工程、⑩农业设施等无法准确推导；推错了责任在系统 | 小 |
| B. 备案时增加选择项 | 在备案页面增加"建设性质"选择，用户手动选 | 数据准确 | 增加用户填写负担；历史数据无法补 | 中 |
| C. 组合方案 | 能推导的自动填，不能推导的让用户确认/修改 | 兼顾体验和准确性 | 开发稍复杂 | 中 |

⚠️ **建议**：选C。自动填好大部分，让用户改（如果系统推的不对）。

---

### 问题三：身份类型和建设单位身份，两个维度怎么对应？

**市局接口**：身份类型（sflx）三选一：1业主 / 2建设单位 / 3即是业主又是建设单位

**我们系统**：建设单位身份二选一：产权人 / 承租人

**问题本质**：这是两个不同的分类维度
- 市局区分的是"业主"和"建设单位"（法律主体角色）
- 我们区分的是"产权人"和"承租人"（物权关系）

**需要你确认映射关系**：

| 我们的值 | 市局可能的对应 | 问题 |
|---------|-------------|------|
| 产权人 | → 1业主 或 3即是业主又是建设单位 | 什么时候是"即是"？ |
| 承租人 | → 2建设单位 ？ | 承租人是否等于"建设单位"？ |

⚠️ **建议向外确认**：直接问市局——我们的"产权人/承租人"分别对应你们"业主/建设单位/即是"中的哪一个？

---

### 问题四：街道办编码缺失，怎么解决？

**市局接口**：要求传"街道办"编码（jdb），字典表里有各区编码，但**唯独缺少青浦区**。

**我们系统**：有所属街镇，但用的是街镇名称。

**三种处理方案**：

| 方案 | 做法 | 风险 |
|-----|------|------|
| A. 让市局补充青浦区的jdb编码 | 向市局要青浦11个街镇对应的编码 | 最规范，但依赖市局响应速度 |
| B. 用街道编码（szjd）代替 | 字典3.6里有青浦各街镇的标准编码（12位） | 市局明确区分了jdb和szjd两个字段，可能不接受 |
| C. 先传空值或默认值 | 接口调用时jdb传空或传一个占位值 | 市局接口可能校验不通过 |

⚠️ **强烈建议**：选A。直接找市局对接人要青浦区的jdb编码表。这是他们的文档缺失，应该由他们补充。

---

### 问题五：安全管理人员我们没有，能复用施工负责人吗？

**市局接口**：要求采集"施工单位安全管理人员"的姓名、身份证、手机号（3个必填字段）

**我们系统**：没有这个字段，只有"施工负责人"信息

**需要你决策**：

| 方案 | 做法 | 风险 |
|-----|------|------|
| A. 复用施工负责人信息 | 安全管理人员的姓名/身份证/手机号 = 施工负责人的对应信息 | 市局可能发现数据重复，或实际业务中安全管理人员和施工负责人不是同一人 |
| B. 新增采集字段 | 在备案流程中增加"安全管理人员"信息录入 | 增加用户填写量；但数据准确合规 |

⚠️ **建议**：
- 先问市局：小型工程中安全管理人员和项目负责人是否可以是同一人？
- 如果市局说可以→选A，系统做复制
- 如果市局说不行→选B，需要在施工单位信息区域新增三个字段

---

### 问题六：涉及情形是多选变单选，怎么处理？

**市局接口**："是否涉及以下情形"（sjqx）是单选字段，6个选项：动火/基坑/有限空间/起重吊装/变动主体结构/不涉及

**我们系统**：是多个独立开关（动火作业、动土作业、拆改作业、其他特种作业中包含起重和有限空间、变动建筑主体结构）

**问题**：用户可能同时开了多个开关，但市局只能传一个值。

**需要你决策优先级规则**：

建议优先级（从高到低）：
1. 变动建筑主体结构和承重结构（最严重）
2. 起重吊装
3. 有限空间作业
4. 动火作业
5. 动土作业（基坑）
6. 拆改作业
7. 不涉及（全关时）

⚠️ **需要你确认**：这个优先级排序是否合理？或者干脆按"开关开启数量"判断：只开1个直接传，开多个传最严重的那个？

## 四、需要新增采集的字段清单（对备案流程的影响评估）

以下字段青浦系统完全没有，需要在备案流程中新增。我按对流程的影响程度做了分级：

### 高影响（改动备案页面）

| 字段 | 用户可见？ | 录入方式 | 对流程的影响 |
|-----|----------|---------|------------|
| 建设性质 | 是 | 选择器（4选1） | 增加一个选择步骤 |
| 是否属于人员密集场所 | 是 | 开关或选择器 | 增加一个步骤，可能后续联动场所分类 |
| 工程内容（工程类型为"其他"时必填） | 是 | 文本输入 | 部分工程类型需要多填一项 |
| 消防设施类型（涉及消防设施变动时必填） | 是 | 多选（13种） | 涉及消防设施变动的备案需要补充选择 |
| 建设规模 | 是 | 文本输入 | 增加一个文本框 |
| 主要施工位置、范围和内容 | 是 | 文本输入 | 增加一个文本框 |
| 施工单位安全管理人员信息（3个字段） | 是 | 文本输入 | 如不复用施工负责人，需新增一个人员信息区域 |
| 保单号 | 是 | 文本输入/OCR | 在保单上传环节增加保单号输入 |

### 中影响（可隐藏或自动处理）

| 字段 | 用户可见？ | 处理方式 |
|-----|----------|---------|
| 项目编号 | 否 | 系统自动生成 |
| 工程类型映射 | 否 | 系统根据用户选的12类自动转 |
| 区域编码 | 否 | 固定传"QP" |
| 街道编码 | 否 | 根据用户选的街镇自动映射 |
| 办理对象类型 | 否 | 根据"单位/个人"自动映射 |
| 证件类型 | 否 | 默认"身份证" |

### 低影响（非必填，可暂不处理）

| 字段 | 是否必填 | 建议 |
|-----|---------|------|
| 设计单位资质类型 | 否 | 默认值"工程设计" |
| 设计单位资质等级 | 否 | 如需准确，让用户选择或OCR识别 |
| 消防技术咨询服务机构（6个字段） | 否 | 建议先问市局小型工程是否真的涉及 |

## 五、需要你向外（市局）确认的问题清单

以下问题需要产品或业务负责人直接向市局对接人确认，建议整理后发一封邮件或微信：

| 序号 | 问题 | 为什么必须确认 |
|-----|------|-------------|
| 1 | 青浦区各街镇对应的"街道办"（jdb）编码是多少？字典3.14里缺少青浦区。 | jdb是必填字段，没有编码无法对接 |
| 2 | 人防工程（⑫）属于"房屋建筑"还是"其他"？ | 影响工程类型映射 |
| 3 | "产权人"对应"业主"还是"建设单位"？"承租人"呢？ | 影响身份类型映射 |
| 4 | 安全管理人员可以和施工负责人（项目负责人）是同一人吗？ | 决定是否需要新增3个采集字段 |
| 5 | 小型工程是否涉及"消防技术咨询服务机构"？ | 如一般不涉及，可暂不新增6个字段 |
| 6 | 建设规模（gcnrsm）具体指什么？是面积还是其他描述？ | 决定字段类型和提示文案 |
| 7 | 涉及情形多选时，你们希望按什么优先级传？ | 影响多开关合并规则 |
| 8 | 项目编号（xmbh）有格式要求吗？我们用区系统内部ID可以吗？ | 决定编号生成规则 |
| 9 | 历史已备案的数据需要 retroactive（追溯）同步吗？还是只同步新备案？ | 影响数据范围和工作量 |
| 10 | 住宅装修和零星作业的数据也需要同步吗？ | 对接文档只写了"小型项目事项"，范围需明确 |

## 六、对内（开发团队）的需求拆分建议

等你确认了上述问题后，建议按以下模块给开发提需求：

### 需求包1：字典映射模块（无页面改动，纯后端）

- 工程类型映射表（12类→2类）
- 街镇编码映射表（名称→12位编码）
- 街道办编码映射表（等市局回复后补充）
- 区域固定值（QP）
- 办理对象类型映射（单位→企业，个人→个人）
- 是否映射（开→1，关→2）
- 涉及情形优先级规则

### 需求包2：备案流程字段新增（有页面改动）

根据你的决策结果，可能包括：
- 建设性质选择器（如选方案B或C）
- 人员密集场所开关/选择
- 工程内容文本框（条件显示）
- 消防设施类型多选（条件显示）
- 建设规模文本框
- 主要施工位置/范围/内容文本框
- 安全管理人员信息区域（如不复用）
- 保单号输入框

### 需求包3：市局接口对接模块（纯后端）

- 获取安全令牌（2.1）
- 保存安全备案信息（2.2）
- 提交（2.3）
- 接收（2.4）
- 办理（2.5）
- 同步时机：建议备案审批通过后自动触发

### 需求包4：项目编号生成规则（系统配置）

- 定义编号规则
- 确保唯一性

## 八、风险提醒

| 风险 | 可能性 | 影响 | 应对 |
|-----|--------|------|------|
| 市局回复慢，jdb编码迟迟不给 | 中 | 阻塞开发 | 先并行推进其他模块；如市局超过1周不回复，先用自己的街镇编码试探 |
| 新增字段导致备案流程变长，用户投诉 | 中 | 用户体验下降 | 尽量自动填充、减少用户输入；非必填的字段不强制 |
| 映射规则推错，市局数据不准确 | 低 | 数据质量问题 | 上线初期人工抽查同步数据，建立校验机制 |
| 历史数据补录工作量大 | 中 | 资源投入 | 先确认市局是否要求追溯；如不要求，只同步新数据 |
| 住宅装修/零星作业也需要同步 | 中 | 范围扩大 | 提前向市局确认对接范围 |

## 九、你可以马上做的三件事

1. **发问题清单给市局**（复制第五章的10个问题）
2. **内部确认备案流程接受度**——评估"增加7-10个字段"对现有备案流程的影响，判断哪些可以自动填充、哪些必须用户填
3. **确认对接范围**——住宅装修和零星作业是否在本次对接范围内？历史数据是否需要补录？

等你确认了这些问题，我可以帮你输出正式的PRD。
