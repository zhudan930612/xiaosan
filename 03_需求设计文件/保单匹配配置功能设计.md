## 功能单元 1：保单要求配置（管理后台）

> 💡 **合并说明**: 本功能单元完成后,将"功能点描述"章节合并到 [管理后台模块.md] 中。

### 功能概述

支持辖区管理员（区级/街道级）在管理后台配置各街道、各类工程的保单要求模式：关保险/开保险非强制/开保险强制。配置即时生效，影响新发起的备案流程。

#### 配置数据结构

```
保单配置表 (policy_requirement_config)
├── id: 主键
├── streetCode: 街道编码（如：xyjd-夏阳街道）
├── streetName: 街道名称
├── projectType: 工程类型（small/renovation/scattered）
├── projectTypeName: 工程类型名称
├── policyMode: 保单模式（off/optional/required）
├── createTime: 创建时间
├── updateTime: 更新时间
├── operatorId: 操作人ID
├── operatorName: 操作人姓名
└── version: 版本号（用于并发控制）

唯一索引：streetCode + projectType
```

**默认配置策略**：
- 新街道初始化：默认全部工程类型为"开保险（强制）"
- 符合现有政策：小型工程强制，住宅装修非强制，零星作业关保险

#### 业务规则

**触发条件**
- 用户选择"街道社区"字段后
- 用户修改已选择的"街道社区"字段时

**处理逻辑**
1. 根据选择的街道编码查询保单配置表
2. 根据当前备案的工程类型确定保单模式
3. 动态控制"匹配保单"Tab的显示/隐藏和强制校验

**边界情况**
- 街道无配置记录：使用默认配置（小型工程强制，其他非强制）
- 用户选择街道后修改工程类型：重新查询配置并更新控制逻辑
- 用户从草稿恢复备案：根据当前保存的街道查询最新配置
- ⚠️ 待确认：已提交备案在审核期间，配置变更是否影响？

#### 保单要求配置页

**入口路径**:
- 管理后台 → 系统配置 → 保单要求配置

**页面名称**: "保单要求配置"页面

**页面功能**:

**1. 配置说明区域**:
   - 说明文字：设置各街道、各类工程的保单要求模式
   - 图例说明：强制/非强制/关保险的图标含义

**2. 配置表格**:
   - 行：街道列表（夏阳街道、盈浦街道、香花桥街道、朱家角镇...）
   - 列：工程类型（小型工程、住宅装修、零星作业）
   - 单元格：下拉选择框
     - 选项1：关保险
     - 选项2：开保险（非强制）
     - 选项3：开保险（强制）
   - 单元格颜色标识（可选）：
     - 强制：红色背景
     - 非强制：黄色背景
     - 关保险：绿色背景

**3. 批量操作**:
   - 全选按钮：选择所有街道
   - 批量设置：对选中的街道批量设置某一工程类型的模式

**4. 操作按钮**:
   - 保存：保存所有修改，即时生效
   - 重置：恢复上次保存的配置
   - 恢复默认：恢复系统默认配置（小型工程强制，其他非强制）

**业务规则**:
- 配置保存后即时生效
- 新发起的备案按最新配置执行
- 已提交审核的备案不受影响（按提交时的配置执行）
- 所有操作记录操作日志（操作人、时间、修改内容）
- 支持并发控制（乐观锁，防止多人同时修改冲突）

## 功能单元 2：备案流程动态控制

> 💡 **合并说明**: 本功能单元完成后,将"功能点描述"章节合并到 [备案管理模块.md] 中。

### 功能概述

备案流程根据用户选择的街道社区和工程类型，动态查询保单配置，控制"匹配保单"Tab的显示/隐藏和强制校验逻辑。

---

### 业务流程

#### 动态控制流程图

```
用户进入备案申报 → 基础信息Tab
      ↓
用户选择"街道社区"
      ↓
触发配置查询：streetCode + projectType → 保单配置表
      ↓
根据返回的policyMode执行不同逻辑
      ↓
┌─────────────────────────────────────────────────────────┐
│ policyMode = 'off'（关保险）                            │
│   • Tab栏只显示3个：基础信息/工程信息/安全信息          │
│   • 隐藏"匹配保单"Tab                                    │
│   • 提交时跳过保单校验                                   │
└─────────────────────────────────────────────────────────┘
      │
      ├─────────────────────────────────────────────────────┐
      │                                                     │
      ▼                                                     ▼
┌─────────────────────────────────────────┐   ┌─────────────────────────────────────────┐
│ policyMode = 'optional'（非强制）       │   │ policyMode = 'required'（强制）         │
│   • Tab栏显示4个，含"匹配保单"          │   │   • Tab栏显示4个，含"匹配保单"          │
│   • 显示保单匹配方式选择                 │   │   • 显示保单匹配方式选择                 │
│   • 提交按钮不校验保单是否填写           │   │   • 提交按钮强制校验保单                 │
│   • 用户可跳过保单直接提交               │   │   • 未填写保单提示"请完成保单匹配"       │
└─────────────────────────────────────────┘   └─────────────────────────────────────────┘
```

---

#### 关键时序

```
用户选择街道社区
      ↓
前端请求：GET /api/policy/config?streetCode=xxx&projectType=xxx
      ↓
后端查询保单配置表
      ↓
返回配置：{ policyMode: 'off' | 'optional' | 'required' }
      ↓
前端根据policyMode更新UI状态
      ↓
├─ off：隐藏匹配保单Tab，更新进度指示器为3步
├─ optional：显示匹配保单Tab，提交按钮不禁用
└─ required：显示匹配保单Tab，提交按钮根据保单状态禁用/启用
```

---

#### 边界情况处理

```
1. 用户选择街道后修改工程类型
   → 重新查询配置，更新Tab显示状态

2. 用户从草稿恢复备案
   → 根据草稿中保存的streetCode重新查询最新配置
   → 按最新配置渲染页面

3. 配置查询失败（网络错误）
   → 使用默认配置（小型工程强制，其他非强制）
   → 提示用户"配置加载失败，使用默认设置"

4. 街道无配置记录
   → 后端返回默认配置
   → 小型工程 → required
   → 住宅装修 → optional
   → 零星作业 → off

5. 用户已填写保单后修改街道
   → 如果新街道policyMode=off：清空已填保单，隐藏Tab
   → 如果新街道policyMode=optional：保留保单，不强制
   → 如果新街道policyMode=required：保留保单，强制校验
```

### 功能点描述

#### 匹配保单Tab动态控制

**触发操作**:
- 用户选择"街道社区"字段
- 用户修改"工程类型"（从草稿恢复时）

**页面名称**: "备案信息填写"页面

**页面功能**:

**1. Tab栏动态渲染**:
   - 根据policyMode决定显示3个Tab还是4个Tab
   - Tab名称：基础信息、工程信息、安全信息、[匹配保单]
   - 进度指示器同步更新（如：步骤 1/3 或 步骤 1/4）

**2. 保单匹配区域控制**:
   - policyMode=off：不显示保单匹配区域
   - policyMode=optional/required：显示保单匹配方式选择

**3. 提交按钮控制**:
   - policyMode=off：正常提交，不校验保单
   - policyMode=optional：正常提交，不强制校验保单
   - policyMode=required：
     - 未填写保单：提交按钮禁用，提示"请完成保单匹配"
     - 已填写保单：提交按钮可用

**4. 配置查询接口**:
   ```
   GET /api/policy/config?streetCode={街道编码}&projectType={工程类型}

   Response:
   {
     "streetCode": "xyjd",
     "streetName": "夏阳街道",
     "projectType": "small",
     "projectTypeName": "小型工程",
     "policyMode": "required", // off | optional | required
     "updateTime": "2026-03-23 10:00:00"
   }
   ```

**业务规则**:
- 街道社区选择后立即查询配置
- 配置变更后页面即时响应
- 保单匹配状态需在页面间保持（切换Tab不丢失）
- 提交备案时前端校验，后端再次校验（防绕过）

**待补充内容**:
1. 配置查询接口的缓存策略（减少重复查询）
2. 配置变更的实时推送机制（是否需要WebSocket？）
