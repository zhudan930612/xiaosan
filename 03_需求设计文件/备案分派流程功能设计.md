## 一、功能概述

备案分派功能实现审核过程中备案归属权的跨街道流转。当审核员查看备案详情时，发现该备案所属街道与当前登录账号所在街道不一致，可将备案转派至正确的目标街道。目标街道接收后需先确认接收，确认无误后方可进行审核操作。

核心目标：解决备案归属街道错误导致的审核效率问题，避免"错街镇"备案长期挂账。

---

## 二、使用角色

| 角色 | 场景 | 操作 |
|------|------|------|
| 街镇审核员（发起方） | 审核备案时发现不归本街道管辖 | 点击"转派"按钮，选择目标街道，提交转派 |
| 街镇审核员（接收方） | 收到其他街道转派的备案 | 查看待确认列表，确认接收或选择"不是本街道"继续转派 |
| 系统 | 自动匹配目标街道有权限的人员 | 转派后自动推送待确认通知给目标街道审核员 |

## 三、完整业务流程

### 3.1 发起转派流程

```
审核员进入备案审核详情页
      ↓
发现备案所属街道与本街道不一致
      ↓
点击底部"转派"按钮
      ↓
进入选择目标街道页面
      ↓
从11个青浦区街镇列表中选择目标街道
      ↓
点击"确认转派"
      ↓
系统校验目标街道与当前街道是否为同一街道
      ↓
校验通过
      ↓
备案状态变更为"转派中"
      ↓
生成转派记录（含转派前街道、目标街道、转派时间、转派人）
      ↓
向目标街道下所有有审核权限的人员推送待确认通知
      ↓
该备案从发起方待审核列表中移除
      ↓
显示"转派成功"结果页
```

### 3.2 确认接收流程

```
目标街道审核员进入项目中心
      ↓
Tab切换到"待确认"状态
      ↓
查看待确认备案列表
      ↓
点击待确认备案进入详情页
      ↓
查看备案基本信息及转派来源信息
      ↓
判断是否为本人街道管辖范围
      ├─ 是 → 点击"确认接收"
      │         ↓
      │       备案状态变更为"待审核"
      │         ↓
      │       生成确认接收记录
      │         ↓
      │       该备案进入本街道待审核列表
      │         ↓
      │       显示"确认接收成功"结果页
      │
      └─ 否 → 点击"不是本街道"
                ↓
              进入选择目标街道页面
                ↓
              重新选择正确街道
                ↓
              点击"确认转派"
                ↓
              备案继续流转至新的目标街道
```

### 3.3 审核流程（确认接收后）

```
目标街道审核员进入待审核列表
      ↓
点击已确认接收的备案进入详情页
      ↓
查看备案详情（显示转派来源信息）
      ↓
点击"驳回"或"通过"
      ↓
完成审核
```

## 四、状态流转

备案审核状态在原有基础上新增以下状态：

```
┌──────────┐    转派      ┌──────────┐   确认接收   ┌──────────┐
│ 待审核   │ ───────────→ │ 转派中   │ ──────────→ │ 待审核   │
│ (正常)   │              │ (待确认) │              │ (已确认) │
└──────────┘              └──────────┘              └──────────┘
      ↑                      │ 不是本街道                  │
      │                      ↓ 继续转派                    │ 驳回/通过
      │                   ┌──────────┐                    ↓
      │                   │ 转派中   │              ┌──────────┐
      │                   │ (新目标) │              │ 已驳回   │
      │                   └──────────┘              │ 已通过   │
      │                                              └──────────┘
      └────────────────────────────────────────────────────┘
                        驳回后重新提交回到待审核
```

状态说明：

| 状态 | 含义 | 可见范围 |
|------|------|----------|
| 待审核 | 正常待审核状态 | 所属街道下所有有权限人员可见 |
| 转派中 | 已发起转派，等待目标街道确认 | 仅目标街道可见（待确认列表） |
| 已确认 | 目标街道已确认接收，等待审核 | 确认街道下所有有权限人员可见 |

## 五、业务规则

### 5.1 转派规则

转派触发条件
- 审核员在审核详情页可随时点击"转派"按钮
- 不强制校验当前街道与备案所属街道是否一致，允许任何审核员发起转派
- 但发起转派时，目标街道不能与当前街道相同

转派操作限制
- 一次只能选择一个目标街道
- 转派后备案从发起方列表中移除，不可撤回
- 转派中的备案，发起方不可再进行驳回或通过操作
- 同一条备案在"转派中"状态下，不可再次发起转派
- V1版本暂不支持批量转派，仅支持单条备案逐条转派

转派原因
- 发起转派时必须填写转派原因，原因文本框限制200字
- 转派原因会记录到转派记录中，供后续追溯查看
- 转派原因仅对审核端可见，申报人（施工方）不可见

### 5.2 目标街道选择规则

可选目标街道范围
- 青浦区全部11个街镇：夏阳街道、盈浦街道、香花桥街道、朱家角镇、练塘镇、金泽镇、赵巷镇、徐泾镇、华新镇、重固镇、白鹤镇
- 若系统支持虚拟园区，则同时包含青浦工业园区、西虹桥商务区等
- 目标街道不能与当前街道相同

选择方式
- 街道列表按名称排序展示
- 支持搜索框按街道名称筛选
- 默认不预选任何街道，需用户手动选择

### 5.3 确认接收规则

确认权限
- 目标街道下所有具有"备案审核"权限的人员均可确认接收
- 第一个确认接收的人员即为该备案的确认人
- 确认后该备案进入"待审核"状态，该街道下所有有权限人员均可审核（有确认权限即等同于有审核权限）
- 确认接收不限定由确认人本人审核，街道内任何有审核权限的人员均可审核

确认操作
- 确认接收：备案状态从"转派中"变为"待审核"，进入确认人所在街道的待审核列表
- 不是本街道：备案保持"转派中"状态，重新进入选择目标街道流程，由当前人员继续转派
- 目标街道选择不限制：接收方继续转派时，目标街道可选择原街道（转回发起方）或其他任意街道

### 5.4 审核规则（转派后）

- 确认接收后的备案，审核流程与正常备案完全一致
- 审核详情页需显示转派来源信息（来源街道、转派时间）
- 审核通过后生成备案回执，归属街道以备案信息中的"所属街镇"字段为准
- 转派记录永久保留，可在转派记录页查看
- 转派全过程在审批记录中留痕，申报人（施工方）不单独发送转派通知

### 5.5 列表展示规则

待确认列表
- 仅展示转派至本街道且状态为"转派中"的备案
- 列表项标注"转派至本街道"及转派时间
- 点击进入详情页，底部显示"确认接收"和"不是本街道"两个按钮

待审核列表（正常）
- 展示本街道下状态为"待审核"的备案（含本街道原始备案和已确认接收的转派备案）
- 已确认接收的转派备案，列表项标注"转派至本街道"

已确认接收的备案进入待审核后
- 列表展示与正常备案一致
- 详情页额外展示转派来源信息条

### 5.6 转派记录规则

记录查询权限
- 每个审核员只能查看与自己相关的转派记录
- 不开放区级管理员全局查看视角，区级不干预转派流程
- 记录按"我发起的"和"我接收的"两个视角分类展示

我发起的
- 展示当前登录人员作为转派人发起的所有转派记录
- 字段：转派时间、目标街道、备案名称、转派原因、当前状态（转派中/已确认/已驳回/已通过）
- 点击备案名称可进入备案详情（只读）

我接收的
- 展示转派至当前登录人员所在街道的所有转派记录
- 字段：转派时间、来源街道、备案名称、转派原因、确认人、确认时间、当前状态
- 点击备案名称可进入备案详情（只读）

记录状态
- 转派中：已发起转派，目标街道尚未确认
- 已确认：目标街道已确认接收
- 已驳回：确认后审核不通过
- 已通过：确认后审核通过

记录保留策略
- 转派记录永久保留，不随备案归档而删除
- 支持按时间范围、街道、状态筛选
- 支持按备案名称搜索

### 6.2 备案状态扩展

在现有备案状态基础上，新增以下状态值：

- 待审核：正常待审核
- 转派中：已发起转派，等待目标街道确认
- 已确认：目标街道已确认接收，等待审核
- 已驳回：审核不通过
- 已通过：审核通过

## 九、决策记录

### 决策1：转派粒度为街道级，不指定具体审核员

**决策内容**：审核员发起转派时，只需选择目标街道，不需要选择具体人员。系统根据目标街道的权限配置自动匹配有审核权限的人员。

**选择方案**：分派到街道，系统自动匹配人员，被分派人确认后生效。

**决策依据/原因**：
- 发起人不需要知道目标街道内部的人员分工，降低使用门槛
- 目标街道接收后需先确认，确认环节本身已经起到了"认领"的作用
- 避免审核员个人排班变动导致分派到具体人却无法处理的问题

### 决策2：目标街道确认接收后方可审核

**决策内容**：备案转派到目标街道后，不直接进入待审核状态，而是先进入"待确认"状态。目标街道审核员确认接收后，才变为"待审核"。

**选择方案**：增加确认接收环节。

**决策依据/原因**：
- 防止转派错误（如选错街道）导致备案进入错误的审核流程
- 给目标街道一个"验货"的机会，确认确实归本街道管辖
- 用户明确要求"别人转派给我，我要确认是我的，我才会进行审核"

### 决策3：转派按钮常驻在审核详情页底部

**决策内容**：审核详情页底部常驻三个操作按钮：驳回、通过、转派，转派与驳回/通过并列。

**选择方案**：三按钮并列布局。

**决策依据/原因**：
- 转派是审核过程中可能随时发生的操作，不需要先判断归属权再显示
- 降低操作门槛，审核员看到不归自己管即可直接转派
- 用户截图明确展示了此布局

### 决策4："不是本街道"允许继续转派

**决策内容**：目标街道确认时，提供"不是本街道"按钮。点击后进入选择目标街道页面，由当前人员继续转派至其他街道。

**选择方案**：允许接收方继续转发。

**决策依据/原因**：
- 解决连续转派错误的问题（A→B→C）
- 避免备案卡在错误的接收方无法处理
- 形成完整的流转链路，最终到达正确的街道

### 决策5：转派功能不可替代，驳回让施工方修改不是等效方案

**决策内容**：用户提出"街道完全可以驳回让施工方修改街道然后再提交，转派的意义是什么？"经分析，两种方案在以下维度存在本质差异，转派功能具有不可替代性。

**驳回让施工方修改 vs 审核员直接转派的对比**：

| 维度 | 驳回让施工方修改 | 审核员直接转派 |
|------|-----------------|---------------|
| 处理时效 | 至少多1-2个工作日（驳回→通知→修改→重新提交→重新排队审核） | 即时流转，目标街道确认后即可审核 |
| 申报人体验 | 施工方收到驳回通知，需重新操作，体验差 | 申报人无感知，审核内部消化 |
| 错误类型 | 仅适合施工方确实填错的情况 | 同时适合系统地址解析错误、跨区域归属争议等审核员更清楚的情况 |
| 数据完整 | 驳回后重新提交，可能重新生成备案编号，记录断裂 | 保留原备案数据，转派记录留痕，数据连续 |
| 时限保障 | 可能超出受理时限（1个工作日） | 在时限内完成纠错，不影响考核 |

**转派的核心不可替代价值**：
1. **内部纠错，不打扰申报人**：街道间管辖范围存在模糊地带（如边界区域、虚拟园区），审核员比施工方更清楚归属，直接内部流转无需让申报人反复折腾
2. **时效保障**：备案有1个工作日受理时限要求，转派可在时限内快速完成纠错，驳回再提交可能超时
3. **责任追溯**：转派记录明确记录"谁转的、从哪转到哪、为什么转"，后续出问题可完整追溯；驳回让施工方改则无法记录这个流转链路
4. **运营效率**：一个街道每天可能收到多条"错街镇"备案，如果每条都驳回让施工方改，沟通成本和重复工作量不可接受

**选择方案**：保留转派功能，与驳回修改方案并存，由审核员根据具体场景选择。

**决策依据/原因**：
- 两种方案适用场景不同，不是替代关系而是互补关系
- 转派解决的是"审核员发现归属错误"的场景，驳回修改解决的是"施工方确实填错"的场景
- 用户已确认转派流程设计（含确认接收环节），该决策仅补充价值说明，不影响已确认的设计方案
