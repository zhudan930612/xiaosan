## 一、文档信息

### 1.1 文档目标

明确备案审核通过后，如何在备案回执右下角新增施工端二维码和监管端二维码，并定义扫码后的落地页面、权限拦截、只读规则和异常处理规则，为后续原型输出和开发实现提供统一口径。

### 1.2 适用范围

适用端：备案回执打印载体、施工端 H5、监管端 H5、随申办 APP 内嵌环境、微信小程序内嵌环境、支付宝小程序内嵌环境

适用对象：施工单位负责人、甲方负责人、施工备案人员、区巡查员、街道巡查员、社区巡查员

适用阶段：需求设计、原型设计前确认、开发前确认

### 1.3 信息来源说明

已确认来源：备案回执右下角原有告示栏位置及新增二维码诉求，来源于用户提供的备案回执截图和访谈结论。施工方扫码后复用施工方工程详情页的结论，来源于 `施工方端功能设计.md`。监管端扫码后复用监管端工程详情页并支持发起巡查的结论，来源于 `安全巡检功能设计.md`。二维码采用普通 URL 形式、两端均为 H5 内嵌多容器运行的结论，来源于用户访谈记录。

## 二、功能概述

### 2.1 业务场景

当前备案回执右下角为“告示栏”文字区域，没有二维码。施工方和监管人员需要先进入系统首页或工程列表，再手动查找对应工程，现场使用效率较低。本次设计希望在备案回执上新增施工端二维码和监管端二维码，让施工方扫码后直接进入该工程的施工方详情页，监管人员扫码后直接进入该工程的监管端详情页，减少查找步骤，提升现场处理效率。

### 2.2 核心价值

| 角色 | 核心价值 |
|------|----------|
| 施工方 | 扫码直达工程详情页，快速查看整改任务并提交整改记录，无需在列表中查找工程 |
| 监管人员 | 扫码直达工程详情页，一键发起巡查登记，减少现场查找工程的时间成本 |
| 监管部门 | 通过备案回执上的二维码强化现场工程与系统工程的一一对应关系 |

### 2.3 功能边界

本次设计包含：备案审核通过后自动生成两个二维码；备案回执右下角二维码展示方案；施工端和监管端扫码后的路由直达能力；扫码后的登录校验、权限拦截、只读控制和异常提示规则。

本次设计不包含：重新设计施工方工程详情页；重新设计监管端工程详情页；为物业巡查员开放监管二维码；接入新的“三合一码”生成接口；改变现有整改、巡查、核销业务本身的处理逻辑。

## 三、业务流程

### 3.1 主流程

备案审核通过后，系统为当前工程生成施工端二维码和监管端二维码，并在备案回执右下角打印展示。施工相关人员扫描施工端二维码后，进入该工程在施工方端的详情页；监管相关人员扫描监管端二维码后，进入该工程在监管端的详情页。两类二维码都以工程唯一编号作为定位参数，页面加载后再根据当前登录态和用户角色判断是否可查看、是否可操作。

### 3.2 关键分支流程

施工方扫码时，未登录用户先完成登录，再回跳工程详情页；已登录但与该工程无关联的账号进入施工方无权限提示页；已登录且为该工程关联账号的用户进入工程详情页，已核销工程以只读模式展示。监管人员扫码时，未登录用户先完成登录，再回跳工程详情页；已登录但工程不在当前用户管辖范围内的账号进入监管端无权限提示页；已登录且具备管辖权限的用户进入工程详情页，已核销工程仅允许查看，不再显示新增巡查主按钮。

### 3.3 流程说明

二维码仅在备案审核通过后生成并展示，审核不通过时不展示二维码区域。二维码长期有效，工程核销后二维码仍可扫码访问，但页面操作能力切换为只读。二维码本身不承载用户身份信息，所有权限判断均在服务端基于登录态和工程关联关系完成。

## 四、核心方案

### 4.1 方案摘要

本次方案在备案回执右下角保留原有告示栏功能的基础上，新增两个二维码展示位，分别服务于施工端和监管端。二维码不再承担首页入口能力，而是直接指向具体工程详情页，实现“扫码即到工程”。

二维码形式采用普通 URL 二维码，而不是三合一码或容器专属二维码。原因是随申办官方不提供当前场景所需的三合一码接口，同时施工端和监管端本身均以 H5 方式嵌入多种容器环境中，普通 URL 更符合现有技术边界。

落地页层面不新建工程详情页，而是复用施工方端和监管端现有工程详情页。这样可以避免页面能力重复建设，也能确保扫码进入与列表进入保持一致的业务规则和展示逻辑。

权限控制层面采用“二维码只负责定位工程、服务端负责拦截权限”的方式。这样既能避免二维码泄露敏感信息，也能让未授权用户在扫码后被明确拦截，不接触工程敏感信息。

### 4.2 页面与能力范围

涉及页面：备案回执页面、施工方工程详情页、监管端工程详情页、施工方无权限提示页、监管端无权限提示页、二维码失效/异常提示页。

涉及能力：二维码生成、二维码打印展示、扫码后工程直达、跨容器登录回跳、服务端权限拦截、核销后只读控制、扫码异常提示。

复用能力：施工方工程详情页复用 `施工方端功能设计.md` 中既有工程详情页；监管端工程详情页复用 `安全巡检功能设计.md` 中既有工程详情页和发起巡查流程；备案回执基础版式复用现有备案管理模块中的回执样式。

### 4.3 方案取舍说明

本次选择两个二维码并行，而不是单一二维码区分角色，是因为回执上原本就存在“施工端”“监管端”的明确使用对象划分，两个二维码的使用场景清晰，用户无需扫码后再进行角色判断。施工端和监管端继续复用各自既有详情页，而不是新建扫码专用页，是因为本次优化的核心在于“入口直达”，而不是“页面重做”。二维码内容采用带工程唯一编号参数的普通 URL，而不采用数据库自增 ID 直出或前端拼接规则，是为了兼顾现有容器技术路径与基础安全要求。

## 五、功能需求描述

> 本章为文档主干。每个页面固定写“入口路径、页面名称、页面功能（页面结构）”，再按需要补充“页面流程”和“需求说明（标注）”。流程和需求说明要分开写，原型如需输出，应直接依据本章内容制作。

### 5.1 备案回执页面

入口路径：备案工程审核通过后生成备案回执，在备案回执打印版式中展示二维码区域。

页面名称：备案回执页面

页面功能（页面结构）：页面沿用现有备案回执主体版式，右下角区域由原有纯文字告示栏调整为“左侧精简告示文字 + 右侧两个二维码展示位”的组合结构。二维码展示位分别标注“施工端”和“监管端”，用于让不同角色在现场直接扫码进入对应工程。

需求说明（标注）：二维码只在备案审核通过后生成并展示，且一张回执固定对应一个工程、两个二维码。备案审核不通过或未通过审核时不展示二维码区域。页面右下角区域保留“告示栏”语义，施工端二维码下方固定标注“施工端”，监管端二维码下方固定标注“监管端”。左侧告示文案可为版式适配做精简，但不能丢失原有告示含义。两个二维码在审核通过后同步生成，分别绑定同一工程唯一编号，但落地地址不同。回执页展示的二维码图片必须来自服务端生成结果，不允许前端临时拼接。工程核销后，纸质回执上的二维码仍保留原样，不做回收或移除。若二维码生成失败，回执生成环节应直接返回异常，不允许展示空白占位二维码。

待补充事项：回执打印尺寸、二维码图片尺寸和告示栏压缩后的版式细节，如需严格落地印刷规范，后续可在原型或附件资料中补充。

### 5.2 施工方工程详情页（扫码进入）

入口路径：施工单位负责人、甲方负责人、施工备案人员通过备案回执上的施工端二维码扫码进入；用户也可能经登录回跳后再次进入同一页面。

页面名称：施工方工程详情页

页面功能（页面结构）：页面不新建，直接复用现有施工方端工程详情页。页面包含顶部状态横幅、整改相关区域、工程信息区域、巡查记录区域、整改记录区域以及底部操作区。扫码进入的主要变化只在于入口从“项目中心列表进入”改为“通过工程编号直达”。

页面流程：
用户扫码
├─ 工程不存在或二维码无法识别 → 进入二维码失效与异常提示页
├─ 未登录或 token 失效 → 进入当前容器登录引导流程，登录成功后回跳当前工程详情页
├─ 已登录但与工程无关联关系 → 进入施工方无权限提示页
└─ 已登录且与工程存在关联关系 → 进入施工方工程详情页
   ├─ 工程状态为停工整改或现场整改 → 优先展示整改信息并提供整改入口
   ├─ 工程状态为正常施工 → 按现有正常施工逻辑展示
   └─ 工程状态为已核销 → 页面只读展示

需求说明（标注）：施工端二维码只服务于该工程的施工方关联人员访问，非关联账号必须被拦截，不得通过扫码获得完整工程信息。扫码进入的页面与施工方端原有工程详情页为同一页面，不额外新建扫码专用页，也不新增新的操作能力。页面加载时以二维码中的工程唯一编号作为唯一查询参数，查询结果与从施工方端“项目中心”进入工程详情页时保持一致。工程状态包括正常施工、现场整改、停工整改和已核销。正常施工时页面按既有规则展示工程信息和可用操作；现场整改和停工整改时优先展示整改信息并提供整改入口；已核销时页面仍可进入，但所有操作入口转为只读。整改记录和巡查记录排序沿用现有页面定义。页面中涉及的工程名称、整改信息、巡查记录、整改记录等字段展示规则沿用 `施工方端功能设计.md` 既有定义。扫码行为本身不生成新的业务记录，但可作为日志事件单独记录。网络异常时提供重新加载入口。

待补充事项：甲方负责人和施工备案人员在系统中的身份字段映射方式，以及施工单位授权管理人员是否纳入同一权限集合，仍需开发和 PM 最终确认。

### 5.3 监管端工程详情页（扫码进入）

入口路径：区巡查员、街道巡查员、社区巡查员通过备案回执上的监管端二维码扫码进入；用户也可能经登录回跳后再次进入同一页面。

页面名称：监管端工程详情页

页面功能（页面结构）：页面不新建，直接复用现有监管端工程详情页。页面包含顶部导航区、工程状态与风险标签区、地图区、工程基本信息区、巡查相关 Tab 区以及底部操作区。扫码进入后的主要变化是直接通过工程唯一编号定位工程，并在满足权限和状态条件时显示“新增巡查”主按钮。

页面流程：
监管人员扫码
├─ 工程不存在或二维码无法识别 → 进入二维码失效与异常提示页
├─ 未登录或 token 失效 → 进入当前容器登录引导流程，登录成功后回跳当前工程详情页
├─ 已登录但工程不在当前用户管辖范围内 → 进入监管端无权限提示页
└─ 已登录且工程在管辖范围内 → 进入监管端工程详情页
   ├─ 工程未核销 → 页面正常展示，可显示【新增巡查】主按钮
   └─ 工程已核销 → 页面只读展示，不显示【新增巡查】主按钮

需求说明（标注）：监管端二维码只面向有管辖权限的监管人员使用，物业巡查员本期不开放。扫码进入的页面与现有监管端工程详情页为同一页面，不新增扫码专用页，也不新增新的监管权限。页面以二维码中的工程唯一编号查询工程数据，同时校验当前登录用户所属组织与工程所属街道、社区的管辖关系。数据加载成功后，详情页内容与从监管端列表进入的内容保持一致。工程未核销且当前用户有管辖权限时，页面按正常监管详情页展示，并允许从底部主按钮发起巡查；工程已核销时，页面仍允许查看，但底部不显示“新增巡查”主按钮。点击“新增巡查”后，直接进入发起巡查页面并自动带入工程信息。工程名称、工程状态、风险等级、施工时间、面积、金额、风险类型标签等字段展示规则，沿用现有监管端工程详情页定义。监管页面中的巡查统计、记录排序、风险信息展示口径全部沿用 `安全巡检功能设计.md` 既有定义。网络异常时提供重新加载入口。

待补充事项：监管端扫码进入时，“物业巡查”Tab 是否仍保持展示但不可操作，还是直接隐藏，仍需产品和开发确认。

### 5.4 扫码确认页（未登录状态）

入口路径：未登录用户或 token 已失效用户扫描施工端二维码或监管端二维码时进入。

页面名称：扫码确认页

页面功能（页面结构）：页面用于在用户未登录时提供明确的上下文提示，告知用户当前正在查看哪个工程，并引导用户在当前容器环境中完成登录。页面结构包含顶部品牌区、工程信息展示区、中部提示文案区、底部登录操作区。

业务规则：

核心原则：扫码确认页只用于处理未登录或 token 失效场景，已登录用户不应进入此页面。页面不直接自动跳转登录流程，而是给用户一个主动点击的入口，避免用户被突然拉去登录而产生困惑。

页面涉及的状态规则：本页面只在“未登录”或“token 已失效”状态下出现；已登录用户直接跳过此页进入权限鉴定和详情页加载流程。

计算口径：无。

字段规则：页面展示完整的工程名称和工程地址，与备案回执上印刷的信息保持一致，用于让用户确认扫码结果正确。例如：工程名称“青浦区夏阳街道XX装修工程”，工程地址“上海市青浦区徐泾镇京华路256弄”。

数据规则：页面加载时仅需根据二维码中的工程唯一编号查询工程基础信息（工程名称、工程地址），不需要加载完整工程详情数据。

交互规则：页面提供【立即登录】主按钮，点击后根据当前运行容器（随申办APP、微信小程序、支付宝小程序、普通浏览器）调起对应的登录流程。登录成功后，系统自动回跳至当前工程详情页。页面同时提供【返回首页】按钮，点击后回到对应端的首页。

异常处理：若查询不到工程信息，直接进入二维码失效提示页，不展示本页面；若登录过程中断或失败，用户回到本页面可再次点击登录；网络异常时提示重新加载。

### 5.5 施工方无权限提示页

入口路径：已登录用户扫描施工端二维码，但账号与当前工程不存在施工方关联关系时进入。

页面名称：施工方无权限提示页

页面功能（页面结构）：页面用于明确告知用户当前账号无权查看该工程，并提供返回首页的明确路径。页面结构包含顶部品牌区、中部提示图标与文案区、底部返回操作区。

需求说明（标注）：本页面只在“已登录但无工程关联权限”的状态下出现，用于明确告知用户当前账号无权查看该工程，并提供返回首页的明确路径。页面只负责提示，不允许透出完整工程详情。主文案统一为“您无权限查看该工程”，辅助文案统一说明“您不是该工程的施工方负责人、甲方负责人或施工备案人员”。页面展示完整的工程名称和工程地址，与备案回执信息保持一致，帮助用户确认是否扫错码。页面默认提供“返回首页”按钮；如后续业务需要，也可增加联系客服或咨询入口。若返回首页路径失效，应至少保证用户可通过系统默认返回机制退出当前页。

### 5.5 监管端无权限提示页

入口路径：已登录监管用户扫描监管端二维码，但当前工程不在其所属组织管辖范围内时进入。

页面名称：监管端无权限提示页

页面功能（页面结构）：页面用于明确告知用户该工程不在其管辖范围内，并提供返回首页的明确路径。页面结构包含顶部品牌区、中部提示图标与文案区、底部返回操作区。

需求说明（标注）：本页面只在“已登录但无管辖权限”的状态下出现，用于明确告知用户该工程不在其管辖范围内，并提供返回首页的明确路径。页面只提示权限范围限制，不展示工程敏感信息，不允许用户在无权限情况下继续尝试访问详情页。主文案统一为“您无权限查看该工程”，辅助文案统一为“该工程不在您的管辖区域内”。页面展示完整的工程名称和工程地址，与备案回执信息保持一致，帮助用户确认是否扫错码。页面默认提供“返回首页”按钮，帮助用户回到监管端首页或工程列表。若页面初始化失败，系统仍应优先返回统一的无权限提示口径，不允许误展示详情页内容。

### 5.6 二维码失效与异常提示页

入口路径：用户扫描的二维码无法识别、对应工程不存在、二维码数据异常，或扫码后页面加载过程中发生不可恢复异常时进入。

页面名称：二维码失效与异常提示页

页面功能（页面结构）：页面用于提示当前二维码无法正常完成工程定位或数据加载失败，并提供重试或返回首页的操作入口。页面结构包含顶部品牌区、中部状态图标与说明文案区、底部操作区。

页面流程：
用户扫码
├─ 工程不存在、二维码内容错误或系统无法识别 → 进入失效提示状态
├─ 页面请求失败或网络异常 → 进入异常提示状态
└─ 工程已核销 → 不进入本页面，直接进入对应详情页只读状态

需求说明（标注）：本页面只用于处理真正无法定位工程或无法完成基本加载的情况，工程已核销不属于二维码失效场景。二维码不存在、二维码内容错误、工程数据不存在时进入失效提示状态；网络异常或接口异常时进入异常提示状态。失效状态下主文案统一为“该工程不存在或二维码已失效”；网络异常状态下主文案统一为“网络连接异常，请检查网络后重试”。系统在确认工程不存在或二维码无法识别后，才进入失效提示页；若只是页面请求失败，应优先归类为异常提示，而不是直接判定二维码失效。页面至少提供“重新加载”或“返回首页”中的一种恢复路径；网络异常时优先提供“重新加载”按钮，失效场景下优先提供“返回首页”按钮。若重复重试仍失败，应保持统一提示口径，不切换到其他业务页面；若用户从异常页返回后重新扫码，系统重新执行完整校验流程。

待补充事项：失效页和异常页是否合并为同一套页面模板，以及文案是否需要按施工端和监管端区分，仍可在原型阶段进一步细化。

## 六、非功能性需求

### 6.1 性能要求

二维码生成应在备案审核通过后的回执生成链路中稳定完成，不允许出现回执已生成但二维码区域为空的情况。扫码后工程详情页的加载性能应与原有工程详情页保持一致，不因新增扫码入口而增加额外加载层级。

### 6.2 安全与权限要求

二维码内容不得包含会话 token、用户身份信息或其他敏感凭证。二维码中的工程编号建议使用不可逆短码或加密标识，避免直接暴露数据库自增 ID。权限校验必须在服务端完成，非授权用户不得通过前端绕过校验进入详情页。扫码行为如进入日志记录，应遵循现有安全与隐私合规要求。

### 6.3 兼容性要求

落地页面需兼容随申办 APP、微信小程序、支付宝小程序及手机浏览器环境。系统需支持根据当前容器环境处理登录态获取与登录后的工程页回跳。页面应适配移动端显示，不额外要求 PC 端布局。

### 6.4 运维与配置要求

二维码图片生成、存储和打印调用链路需具备可排查能力。若后续确认需要记录扫码日志，建议记录扫码时间、二维码类型、工程唯一编号、用户角色、扫码结果等基础字段，用于运营分析和问题追踪。当前是否记录扫码日志及日志保留时长仍待确认。
