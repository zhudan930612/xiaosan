## 一、任务目标

本次原型设计用于补齐大屏首页中的“巡查统揽”能力，并形成与“实时统揽”一致的进入方式和页面层级。

目标不是只补首页两个指标，而是完成一套可演示的巡查监管大屏原型，包括：

1. 首页四大统揽中的“巡查统揽”入口卡片
2. 点击“巡查统揽”后的专题页
3. 街道或工程点击后的右侧弹层
4. 工程独立详情页

## 三、总体设计策略

本次采用“方案 A + 方案 B 结合版”：

### 3.1 外层结构延续实时统揽

巡查统揽入口、专题页层级、返回方式、顶部筛选、左排行右趋势下表格的骨架，尽量与“实时统揽”保持一致。

这样做的目的：

- 让用户理解为同一套大屏系统下的不同专题，而不是一套新页面
- 减少学习成本
- 保持现有视觉资产可复用

### 3.2 内层内容增强为巡查驾驶舱

虽然外层结构复用“实时统揽”，但巡查统揽内部内容不只展示总量，而是增强为巡查监管分析页，重点突出：

- 巡查执行情况
- 工程状态变化趋势
- 隐患发现与整改闭环
- 街道之间的执行差异
- 工程级记录下钻

## 四、页面结构

### 4.1 页面 1：首页巡查统揽入口增强

**目标**

保留首页现有四大统揽布局，只替换“巡查统揽”卡片中的占位内容。

**卡片展示建议**

- 本周巡查覆盖率
- 本周隐患整改率
- 本周巡查次数
- 待整改隐患数

**交互**

- 点击“巡查统揽”卡片，进入“巡查统揽专题页”
- 点击卡片中的指标区域，也进入同一专题页

### 4.2 页面 2：巡查统揽专题页

**设计目标**

作为巡查监管专题大屏，既能总览，也能分析街道差异，并继续下钻到工程详情。

**结构**

1. 顶部操作区
- 返回上一页
- 刷新数据
- 时间维度切换：今日 / 本周 / 近7日 / 本月
- 街道筛选：全部街道 / 指定街道
- 组织维度：全部 / 街道 / 社区 / 物业

2. 第一排核心指标卡
- 巡查次数
- 巡查覆盖率
- 隐患总数
- 隐患整改率
- 待整改隐患数
- 停工整改工程数

3. 中部分析区
- 左侧：街道排名
- 右侧：多指标趋势分析

4. 底部明细区
- 街道巡查数据明细表

### 4.3 页面 3：街道/工程右侧弹层

**用途**

承接专题页中的快速下钻，不直接打断大屏观看流。

**触发方式**

- 点击街道排名项
- 点击街道明细表某一行
- 点击地图中的工程点位

**展示内容**

- 当前街道或工程名称
- 当前统计周期
- 核心指标摘要
- 工程状态分布摘要
- 最新巡查时间
- 待整改隐患数
- 查看详情按钮

### 4.4 页面 4：工程独立详情页

**设计目标**

承接大屏的最终业务下钻，延续小程序“工程详情”的业务结构，但转化为适合大屏查看的桌面信息布局。

**模块结构**

1. 工程基础信息
- 工程名称
- 工程地址
- 所属街道
- 工程类别
- 风险标签
- 当前状态
- 计划完工时间
- 是否超期未核销

2. 地图与定位信息
- 小型地图定位卡
- 工程点位状态

3. 信息记录区
- 备案记录
- 巡查记录
- 整改记录

4. 关键节点摘要
- 最新巡查
- 最新整改
- 最新备案变更

## 五、专题页信息设计

### 5.1 核心指标卡

| 指标 | 含义 |
|---|---|
| 巡查次数 | 当前筛选周期内提交的巡查记录数 |
| 巡查覆盖率 | 当前筛选范围内被巡查过的在建工程占比 |
| 隐患总数 | 当前筛选周期内发现的隐患总数 |
| 隐患整改率 | 当前筛选周期内已完成整改的隐患占比 |
| 待整改隐患数 | 当前未闭环的隐患数 |
| 停工整改工程数 | 当前处于停工整改状态的工程数 |

### 5.2 左侧街道排名

支持切换排序口径：

- 按巡查次数
- 按巡查覆盖率
- 按隐患数
- 按隐患整改率
- 按停工整改工程数

展示方式延续“实时统揽”的排行榜风格，但每一行支持点击后弹出右侧弹层。

### 5.3 右侧趋势分析

趋势图支持两个模式切换：

**模式一：工程状态趋势**
- 正常施工
- 现场整改
- 停工整改
- 已核销

**模式二：巡查执行趋势**
- 巡查次数
- 巡查覆盖率
- 隐患发现数
- 隐患整改率

趋势维度支持：

- 一周视图
- 一个月视图

### 5.4 街道明细表

建议字段如下：

- 街道名称
- 在建工程数
- 正常施工
- 现场整改
- 停工整改
- 已核销
- 本周巡查次数
- 本月巡查次数
- 巡查覆盖率
- 隐患数
- 隐患整改率
- 操作

操作列提供：

- 快览
- 详情

## 六、工程详情页信息设计

### 6.1 页面定位

工程详情页不是移动端页面的简单放大，而是大屏场景下的桌面化详情页。

因此会保留移动端已有的信息逻辑，但改成多栏布局，让信息密度更高、切换更少。

### 6.2 记录区结构

**备案记录**
- 备案编号
- 备案时间
- 工程类型
- 风险类别
- 施工单位
- 施工时间

**巡查记录**
- 巡查时间
- 巡查组织
- 巡查人员
- 隐患标签
- 当前处理状态
- 进入详情

**整改记录**
- 整改提交时间
- 整改状态
- 审核节点
- 审核结果
- 对应巡查记录

### 6.3 状态表达

工程详情页中需统一使用以下状态标签：

- 正常施工
- 现场整改
- 停工整改
- 已核销
- 待整改
- 整改中
- 已整改
- 已驳回

## 七、交互说明

### 7.1 页面跳转关系

```text
首页
  -> 点击巡查统揽
巡查统揽专题页
  -> 点击街道排行 / 表格行 / 地图点位
右侧弹层
  -> 点击查看详情
工程详情页
```

### 7.2 原型交互要求

- 所有页面在单个 HTML 文件中完成切换
- 使用按钮、Tab、筛选项、卡片点击完成页面切换
- 右侧弹层支持打开与关闭
- 趋势图支持“一周 / 一个月”和“状态趋势 / 巡查趋势”切换
- 明细表支持点击进入弹层或详情页

## 八、视觉与风格要求

### 8.1 必须延续的视觉语言

- 深色蓝绿大屏底色
- 霓虹青色边框与高光
- 现有标题装饰条和顶部科技感边框
- 现有统计卡的发光边缘和分栏结构
- 现有地图与面板的布局比例

### 8.2 不做的改动

- 不改现有大屏的整体主题色
- 不改四大统揽的基本位置关系
- 不改成扁平化后台样式
- 不引入与现有视觉冲突的暖色主背景
