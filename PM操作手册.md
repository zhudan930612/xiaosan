## 文档体系说明

### 产品上下文 - 业务知识库

**定位**: 变更频次低的业务背景知识

**包含内容**:
- 政策背景(产品基于的政策法规或业务需求)
- 用户角色(各类用户及其职责定义)
- 术语表(业务专业术语及定义)
- 核心业务流程(完整的业务流程描述)
- 模块清单(产品的功能模块列表)

**更新频率**: 低频(政策变更、新增模块时)

**使用场景**: AI设计功能时自动读取,提供业务背景

### 产品规划 - 功能清单

**定位**: 基于大模块拆分子功能,标注实施状态

**包含内容**:
- 已上线子功能(✅ 列表查询、在线申报等)
- 待规划子功能(❌ AI辅助审核、高级数据分析等)
- 待解决问题(⚠️ 业务规则待确认、技术方案待评估等)

**更新频率**: 中频(功能上线、规划调整时)

**使用场景**:
- PM查看功能实现现状
- AI设计功能时了解可复用的能力

### 功能描述文档 - 详细记录

**定位**: 已上线功能的详细页面、字段、流程

**包含内容**:
- 模块概述(业务场景、受众角色、业务价值)
- 通用规则(文件上传、字段校验、页面交互)
- 完整业务流程(流程图、详细步骤)
- 功能点描述(页面功能、字段规则、业务规则)

**更新频率**: 高频(功能上线、优化时)

**使用场景**:
- AI设计新功能时参考现有模式
- 开发人员查看实现细节

### 文档协作关系

三层文档通过自动同步机制保持一致,形成完整的知识循环:

```
PM记录功能 → 功能描述文档
     ↓ 使用 /sync 技能
产品上下文 + 产品规划 (自动同步更新)
     ↓ AI读取
设计新功能时有完整上下文
```

**工作原理**:

1. **记录阶段**: PM使用 `/write-feature-doc` 记录已上线功能,生成功能描述文档
2. **同步阶段**: 使用 `/sync` 技能后,AI自动提取功能信息,更新产品上下文和产品规划
3. **设计阶段**: 当PM使用 `/design` 设计新功能时,AI读取三层文档,获得完整的业务背景、已有功能、技术模式
4. **价值**: 确保AI设计新功能时,能基于真实的产品现状和业务背景,而不是凭空想象

## 日常使用场景

### 场景矩阵

| 场景 | 使用技能 | 输入示例 | 输出结果 |
|------|---------|---------|----|
| 从零开始搭建 | 手动创建 | "帮我创建产品上下文文档" | 引导整理业务背景→创建产品上下文→产品规划 |
| 记录已上线功能 | `/write-feature-doc` | "编写XX模块的XX功能文档" | 交互式收集信息→生成功能描述文档 |
| 设计新功能 | `/design` | "我要做XX功能" | 读取上下文→输出3个原型方案→生成设计文档→合并到功能描述文档 |
| 规划下一步 | `/plan-advice` | "下一步应该做什么功能" | 分析优先级→输出高/中/低优先级建议 |
| 同步文档 | `/sync` | "同步文档" | 自动更新产品上下文和产品规划 |

### 场景0: 从零开始搭建文档体系

**触发条件**: 新产品、新项目,还没有任何文档

**使用模板**: 参考现有的产品上下文和产品规划文档结构

**操作步骤**:

**第一步: 创建产品上下文**
1. 输入: "帮我创建产品上下文文档"
2. AI会引导你整理:
   - **政策背景**: 这个产品是基于什么政策或业务需求?
   - **商业模式**: 如何盈利?核心价值是什么?
   - **用户角色**: 有哪些用户?他们的职责是什么?
   - **术语表**: 业务中的专业术语定义
   - **核心业务流程**: 完整的业务流程是什么?
   - **模块清单**: 产品分为哪些大模块?
3. 逐步回答AI的问题,完成产品上下文文档

**第二步: 创建产品规划**
1. 输入: "帮我创建产品规划文档"
2. AI会基于产品上下文,引导你:
   - 将每个大模块拆分为子功能
   - 标注每个子功能的实施状态(已上线/待规划)
   - 记录待解决的问题
3. 完成产品规划文档

**第三步: 记录已上线功能**
- 如果有已上线功能,参考"场景1: 功能刚上线,需要记录"进行记录
- 记录完成后,使用 `/sync` 同步到产品上下文和产品规划

**注意事项**:
- 产品上下文是基础,必须先完成
- 产品规划基于产品上下文,需要第二步完成
- 功能描述文档是最后一步,记录具体实现

**搭建顺序**:
```
产品上下文 (第一步,业务背景)
    ↓
产品规划 (第二步,功能清单)
    ↓
功能描述文档 (第三步,详细记录)
```

### 场景1: 功能刚上线,需要记录

**触发条件**: 开发完成、测试通过、已部署生产

**使用技能**: `/write-feature-doc`

**操作步骤**:
1. 准备好功能截图(每个页面、每个状态)
2. 输入: "编写[模块名]的[功能名]功能文档"
3. AI会逐步询问:
   - 功能入口在哪里?
   - 页面有哪些功能区域?
   - 每个字段的规则是什么?
4. 每回答一个问题,AI立即写入文档
5. 完成后输入 `/sync` 同步到产品规划

**注意事项**:
- 只看截图说话,不要凭记忆
- 不确定的地方标注"待补充"
- 每个功能点完成后立即保存

### 场景2: 需要设计新功能

**触发条件**: 产品规划中有待开发功能

**使用技能**: `/design`

**操作步骤**:
1. 输入: "我要做[功能名]"
2. AI自动读取产品上下文、产品规划、功能描述文档
3. AI输出3个原型方案(A/B/C)
4. PM选择方案并细化
5. AI生成完整设计文档(业务流程、原型、功能点描述、验收标准)

**输出文档**: 功能设计文档(可直接交付开发)

### 场景3: 不知道下一步做什么

**触发条件**: 规划迷茫、优先级不清

**使用技能**: `/plan-advice`

**操作步骤**:
1. 输入: "下一步应该做什么功能"
2. AI分析产品规划、产品上下文、功能规划原则
3. AI输出:
   - 高优先级建议(政策合规、用户高频需求)
   - 中优先级建议(商业价值驱动)
   - 低优先级建议(技术创新)
   - 待解决问题建议
   - 已上线功能优化建议

### 场景4: 功能文档完成后

**触发条件**: 功能描述文档编写完成

**使用技能**: `/sync`

**操作步骤**:
1. 输入: `/sync`
2. AI自动:
   - 在产品上下文中添加新功能(如果是新模块)
   - 在产品规划中更新功能状态(待规划→已上线)
   - 保持文档体系一致性

## 技能使用指南

### write-feature-doc - 编写功能文档

**触发方式**:
- 输入: "编写功能文档" / "记录功能" / "文档化功能"
- 或直接输入: `/write-feature-doc`

**核心原则**:
- **只记录已上线功能**: 规划中的功能不记录
- **严格基于截图**: 不凭记忆,不推测
- **每完成一个功能点立即保存**: 逐步收集,实时写入
- **功能文档完成后使用 `/sync` 同步**: 保持文档体系一致

**工作流程**:
```
Step 1: 读取模板和上下文
  ↓
Step 2: 交互式收集信息
  - 功能入口路径
  - 页面功能区域
  - 字段规则
  - 业务规则
  ↓ 每收集一个功能点立即写入
Step 3: 基于已收集流程生成其他章节
  - 模块概述
  - 通用规则
  - 完整业务流程
  ↓
Step 4: 提示使用 /sync 同步文档
```

**关键特性**:
- **交互式工作模式**: 逐步收集,实时写入,不丢失进度
- **严格基于截图**: AI会提醒"只记录截图中的内容"
- **自动生成流程图**: 基于收集的功能点自动绘制流程图
- **支持中断恢复**: 可以随时退出,下次继续

**使用技巧**:
- 准备好所有截图,按流程顺序排列
- 每个页面截图要包含:页面标题、所有字段、操作按钮
- 不同状态的页面要分别截图(如:待审核、审核通过、审核不通过)
- 遇到不确定的地方,直接说"待补充",不要猜测

### feature-design - 功能设计

**触发方式**:
- 输入: "我要做XX功能" / "优化XX功能" / "设计XX功能"
- 或直接输入: `/design`

**工作流程**:
```
Step 1: 读取上下文
  - 产品上下文
  - 产品规划
  - 功能描述文档
  ↓
Step 2: 输出3个原型方案
  - 方案A: [方案名称]
  - 方案B: [方案名称]
  - 方案C: [方案名称]
  - AI推荐: [A/B/C] + 理由
  ↓
Step 3: PM选择并细化
  ↓
Step 4: 生成完整设计文档
  - 功能概述
  - 业务流程(流程图+业务规则)
  - 原型设计(PM迭代原型+交互说明)
  - 功能点描述(页面功能+字段规则)
  - 验收标准
```

**输出文档结构**:
```markdown
# 产品功能设计文档

## 文档信息
- 规划池记录ID
- 产品规划摘要
- 设计日期
- 状态: 🤖 AI草稿 → 🔄 PM迭代中 → ✅ 可交付开发

## 功能单元1: [功能名称]
### 功能概述
### 业务流程
### 原型设计
  - AI初稿方案(3个方案)
  - PM迭代原型
### 功能点描述
### 验收标准

## 附录: 合并指南
```

**关键特性**:
- **3个原型方案**: AI提供3种不同交互方式供选择
- **明确合并路径**: 设计文档中的"功能点描述"章节会合并到功能描述文档
- **设计依据透明**: 每个设计决策都会说明依据(来自产品上下文/产品规划/功能描述文档)

**使用技巧**:
- 先明确功能目标,再调用技能
- 仔细对比3个方案的优劣
- 选择方案后,可以要求AI细化或调整
- 设计文档完成后,标记状态为"✅ 可交付开发"

### planning-advice - 规划建议

**触发方式**:
- 输入: "下一步应该做什么功能"
- 或直接输入: `/plan-advice`

**工作流程**:
```
Step 1: 读取规划和上下文
  - 产品规划
  - 产品上下文
  - 功能描述文档
  ↓
Step 2: 分析优先级
  - 政策合规功能(必须做)
  - 用户高频需求(优先做)
  - 商业价值驱动(重点打磨)
  - 技术创新功能(适度投入)
  ↓
Step 3: 输出结构化建议
  - 高优先级建议
  - 中优先级建议
  - 低优先级建议
  - 待解决问题建议
  - 已上线功能优化建议
```

**输出格式**:
```markdown
## 高优先级建议(政策合规+用户高频需求)
1. [功能名称]
   - 优先级理由: [政策要求/用户痛点]
   - 当前状态: [待规划/部分上线]
   - 建议行动: [具体建议]

## 中优先级建议(商业价值驱动)
...

## 低优先级建议(技术创新)
...

## 待解决问题建议
...

## 已上线功能优化建议
...
```

**关键特性**:
- **基于功能规划原则**: 优先级判断遵循"政策合规 > 用户痛点 > 商业价值 > 技术创新"
- **结合实际情况**: 分析当前已上线功能和待规划功能
- **可操作建议**: 每个建议都包含具体行动

**使用技巧**:
- 定期(如每月)使用一次,保持规划清晰
- 结合团队资源情况调整优先级
- 将建议记录到产品规划文档的"待规划子功能"中

### doc-sync - 文档同步

**触发方式**:
- 输入: "同步文档" / "更新产品上下文" / "更新产品规划"
- 或直接输入: `/sync`

**工作流程**:
```
Step 1: 读取功能描述文档
  ↓
Step 2: 提取新增内容
  - 新增模块
  - 新增子功能
  - 新增业务规则
  ↓
Step 3: 同步到产品上下文
  - 添加新模块到模块清单
  - 更新核心业务流程
  ↓
Step 4: 同步到产品规划
  - 更新子功能状态(待规划→已上线)
  - 添加新的子功能
  ↓
Step 5: 保持文档一致性
```

**关键特性**:
- **只添加不删除**: 只添加新内容,不删除或修改已有内容
- **动态文档定位**: 自动搜索 `*产品上下文.md` 和 `*产品规划.md`
- **保持格式一致**: 严格遵循文档原有格式

**使用技巧**:
- 每次完成功能文档后立即同步
- 同步前检查功能描述文档是否完整
- 同步后检查产品规划中的功能状态是否正确

## 工作流程示例

### 案例: 记录"批量审核"功能

**背景**:
- 功能已上线
- 用户反馈审核效率低,希望支持批量审核
- 开发已完成,需要记录到文档

**Step 1: 准备截图**
- 列表页(勾选多个记录)
- 批量审核页(显示所有选中的记录)
- 审核结果页(显示批量审核结果)

**Step 2: 调用技能**
```
PM: 编写XX模块的批量审核功能文档
```

**Step 3: AI交互式收集**
```
AI: 我会帮你编写XX模块的批量审核功能文档。

首先,请告诉我:
1. 这个功能的入口在哪里?

PM: 列表页,勾选多个记录后,点击顶部的"批量审核"按钮

AI: 好的,我已记录。接下来:
2. 批量审核页面有哪些功能区域?

PM:
- 顶部显示选中的记录数量
- 中间是记录列表(显示关键信息)
- 底部有"全部通过"和"逐个审核"两个按钮

AI: 明白了。继续:
3. "全部通过"和"逐个审核"的区别是什么?

PM:
- 全部通过: 一键通过所有选中的记录,生成批量回执
- 逐个审核: 进入逐个审核页面,可以对每个记录单独审核

AI: 收到。最后:
4. 批量审核的业务规则有哪些?

PM:
- 只能批量审核"待审核"状态的记录
- 最多支持一次审核50条记录
- 批量审核后,所有记录状态变更为"审核通过"
- 生成批量回执,可以下载

AI: 好的,我已完成功能点描述的记录。现在我会基于这些信息生成完整的功能文档...

[AI自动生成模块概述、通用规则、流程图等章节]

AI: 功能文档已生成,保存在 `模块功能描述/XX模块.md`。
接下来,请使用 `/sync` 命令同步更新产品上下文和产品规划。
```

**Step 4: 同步文档**
```
PM: /sync

AI: 正在同步文档...
- ✅ 已在产品规划中更新"批量审核"功能状态为"已上线"
- ✅ 已在产品上下文中添加批量审核的业务规则
- ✅ 文档同步完成

PM: 谢谢!

AI: 不客气!如果后续需要设计批量审核的优化功能,可以使用 `/design` 命令。
```

**总结**:
- 耗时: 约10分钟
- 输出: 完整的功能描述文档 + 同步更新的产品上下文和产品规划
- 关键: 严格基于截图,逐步收集,实时写入

## 文档维护规范

### 功能描述文档维护

**更新时机**:
- 新功能上线
- 功能优化调整
- 发现文档遗漏或错误

**更新原则**:
- **只记录已上线功能**: 规划中的功能不记录
- **严格基于截图**: 不凭记忆,不推测
- **保持结构一致**: 遵循功能描述文档模板
- **及时标注待补充**: 不确定的地方明确标注

**更新流程**:
1. 准备截图
2. 使用 `/write-feature-doc` 技能
3. 逐步收集信息
4. 使用 `/sync` 同步文档

### 产品规划维护

**更新时机**:
- 功能上线(待规划→已上线)
- 新增规划功能
- 调整功能优先级
- 发现新的待解决问题

**更新方式**:
- **自动更新**: 使用 `/sync` 技能自动同步
- **手动更新**: 直接编辑产品规划文档

**更新内容**:
- 子功能状态(已上线/待规划/待解决)
- 功能说明、核心能力、能力边界

## 常见问题FAQ

**Q0: 新产品从零开始,应该先做什么?**

A: 按照以下顺序搭建文档体系:
1. **产品上下文**(第一步): 整理政策背景、用户角色、术语表、核心流程、模块清单
2. **产品规划**(第二步): 基于模块清单拆分子功能,标注实施状态
3. **功能描述文档**(第三步): 记录已上线功能的详细实现

不要跳过前两步直接记录功能,否则AI设计新功能时缺少业务背景。

**Q1: 功能还在规划中,可以先记录到功能描述文档吗?**

A: 不可以。功能描述文档只记录已上线功能。规划中的功能应该记录在产品规划文档的"待规划子功能"中。

**Q2: 截图不清晰,看不清某个字段,怎么办?**

A: 在文档中明确标注"待补充",不要猜测。后续重新截图后再补充。

**Q3: 功能上线后发现文档记录有误,如何修正?**

A: 直接编辑功能描述文档,修正错误内容,然后使用 `/sync` 同步到产品规划。

**Q4: 如何判断一个功能是"已上线"还是"部分上线"?**

A:
- 已上线: 功能完整可用,用户可以正常使用
- 部分上线: 功能核心流程可用,但部分细节待完善
- 待规划: 功能尚未开发

**Q5: AI生成的原型方案都不满意,可以自己设计吗?**

A: 可以。在功能设计文档的"PM迭代原型"章节,可以完全重新设计,不必基于AI方案。

**Q6: 功能设计文档和功能描述文档有什么区别?**

A:
- 功能设计文档: 设计阶段使用,包含原型方案、业务流程、验收标准,交付给开发
- 功能描述文档: 功能上线后使用,记录实际页面、字段、规则,供AI设计新功能时参考

**Q7: 如何知道下一步应该做什么功能?**

A: 使用 `/plan-advice` 技能,AI会基于功能规划原则给出优先级建议。

**Q8: 文档同步失败了,怎么办?**

A:
1. 检查功能描述文档是否完整
2. 检查产品上下文和产品规划文档是否存在
3. 手动编辑产品规划,更新功能状态

**Q9: 可以同时记录多个功能吗?**

A: 建议一次记录一个功能,完成后再记录下一个。这样可以保持专注,避免遗漏。

**Q10: 如何查看某个模块的所有功能?**

A: 打开产品规划文档,找到对应模块,查看"已上线子功能"和"待规划子功能"列表。
