# 禅道 Excel 导入小工具设计文档

> 目标：提供一个不需要编程的本地小工具。用户只需准备标准 Excel、输入项目名称、双击运行，即可批量在禅道中创建需求和任务。

## 背景

当前用户在禅道中的日常操作路径是：

1. 创建 Scrum 项目
2. 在项目下创建需求
3. 在项目下创建任务

用户不具备编程能力，不希望维护脚本、命令行参数或 Skill 工作流。工具应优先满足“最少操作即可完成导入”，而不是追求技术方案的抽象完备性。

## 目标用户

- 产品经理
- 项目经理
- 不具备开发能力、但需要频繁批量录入需求和任务的业务人员

## 设计结论

### 结论 1：交付形态选择

采用“本地双击运行工具”，不以 Skill 作为主交付物。

来源：基于用户明确反馈“不会编程”“给我最简单的”。

原因：

- Skill 适合熟悉 Codex 工作流的用户，不适合作为首要使用入口
- 双击运行的工具更符合用户心智
- 后续仍可补一个轻量 Skill，用于辅助触发该工具，但不作为第一版核心能力

### 结论 2：导入方式选择

第一版优先采用浏览器自动化方式录入禅道，而不是要求用户理解 API、Token、执行等概念。

来源：

- 基于用户实际页面操作路径
- 基于用户提供的项目截图
- 基于用户明确要求“建立 scrum 项目然后建需求、建任务”

原因：

- 浏览器自动化最贴近用户现有操作路径
- 页面行为与用户当前习惯一致，便于核对和排错
- 可避免第一版因接口模型差异导致用户理解成本上升

⚠️ 待确认：禅道当前版本页面 DOM 结构是否稳定；若页面升级频繁，后续可评估将底层录入逻辑替换为 API。

### 结论 3：输入文件格式

第一版只支持单个 Excel 宽表。

来源：基于用户提供的实际样例。

支持格式：

- 文件类型：`.xlsx`
- 默认读取第一个工作表
- 第一列表头固定为：`需求名称`
- 后续列头为人员姓名
- 单元格值为该人员在该需求下的工时

示例：

| 需求名称 | 曾凡圆 | 徐大圣 | 陈宇豪 | 蔡智旭 | 周宏亮 | 袁崇轩 | 王天虹 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 灾害信息员系统-系统监控\安全漏洞修复 | 4 | 4 |  |  |  |  |  |
| 小型工程-CMS-应用配置-备案指南 |  |  | 8 |  |  | 12 | 4 |

解析规则：

- 空白或 `0` 表示该人员不参与该需求
- 非空且大于 `0` 的值视为该人员任务预计工时
- 每一行生成 1 条需求
- 每一行中每个非空工时单元格生成 1 条任务

### 结论 4：需求与任务生成规则

来源：基于用户明确规则说明。

#### 需求生成规则

- 需求标题 = Excel 中的 `需求名称`
- 需求预计工时 = 该行所有人员工时求和

示例：

- 需求名称：`灾害信息员系统-系统监控\安全漏洞修复`
- 人员工时：`4 + 4`
- 导入结果：需求预计工时为 `8`

#### 任务生成规则

- 每个有工时的人员生成 1 个任务
- 任务标题格式固定为：`【角色】需求名称`
- 任务预计工时 = 该人员单元格工时
- 任务需要关联到当前导入创建的对应需求

示例：

- `【后端1】灾害信息员系统-系统监控\安全漏洞修复`
- `【后端2】灾害信息员系统-系统监控\安全漏洞修复`

### 结论 5：人员映射规则

第一版不再内置固定人员名单，改为读取用户可维护的“人员映射表”。

来源：基于用户新增反馈“这个用户人员名单可能会修改，这个工具可能要给其他组的用”。

推荐方案：使用独立 Excel 映射文件，而不是要求用户修改代码或 JSON。

原因：

- 更适合非技术用户维护
- 适合多个小组共用同一套工具
- 人员变动时只需要改映射表，不需要重新开发

映射文件建议命名为：`人员映射.xlsx`

建议字段：

| 人员姓名 | 角色 | 禅道显示名 | 是否启用 |
| --- | --- | --- | --- |
| 曾凡圆 | 后端1 | 曾凡圆 | 是 |
| 徐大圣 | 后端2 | 徐大圣 | 是 |
| 王天虹 | 测试 | 王天虹 | 是 |

规则：

- Excel 导入表中的列名，按“人员姓名”匹配映射表
- 任务标题中的 `【角色】` 来自映射表中的“角色”
- 任务指派默认优先使用映射表中的“禅道显示名”
- “禅道显示名”为空时，回退使用“人员姓名”进行匹配
- `是否启用=否` 的人员，不参与本次导入匹配
- 如果导入表出现映射表外的人名，导入前即报错并阻止正式提交
- 第一版随工具附带一份默认映射表，内容初始化为当前这组人员名单，但允许用户后续自行修改

### 结论 6：项目定位方式

工具运行时由用户输入项目名称，例如：`风险科技-Sprint178`。

来源：基于用户实际使用流程。

规则：

- 工具登录禅道后，按项目名称查找目标项目
- 若找到多个同名项目，则提示用户中止并手工调整
- 若找不到项目，则提示用户先在禅道中手工创建项目

不由第一版处理的能力：

- 自动创建项目
- 同时导入多个项目

## 交互方式

### 用户操作流程

1. 将 Excel 放到指定目录
2. 双击运行 `导入禅道.bat`
3. 输入项目名称
4. 工具先显示预览结果
5. 用户确认后开始正式导入
6. 导入完成后输出结果文件

### 预览阶段

预览阶段必须展示以下内容：

- 识别到的项目名称
- 需求总数
- 任务总数
- 每条需求的预计工时
- 每条任务的标题、负责人、工时
- 发现的异常项

预览阶段不允许真正写入禅道。

来源：基于“最简单但要安全”的设计原则。

## 工具组成

第一版交付以下文件：

- `导入禅道.bat`
- `zentao_import.py`
- `config.json`
- `人员映射.xlsx`
- `templates/禅道导入模板.xlsx`
- `README_禅道导入工具.md`
- `output/` 目录

各文件职责：

- `导入禅道.bat`
  - 双击入口
  - 调用 Python 脚本
- `zentao_import.py`
  - 读取 Excel
  - 读取人员映射表
  - 计算需求和任务
  - 执行浏览器自动化
  - 输出结果报告
- `config.json`
  - 保存禅道地址、账号、密码、默认文件路径
- `人员映射.xlsx`
  - 保存可编辑的人员姓名、角色、禅道显示名、启用状态
- `templates/禅道导入模板.xlsx`
  - 给用户复用的标准表
- `README_禅道导入工具.md`
  - 面向非技术用户的使用说明
- `output/`
  - 保存预览结果、导入日志、失败清单

## 导入流程设计

### 流程概览

```text
用户双击运行工具
  -> 读取 config.json
  -> 读取人员映射.xlsx
  -> 读取 Excel
  -> 校验表头和工时
  -> 生成需求与任务预览
  -> 用户确认继续
  -> 自动登录禅道
  -> 打开指定项目
  -> 逐条创建需求
  -> 记录“需求名称 -> 禅道需求标识”
  -> 逐条创建任务并关联需求
  -> 输出成功/失败报告
```

### 数据校验规则

导入前必须执行以下校验：

1. Excel 文件存在且可读取
2. 第一列必须为 `需求名称`
3. 所有人员列名必须存在于 `人员映射.xlsx` 且为启用状态
4. 工时必须为数字，且大于等于 `0`
5. 同一需求名称不能为空
6. 去除首尾空格后，需求名称不能重复

边界处理：

- 空行自动忽略
- 全员工时为空的需求行，提示异常并跳过
- 同一需求下只有 1 人参与时，需求工时即该人的工时

## 浏览器自动化设计

### 页面操作目标

工具应模拟用户现有路径：

1. 打开禅道登录页
2. 输入账号密码登录
3. 进入项目列表并定位目标项目
4. 在项目下进入“需求”页
5. 创建需求
6. 在项目下进入“任务”页
7. 创建任务并关联对应需求

### 选择浏览器自动化的要求

- 需要支持稳定等待页面加载
- 需要支持表单填写、下拉选择、页面跳转
- 需要支持失败截图留档

⚠️ 待确认：第一版所选自动化库，以实现时环境可用性为准。

## 失败处理设计

### 可恢复异常

- 单条需求创建失败
- 单条任务创建失败
- 单次页面超时
- 下拉列表未找到指派人

处理方式：

- 记录失败原因
- 继续处理后续记录
- 最终输出失败清单

### 不可恢复异常

- 登录失败
- 找不到目标项目
- 页面结构变化导致关键按钮不可定位
- Excel 结构不符合模板

处理方式：

- 立即终止
- 在终端和结果文件中给出明确提示

## 输出结果设计

导入完成后，工具至少输出两份结果：

1. 预览文件
2. 导入结果文件

导入结果文件需包含：

- 导入时间
- 目标项目名称
- 需求创建成功数/失败数
- 任务创建成功数/失败数
- 每条失败记录的需求名称、人员姓名、错误原因

## 第一版范围

第一版必须实现：

- 读取标准 Excel 宽表
- 读取人员映射表
- 自动计算需求工时
- 自动生成任务标题
- 支持预览确认
- 自动登录禅道
- 在指定项目中创建需求
- 在指定项目中创建任务
- 任务关联对应需求
- 输出导入结果

第一版不实现：

- 自动创建项目
- 更新已有需求或任务
- 删除误导入数据
- 支持多工作表
- 支持复杂字段映射
- 支持工时日志回填
- 支持多人同名自动消歧

## 风险与约束

### 风险 1：页面结构变动

来源：浏览器自动化依赖页面结构。

影响：

- 工具可能在禅道升级后失效

应对：

- 关键步骤失败自动截图
- 选择器集中管理，便于后续维护

### 风险 2：用户姓名与禅道显示名不一致

来源：任务指派依赖页面中的用户选项，且不同小组可能存在姓名差异。

影响：

- 工具可能找不到正确的负责人

应对：

- 第一版优先按映射表中的“禅道显示名”精确匹配
- 映射表未填写“禅道显示名”时，再使用“人员姓名”匹配
- 找不到时中止该条任务并记录失败

### 风险 3：重复导入

来源：用户可能重复运行同一份 Excel。

影响：

- 可能在禅道中创建重复需求和任务

应对：

- 第一版通过预览和结果报告降低误操作
- 第二版再评估增加“重复导入检测”

## 测试策略

第一版至少覆盖以下验证：

1. 使用用户提供的样例 Excel 做预览校验
2. 使用测试项目验证需求创建流程
3. 使用测试项目验证任务创建和关联需求流程
4. 使用非法表头验证拦截提示
5. 使用未知人员列名验证拦截提示
6. 修改人员映射表后验证不同小组可正常导入
7. 使用空工时、非数字工时验证拦截提示

## 待确认事项

1. 任务标题最终是否固定采用 `【角色】需求名称`
   - 当前理解：是
   - 来源：用户明确说明

2. 人员映射表是否需要支持额外字段，例如所属组别、备注
   - 当前理解：第一版先只支持姓名、角色、禅道显示名、是否启用
   - 状态：⚠️ 待确认

3. 创建任务时是否还需要填写开始时间、截止时间、任务类型等额外字段
   - 当前理解：第一版优先只填必填项和工时
   - 状态：⚠️ 待确认

4. 禅道当前版本在“项目任务创建页”中，关联需求字段的具体交互方式是否固定
   - 当前理解：页面可选关联需求
   - 状态：⚠️ 待确认

5. 浏览器自动化库选择
   - 当前理解：实现阶段按本机环境确定
   - 状态：⚠️ 待确认

## 结论

第一版不做复杂平台能力，不要求用户理解 API、脚本、Skill 或命令行，只交付一个面向非技术用户的双击运行小工具。

用户只需要：

1. 准备标准 Excel
2. 维护一份人员映射.xlsx
3. 输入项目名称
4. 确认预览
5. 等待导入结果

这符合当前用户能力边界，也更适合多个小组共用。
