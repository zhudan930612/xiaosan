## 一、功能背景

### 1.1 业务痛点

当前小型工程施工期间的安全监管存在以下问题:

1. **巡查记录不规范**:纸质记录易丢失,信息不完整,无法追溯
2. **隐患整改难跟踪**:发现隐患后缺少闭环管理,整改情况难以跟踪
3. **违规处置不及时**:发现违规行为后处置流程不清晰,容易推诿扯皮
4. **数据统计困难**:巡查数据分散,无法统计分析,难以支撑决策

### 1.2 解决方案

通过"安全巡检"功能,实现:

1. **规范巡查流程**:标准化巡查记录,现场拍照取证,数据实时上传
2. **闭环整改管理**:隐患登记→整改通知→整改提交→验收销号,全流程跟踪
3. **分级处置机制**:根据违规严重程度,自动触发不同处置流程
4. **数据可视化**:巡查数据实时统计,支持导出和分析

## 二、核心概念

### 2.1 工程状态定义

工程共有四种状态，状态流转关系如下：

```
正常施工 ──→ 现场整改 ──→ 正常施工
         └─→ 停工整改 ──→ 正常施工
         └─→ 已核销（终态）
```

---

#### 正常施工

**定义**：工程正常进行，无未处理的安全隐患。

**进入条件**：
- 工程初始状态（开工时默认为正常施工）
- 现场整改或停工整改的整改记录审核通过后，恢复为正常施工

**该状态下可执行的操作**：

| 角色 | 可执行操作 |
|------|-----------|
| 巡查人员 | 发起巡查、登记违规 |
| 巡查员 | 提交完工记录（触发核销） |
| 施工方 | 申请核销 |

**状态流转**：
- → **现场整改**：巡查登记违规，处置方式选择"现场整改"
- → **停工整改**：巡查登记违规，处置方式选择"停工整改"
- → **已核销**：监管提交完工记录，或施工方申请核销且监管审核通过

---

#### 现场整改

**定义**：巡查发现需要限时整改的违规，工程暂停施工，等待整改完成并通过审核。

**进入条件**：巡查登记违规时，"是否需要限时整改"选择"是"，处置方式选择"现场整改"。

**该状态下可执行的操作**：

| 角色 | 可执行操作 |
|------|-----------|
| 巡查人员 | 可继续对该工程发起新的巡查 |
| 施工方 | 提交整改记录（上传整改照片和整改说明） |
| 审核员 | 审核整改记录（通过/驳回） |
| 巡查员 | 查看整改进度、催办整改 |

**状态流转**：
- → **正常施工**：整改记录审核通过
- → **停工整改**：巡查发现更严重违规，处置方式升级为"停工整改"（新违规记录覆盖工程状态）

---

#### 停工整改

**定义**：巡查发现严重违规行为，工程强制停工，必须完成整改并通过审核后方可复工。

**进入条件**：巡查登记违规时，"是否需要限时整改"选择"是"，处置方式选择"停工整改"。

**该状态下可执行的操作**：

| 角色 | 可执行操作 |
|------|-----------|
| 巡查人员 | 可继续对该工程发起新的巡查 |
| 施工方 | 提交整改记录（上传整改照片和整改说明） |
| 审核员 | 审核整改记录（通过/驳回） |
| 巡查员 | 查看整改进度、催办整改 |

**状态流转**：
- → **正常施工**：整改记录审核通过

---

#### 已核销

**定义**：工程已完工，核销审核通过后进入该状态。已核销为终态，不可撤销，不可流转至其他状态。

**进入条件**（二选一）：
1. **巡查员主动核销**：巡查员提交完工记录，直接核销
2. **施工方申请核销**：施工方发起核销申请，巡查员审核通过后核销

**该状态下可执行的操作**：

| 角色 | 可执行操作 | 说明 |
|------|-----------|------|
| 所有角色 | 查看历史巡查记录 | 只读，不可修改 |
| 施工方 | 继续提交未完成的整改记录 | 用于完成未结束的整改闭环 |
| 巡查人员 | **不可**发起新的巡查 | — |
| 巡查员 | **不可**撤销核销状态 | — |

> **注意**：已核销工程仍可继续走完未结束的整改流程（提交整改记录→审核通过），但不产生新的巡查记录。

### 2.2 违规行为分类

根据是否需要限时整改,违规行为分为两类:

**不需要限时整改的违规**:
- 轻微违规,不影响施工安全
- 记录在案,用于后续查询和统计
- 不触发整改提交流程
- 工程状态不改变

**需要限时整改的违规**:
- 存在安全隐患,需要整改后才能继续施工
- 必须提交整改记录并通过审核
- 工程状态变更为"现场整改"或"停工整改"
- 整改通过后恢复"正常施工"状态

### 2.3 角色与权限

| 角色 | 说明 | 核心操作 |
|------|------|----------|
| 巡查人员 | 街道/社区/物业的巡查工作人员 | 发起巡查、登记违规 |
| 审核员 | 与发起巡查的组织**相同**的审核成员（街道/社区/物业各自内部审核） | 审核整改记录（通过/驳回） |
| 巡查员 | 上级监管部门工作人员 | 查看整改进度、催办整改 |
| 施工方 | 施工单位负责人或相关人员 | 查看违规记录、提交整改记录、申请核销 |

**审核规则**：整改审核由**同组织内部**相互审核，即：
- 街道巡查员发起的违规 → 街道审核员审核整改
- 社区巡查员发起的违规 → 社区审核员审核整改
- 物业巡查员发起的违规 → 物业审核员审核整改

### 2.4 巡查频次规则

系统根据工程风险等级和类型，计算每个工程的"应巡查频次"，超出频次要求的工程在**待巡查列表**中提醒，不强制阻断操作。

> **来源**：当前频次规则采用内部先行口径，基于深圳参考方案整理；后续如区级正式规则下发，再同步调整。

**频次规则**（当前版本）：

**高风险工程**：

| 组织 | 频次要求 |
|------|----------|
| 街道 | 每10天（含）内至少1条巡查记录 |
| 社区 | 每14天（含）内至少1条巡查记录 |
| 物业 | 每2天（含）内至少1条巡查记录 |

**非高风险工程 · 家装工程**（工程类型为"限额以下竣工后验收合格的室内住宅装修类型"）：

| 组织 | 频次要求 |
|------|----------|
| 街道 | 每30天（含）内至少1条巡查记录 |
| 社区 | 每21天（含）内至少1条巡查记录 |
| 物业 | 每7天（含）内至少1条巡查记录 |

**非高风险工程 · 非家装工程**：

| 组织 | 频次要求 |
|------|----------|
| 街道 | 每21天（含）内至少1条巡查记录 |
| 社区 | 每14天（含）内至少1条巡查记录 |
| 物业 | 每3天（含）内至少1条巡查记录 |

**系统行为**：
- 超出频次要求的工程显示在"待巡查"列表，并标注超期天数
- 仅作提醒，不阻断其他操作
- 待巡查数量在首页概览中统计展示

### 2.5 隐患分级与处置方式映射规则

> **来源**：隐患分级处置规则基于深圳参考方案与当前内部确认口径；高风险工程判定口径以 [高风险工程定义.md](./高风险工程定义.md) 为准。

巡查使用标准化《隐患清单》进行现场核查，每个检查项已预设隐患等级，等级直接映射处置方式：

| 隐患等级 | 定义 | 处置方式 | 对工程状态的影响 |
|---------|------|---------|----------------|
| **较大隐患** | 存在重大安全风险，需立即停工 | 停工整改（强制，不可修改） | 工程状态变为"停工整改" |
| **一般隐患** | 存在一般安全问题 | 由巡查员判断：现场整改 / 不需要限时整改 | 变为"现场整改"，或保持不变 |

**业务规则**：
- 巡查员选择**较大隐患**检查项后，处置方式自动锁定为"停工整改"，不可更改
- 巡查员选择**一般隐患**检查项后，可自行选择"现场整改"或"不需要限时整改"
- 同一次巡查登记多条隐患时，工程状态取最严级别（有一条较大隐患即停工整改）

> 完整隐患清单详见文末附录。

## 三、整体页面结构

```
首页（巡查概览）
  ├─ 当前身份栏（姓名 + 当前组织角色 + "切换身份"入口）
  ├─ 在建工程统计：小型工程 / 住宅装修 / 零星作业 各类型在建总数
  ├─ 待巡查数量（超出频次要求的工程数）
  ├─ 快捷功能区：待巡查列表 / 全部工程 / 整改跟踪
  ├─ 工程状态汇总：正常施工 / 现场整改 / 停工整改
  ├─ 我的巡查统计（当前用户）
  │     ├─ 本月：已完成巡查 / 发现违规 / 待跟进整改
  │     └─ 累计：巡查次数 / 违规条数
  └─ 最近巡查记录（近3条，含工程名、日期、结果、工程状态）

功能区
  ├─ 待巡查列表 → 工程列表（需尽快巡查）
  └─ 全部工程列表
        ├─ 区域选择栏（行政区域切换）
        ├─ 搜索栏（类型下拉 + 输入框 + 搜索按钮）
        ├─ 类型Tab（胶囊样式，含数量）：小型工程 / 住宅装修 / 零星作业
        ├─ 状态Tab（含数量徽章）：正常施工 / 现场整改 / 停工整改 / 已核销
        └─ 工程卡片（含风险标签 + 特殊施工标签 + 施工时间 + 组织统计）
              → 工程详情页
                    ├─ 简化版备案信息（工程名、地址、施工单位、施工时间、负责人）
                    │     └─ "查看完整备案" 按钮 → 完整备案详情页（只读）
                    ├─ 地图（显示工程位置，方便巡查员导航）
                    ├─ 巡查记录列表（多条，按巡查记录提交时间倒序）
                    │     └─ 巡查详情页
                    ├─ 违规整改记录列表（按违规条目展示，多条，按整改记录最新提交时间倒序）
                    │     └─ 整改详情页（含逐条整改进度）
                    └─ 操作按钮（根据角色和工程状态显示）
                          ├─ 发起巡查（巡查人员）
                          ├─ 提交整改记录（施工方，有待整改违规时）
                          ├─ 审核整改（审核员，有待审核整改时）
                          ├─ 申请核销（施工方，正常施工状态）
                          └─ 提交完工核销（巡查员，正常施工状态）
```

## 四、功能详细设计

### 4.1 首页概览

#### 4.1.1 页面结构

**页面名称**："安全巡检"首页

**导航栏**：标题"安全巡检"（左）

**页面模块（从上到下）**：

**① 当前身份栏**（导航栏下方，蓝色渐变底色）

| 元素 | 说明 |
|------|------|
| 头像 | 圆形，取姓名首字，白色字 |
| 姓名 | 如"张三" |
| 当前身份 | 格式："组织名 · 角色"，如"夏阳街道 · 巡查员" |
| 切换按钮 | "切换身份 ▾"，圆角胶囊样式，点击弹出身份选择弹层 |

切换规则：同一账号可绑定多个组织角色；切换后，首页所有数据模块均刷新为所选身份对应组织的数据视角。

---

**② 在建工程统计**（白色卡片）

标题行显示总数（如"在建工程统计（共 542 项）"），内容区横向三列：小型工程 / 住宅装修 / 零星作业。

**数据规则：**

| 项目 | 规则说明 |
|------|---------|
| 统计范围 | 当前身份所属组织负责监管的所有工程 |
| "在建"定义 | 备案审核状态 = `已通过` 且 工程状态 ≠ `已核销` |
| 分类依据 | 来自备案字段"备案类型"（小型工程 / 住宅装修 / 零星作业） |
| 合计数 | 三类数量之和 |
| 数据刷新时机 | 切换身份后立即刷新；页面首次加载时拉取 |

---

**③ 待巡查提醒大卡**（橙红渐变卡片）

左侧显示待巡查工程数量 + "个工程待巡查（超出频次要求）"，右侧有 ⓘ 图标。

**数据规则：**

| 项目 | 规则说明 |
|------|---------|
| 统计范围 | 当前身份所属组织负责监管的所有"在建"工程（定义同②） |
| "待巡查"判断 | `今天 - 该组织对该工程最后一次巡查日期` ＞ 该组织类型对该工程类型的频次阈值天数（见下表） |
| 从未巡查的工程 | 视为"距上次巡查天数 = ∞"，直接计入待巡查 |
| 频次阈值配置 | 见 4.1.3 频次说明弹层；高风险工程与非高风险工程阈值不同 |
| 高风险判定 | 见②工程卡片标签规则（4.2.2） |
| 计数口径 | 工程去重计数，一个工程只计一次（即使有多个违规） |

> **说明**：该数字仅提示巡查频次不足，与工程是否有违规、是否在整改无关。

---

**④ 快捷功能区**（横向三个入口，白色卡片样式）

| 入口 | 角标/红点来源 | 数量计算规则 |
|------|------------|------------|
| 全部工程 | 无角标 | — |
| 整改跟踪 | 红点 + 数字 | 当前身份所属组织监管的工程中，整改记录状态为`待整改`或`已驳回`的工程数（去重，一个工程多条违规只算一次） |
| 完工核销 | 红点 + 数字 | 当前用户（巡查员角色时）有权审核的待审核核销申请数；巡查员角色时此角标不显示 |

---

**⑤ 工程状态汇总**（白色卡片）

横向三列：正常施工 / 现场整改 / 停工整改（不含已核销）。

**数据规则：**

| 状态 | 判断规则 |
|------|---------|
| 正常施工 | 工程状态字段 = `正常施工`：备案已通过，且当前无未完成整改的违规记录 |
| 现场整改 | 工程状态字段 = `现场整改`：该工程有巡查结论为"现场整改"的违规记录，且对应整改记录未审核通过 |
| 停工整改 | 工程状态字段 = `停工整改`：该工程有巡查结论为"停工整改"的违规记录，且对应整改记录未审核通过 |
| 统计范围 | 当前身份所属组织监管的所有在建工程（不含已核销） |
| 状态优先级 | 同一工程若同时存在"停工整改"和"现场整改"违规，优先显示`停工整改` |

> **说明**：工程状态由系统根据违规记录整改情况自动维护，非人工设置字段。

---

**⑥ 我的巡查统计**（白色卡片）

标题右侧显示"本月（X月）"，本月指**当前自然月**（当月1日0:00 至今）。

**数据规则：**

| 字段 | 统计口径 |
|------|---------|
| 本月已完成巡查 | 当前用户提交的、巡查日期在本自然月内的巡查记录数（状态 = `已提交`） |
| 本月发现违规 | 上述本月巡查记录中，包含至少一条违规记录的巡查次数（按巡查维度统计，一次巡查发现多处违规仍算1次） |
| 待跟进整改 | 当前用户历史所有巡查记录中登记的违规条目，整改状态仍为`待整改`或`已驳回`的条数；包含本月以外的历史违规 |
| 累计巡查 | 当前用户账号创建以来所有已提交的巡查记录总数（不限时间） |
| 累计发现违规 | 当前用户账号创建以来，包含至少一条违规记录的巡查次数（按巡查维度统计，口径与"本月发现违规"一致） |

---

**⑦ 最近巡查记录**（白色卡片）

展示当前用户（当前身份）最近 **3 条**巡查记录，按**巡查记录提交时间倒序**排列。

**数据规则：**

| 字段 | 来源 |
|------|------|
| 工程名称 | 来自对应工程备案的"工程项目"字段 |
| 巡查提交时间 | 巡查记录的提交时间 |

### 4.2 工程列表页

#### 4.2.1 页面结构

**页面名称**："工程列表"页面

**页面功能**：

1. **区域选择栏**
   - 显示当前所选行政区域（如"南山区"），可点击切换
   - 位于页面顶部，方便跨区域巡查人员快速切换辖区

2. **搜索栏**
   - 搜索类型下拉（工程名称 / 工程地址，默认"工程名称"）
   - 文本输入框
   - "搜索"按钮
   - 右侧"筛选"按钮 → 触发高级筛选面板（详见 4.2.4）

3. **工程类型 Tab**（胶囊样式，3个，Tab 标题附带数量）
   - 小型工程(X) / 住宅装修(X) / 零星作业(X)
   - 数量 = 当前所有筛选条件下（区域、搜索/筛选条件）对应类型的工程总数
   - 来源字段：备案字段"备案类型"

4. **工程状态 Tab**（下划线样式，4个，Tab 标题附带数量徽章）
   - 正常施工 · X / 现场整改 · X / 停工整改 · X / 已核销 · X
   - 数量 = 在当前"工程类型 Tab"和搜索/筛选条件下，各状态的工程数

   **工程状态定义**（与首页⑤保持一致）：

   | 状态 | 判断规则 |
   |------|---------|
   | 正常施工 | 工程状态 = `正常施工`：备案已通过，当前无未完成整改的违规记录 |
   | 现场整改 | 工程状态 = `现场整改`：存在巡查结论为"现场整改"的违规记录，对应整改记录未审核通过 |
   | 停工整改 | 工程状态 = `停工整改`：存在巡查结论为"停工整改"的违规记录，对应整改记录未审核通过 |
   | 已核销 | 工程核销申请已审核通过 |
   | 状态优先级 | 停工整改 ＞ 现场整改 ＞ 正常施工；同一工程以最高级状态显示 |

5. **工程卡片列表**

   **基本信息区**：
   - 工程名称
   - 工程状态标签（正常施工 / 现场整改 / 停工整改 / 已核销）

   **风险与特殊施工标签行**（位于工程名下方）：
   - 风险等级标签（二选一，必显示）
     - 🔴 高风险：工程属于高风险的显示此标签（红色）
     - 一般项目：非高风险工程显示此标签（灰色）
   - 特殊施工标签（有才显示，可同时显示多个）
     - 🌙 夜间施工：备案中勾选了"夜间施工"时显示（紫色）
     - 📅 周末施工：备案中勾选了"周末施工"时显示（紫色）

   **地址与施工信息区**：
   - 📍 工程地址
   - 🏢 施工单位
   - 📅 施工时间（开始日期 ~ 结束日期）

   **三方巡查统计栏**（卡片底部，横向三列，分别对应街道 / 社区 / 物业）：

   每列格式：`[组织类型][巡查次数] / 隐患[X]`（第一行）+ `[X天/月]未巡查`（第二行）

   示例：
   ```
   街道1 / 隐患0    社区4 / 隐患1    物业8 / 隐患0
     5月未巡查        0天未巡查         7天未巡查
   ```

   **字段说明**：

   | 字段 | 计算规则 |
   |------|---------|
   | 巡查次数 | 该组织对该工程全部已提交的巡查记录数 |
  | 隐患数 | 该工程下，**该组织**登记的违规条目中，整改状态为`待整改`或`已驳回`的条数 |
   | 未巡查时长 | `今天` - `该组织对该工程最近一次巡查日期`；从未巡查显示"尚未巡查" |

   **显示规则**：

   | 条件 | 显示样式 |
   |------|---------|
   | 隐患数 = 0 | 绿色 |
   | 隐患数 > 0 | 红色加粗 |
   | 未巡查时长 < 30天 | 显示"X天未巡查" |
   | 未巡查时长 ≥ 30天 | 换算为月，显示"X月未巡查" |
   | 未巡查时长 > 该组织对该工程的频次阈值 | 标红 + ⚠ 图标（频次阈值见 4.1.3） |
   | 该组织从未巡查 | 巡查次数显示 0，未巡查时长显示"尚未巡查" |

   > **设计意图**：该栏不区分当前登录身份，始终展示三类组织的横向对比，方便任何角色快速判断哪个组织的巡查覆盖不足。

#### 4.2.2 标签规则说明

| 标签 | 来源字段 | 判断条件 | 颜色 |
|------|---------|---------|------|
| 🔴 高风险 | 备案字段`是否高风险工程` | 字段值 = `是`；该字段按 [高风险工程定义.md](./高风险工程定义.md) 自动判定 | 红色 |

| 📅 周末施工 | 备案"施工作业风险类别" | S11（周末施工）开关 = 开启 | 紫色 |

**说明**：
- 风险等级标签（高风险 / 一般项目）二选一，每张卡片必显示其中一个。
- 特殊施工标签（夜间施工 / 周末施工）仅在备案中对应开关开启时显示，可同时显示两个。

**触发条件**：在工程列表页点击工程卡片

**导航栏**：左侧返回箭头，右侧"···"更多菜单 + 扫码图标

**① 地图区域**（顶部，全宽）

- 显示工程位置地图（高德/百度地图组件）
- 地图上显示工程位置标注 + 工程名称气泡
- 可点击放大或跳转导航

**② 工程基本信息区**（地图下方，白色卡片）

| 字段 | 说明 |
|------|------|
| 工程名称 | 大字，加粗展示 |
| 工程状态标签 | 右侧悬浮标签，颜色规则同工程列表页 |
| 施工时间 | 开始日期 ~ 结束日期，右侧可能出现逾期提醒（见下） |

**③ 巡查与整改 Tab 区**（Tab 下划线样式，5个 Tab）

| Tab | 数量说明 |
|-----|---------|
| 街道巡查(X) | 街道人员对本工程提交的巡查记录 |
| 社区巡查(X) | 社区人员对本工程提交的巡查记录 |
| 物业巡查(X) | 物业人员对本工程提交的巡查记录 |
| 整改记录(X) | 施工方针对巡查记录提交的整改记录版本列表；一条巡查记录对应一条整改主记录，若驳回后重新提交，则形成多次提交版本，按整改记录最新提交时间倒序展示，最新提交排在最前面 |
| 备案 | 无数量，跳转或展示备案信息 |

> **计数口径**：各组织 Tab 上的数字 = 该组织对该工程**已提交的巡查记录总条数**（含有违规和无违规的记录均计入）。
>
> **排序口径**：街道/社区/物业巡查 Tab 下的巡查记录，统一按**巡查记录提交时间倒序**展示；整改记录 Tab 下的整改记录，统一按**整改记录最新提交时间倒序**展示。两类列表保持"最新提交在前"的一致原则，但排序字段不共用。

---

**④ 巡查记录卡片列表**（在街道/社区/物业 Tab 下显示）

每条卡片对应**一次巡查记录**，一次巡查可包含多条违规，展示以下内容：

**卡片左侧**：
- **隐患类型标签行**：本次巡查登记的**所有违规条目**的隐患类型（多个，胶囊标签样式），如"高处"、"用电"、"消防"；同类型可重复，代表该类型有多条隐患
- **巡查人信息**：姓名 + 组织名称（如"张三 夏阳街道办"）
- **提交时间**：巡查记录提交时间，格式 MM-DD HH:mm

**卡片右侧**：本次巡查记录对应的整改状态（每条巡查记录只对应一条整改记录，仅有一种状态）

**整改状态枚举**：

| 状态 | 显示文字 | 样式 |
|------|---------|------|
| 待整改（施工方尚未提交整改记录） | 待整改 › | 橙红色文字 |
| 待审核——当前用户与发起巡查的组织**相同**，且有审核权限 | 去审核 >> | 蓝色实心按钮 |
| 待审核——当前用户与发起巡查的组织**不同** | 待[发起巡查组织]审核 › | 蓝色文字 |
| 已驳回（审核驳回，施工方需重新提交） | 已驳回 › | 红色边框 + 红色文字 |
| 已整改（审核通过） | 已整改 › | 绿色文字 |
| 本次巡查无违规 | （不显示整改状态） | — |

**"待XX审核"组织名称填充规则**（当前用户非发起巡查组织时，"XX"取发起巡查的组织名称）：

| 当前登录身份 | 属于街道巡查 Tab 的卡片 | 属于社区巡查 Tab 的卡片 | 属于物业巡查 Tab 的卡片 |
|------------|----------------------|----------------------|----------------------|
| 街道身份 | 去审核 >>（有审核权）| 待社区审核 › | 待物业审核 › |
| 社区身份 | 待街道审核 › | 去审核 >>（有审核权）| 待物业审核 › |
| 物业身份 | 待街道审核 › | 待社区审核 › | 去审核 >>（有审核权）|

> **审核规则**：同组织才可审核——街道巡查发起的违规整改由街道审核员审核，社区同理，物业同理。跨组织用户仅可查看状态，无审核操作权限。

### 4.4 发起巡查

#### 4.4.1 页面结构

**页面名称**："发起巡查"页面

**触发条件**：在工程详情页点击"发起巡查"按钮

**页面功能**：

1. **工程信息**（只读，来自工程详情页）
   - 工程名称、工程地址（顶部展示条，不可修改）

2. **巡查信息**
   - 巡查时间（默认当前时间，可修改）
   - 巡查人员（默认当前登录用户，只读）
   - 是否施工中（单选：是 / 否）
   - 现场施工人员（可选择，支持多选）

3. **巡查结果选择**
   - 未发现违法违规（单选）
   - 发现违法违规行为（单选）

#### 4.4.2 交互说明

- 选择"未发现违法违规" → 直接提交，不改变工程状态
- 选择"发现违法违规行为" → 进入违规登记页面

### 4.5 违规登记

#### 4.5.1 页面结构

**页面名称**："违规登记"页面

**设计原则**：**每条违规独立登记**，一次巡查可添加多条违规记录，每条违规独立设置整改要求、期限和处置方式。一次巡查产生的所有需限时整改的违规，由施工方合并提交**一条整改记录**（包含多条违规各自的整改照片和说明），审核员对该整改记录**整体通过或整体驳回**；驳回后施工方重新提交整条整改记录。

**页面功能**：

**顶部工程信息条**（只读）
- 工程名称

**违规记录卡片列表**（可添加多条，每条为一个独立卡片）

每条违规卡片包含：

1. **隐患选择**（必填，从标准隐患清单中选择）
   - 按类别展示检查项（信息登记管理 / 安全管理 / 高处作业 / 施工用电 / 施工机具 / 消防安全 / 物体打击 / 文明施工 / 拆除作业 / 脚手架 / 吊篮作业 / 起重吊装作业 / 有限空间作业 / 物料提升机）
   - 选中后自动带出"隐患等级"标签（较大隐患 / 一般隐患）
   - 支持搜索关键字快速定位检查项
   - 可选"其他"并手动输入描述（手动输入默认为一般隐患）

2. **违规详情**
   - 违规描述（文本输入，限500字，选填，用于补充说明）
   - 现场照片（必传，最多9张）

3. **整改要求**
   - **较大隐患**：处置方式自动锁定为"停工整改"，不可更改；整改期限必填
   - **一般隐患**：需选择处置方式（单选）
     - 不需要限时整改 → 此条隐患不影响工程状态，仅记录在案
     - 现场整改 → 工程状态变为"现场整改"，整改期限必填
   - 整改要求说明（文本输入，限500字）

**底部操作**
- "＋ 继续添加违规" 按钮（添加下一条违规卡片）
- "提交违规登记" 按钮

#### 4.5.2 工程状态变更规则

同一次巡查中登记多条违规时，工程状态取**最严级别**：
- 有任意一条选择"停工整改" → 工程状态变为"停工整改"
- 全部为"现场整改" → 工程状态变为"现场整改"
- 全部为"不需要限时整改" → 工程状态不变

#### 4.5.3 交互说明

- 每条违规卡片可独立展开/折叠
- 提交后，本次巡查的所有违规合并为一条整改任务，通知施工方统一处理

### 4.6 整改提交（施工方）

#### 4.6.1 页面结构

**页面名称**："提交整改记录"页面

**触发条件**：施工方收到整改通知后进入，或在工程详情页点击"提交整改记录"按钮

**数据关系**：一条巡查记录对应一条整改记录（1:1）；整改记录内按违规条目分组展示。

**页面功能**：

**顶部巡查信息头**（只读，蓝色背景卡片）
- 巡查编号
- 巡查时间 / 工程名称
- 巡查员姓名 · 所属组织
- 工程当前状态提示（如"⚠ 停工整改，请完成整改后提交"）

**违规整改列表**（每条违规一个独立分区，不可删减）

每条违规分区包含两部分：

① **违规信息**（只读）
- 违规类型标签（颜色标注处置方式：停工整改/现场整改）
- 违规描述
- 整改要求说明
- 整改期限（标注剩余天数，逾期时标红）
- 隐患现场照片（可查看，不可修改）

② **整改内容填写**（施工方填写）
- 整改说明（文本输入，限500字，必填）
- 整改照片（必传，最多9张）

**底部操作**
- "提交整改记录（X条违规）" 按钮（一次提交全部，不可分条提交）

#### 4.6.2 交互说明

- 所有违规的整改内容填写完毕后，点击底部按钮一次性提交整条记录
- 提交后整改记录进入"待审核"状态，页面变为只读；同组织审核员收到通知
- 审核驳回后，施工方可修改**所有**违规的整改内容并重新提交整条记录
- 整改记录审核通过后，工程状态自动恢复"正常施工"

### 4.7 整改审核（审核员）

#### 4.7.1 页面结构

**页面名称**："审核整改记录"页面

**触发条件**：审核员在工程详情页点击"审核整改"按钮，或从整改跟踪列表进入

**权限规则**：只能审核**本组织**巡查员登记的违规对应的整改记录（街道审核街道，社区审核社区，物业审核物业）。

**页面功能**：

**顶部整改记录头**（只读）
- 巡查编号 / 巡查时间 / 工程名称
- 施工方提交时间
- 提示文字："驳回后施工方需重新提交完整整改记录"

**违规整改内容列表**（每条违规一个分区，顺序与巡查记录一致）

每条违规分区包含两部分：

① **原始违规信息**（只读）
- 违规类型标签
- 违规描述
- 整改要求 / 整改期限
- 隐患现场照片

② **施工方整改材料**（只读）
- 整改说明
- 整改照片

**验收材料**（通过时必填，位于所有违规条目下方）
- 验收照片（必传，最多9张）
- 验收说明（选填，限200字）

**底部操作**
- "驳回"（红色）→ 弹出驳回原因输入框（必填，限200字）→ 确认驳回
- "审核通过"（蓝色）→ 确认通过，工程状态自动恢复"正常施工"

#### 4.7.2 交互说明

- 审核为**整体操作**：整条整改记录整体通过或整体驳回，不支持部分通过
- 审核通过 → 整改记录标记为已整改，工程状态恢复"正常施工"，通知施工方
- 审核驳回 → 整改记录退回"待整改"状态，施工方收到驳回通知（含驳回原因），可修改后重新提交整条记录

### 4.8 整改跟踪

#### 4.8.1 页面结构

**页面名称**："整改跟踪"页面

**展示维度**：**按项目聚合展示**（每个工程一张卡片），方便巡查员核查时直接定位到具体工程。

**页面布局**（从上到下）：
1. 顶部导航栏：标题"整改跟踪" + 返回按钮
2. 搜索框：搜索工程名称或地址
3. Tab筛选栏
4. 工程整改卡片列表（滚动区域）

| 待审核 | 施工方已提交整改、等待审核员审核的工程 | 含"待审核"整改记录的工程数 |

**卡片列表排序**：逾期 > 待整改 > 待审核（紧急程度降序）

┌─────────────────────────────────────────┐

│                                          │

| 字段 | 说明 |

## 五、业务流程

### 5.1 完整流程图

```
发起巡查（从工程详情页进入，工程已确定）
  │
  ├─ 未发现违法违规 → 提交巡查记录 → 不改变工程状态 → 结束
  │
  └─ 发现违法违规行为 → 逐条登记违规（每条独立设置整改要求）
         │
         ├─ 含"不需要限时整改"的违规（可单独存在或与限时整改同时存在）
         │    └─ 记录在案，不影响工程状态，不触发整改流程
         │
         ├─ 含"现场整改"违规（无停工整改违规时）
         │    └─ 工程状态变更为"现场整改"
         │
         └─ 含"停工整改"违规（优先级最高）
               └─ 工程状态变更为"停工整改"

【整改流程（针对本条巡查记录中所有限时整改违规）】

巡查记录提交后 → 通知施工方
  │
  └─ 施工方提交整改记录（一条，包含每条限时整改违规各自的照片和说明）
         │
         └─ 同组织审核员审核（整体操作）
               │
               ├─ 整体通过（上传验收照片）
               │    └─ 工程状态恢复"正常施工" → 结束
               │
               └─ 整体驳回（填写驳回原因）
                    └─ 通知施工方 → 施工方修改后重新提交整条记录（循环）
```

### 5.2 整改逾期处理流程

```
整改期限到期（以整条整改记录的整改期限为准，取所有违规中最早的期限）
  │
  ├─ 施工方已提交整改记录 → 等待同组织审核员审核（不算逾期）
  │
  └─ 施工方未提交整改记录
       │
       ├─ 逾期1-3天 → 系统自动发送催办通知给施工方
       │
       ├─ 逾期3-7天 → 巡查员可手动催办
       │
       └─ 逾期后 → 巡查员可手动催办
```

### 5.3 工程核销流程图

```
工程正常施工状态（无未处理整改）
  │
  ├─ 路径一：监管主动核销
  │    └─ 巡查员提交完工记录 → 工程状态变更为"已核销"→ 结束
  │
  └─ 路径二：施工方申请核销
       └─ 施工方发起核销申请
            └─ 巡查员审核
                 ├─ 审核通过 → 工程状态变更为"已核销"→ 结束
                 └─ 审核驳回 → 通知施工方 → 结束

注：
- 已核销为终态，不可撤销
- 已核销工程存在未完成整改记录时，仍可继续提交整改→审核通过
- 已核销工程不可发起新的巡查
```

3. 审核驳回提示（含驳回原因输入）
4. 催办确认弹窗

## 七、技术实现要点

### 7.1 工程状态管理

- 工程状态字段：status（正常施工/现场整改/停工整改/已核销）
- 状态变更规则：取同一次巡查中所有违规的最严处置方式
- 状态变更记录：记录状态变更时间、变更原因、操作人
- 状态联动：工程状态变更后，自动通知相关方
- 已核销为终态：数据库层面限制已核销工程的状态字段不可被修改

### 7.2 违规逐条登记与整改主记录管理

- 每条违规作为独立记录（violation），含：违规类型、描述、照片、整改要求、期限、处置方式
- 每次巡查生成一条整改主记录（rectification），主记录下包含多条违规整改明细，含：施工方提交内容、审核结果、审核照片
- 工程状态由所有**未完成的限时整改违规**中最严级别决定
- 所有限时整改违规均审核通过 → 工程状态恢复正常施工

### 7.3 整改期限提醒

- 整改期限到期前1天:发送提醒通知给施工方
- 整改期限到期:标记为逾期,发送催办通知
- 逾期后:支持巡查员手动催办

### 7.4 照片上传与存储

- 支持拍照和相册选择
- 照片压缩后上传,减少流量消耗
- 照片存储在云端,支持查看和下载

### 7.5 消息通知

- 整改通知（按整改记录）：巡查记录提交后向施工方发送一次整改通知
- 审核结果通知（按整改记录）：整改记录审核完成后发送给施工方
- 催办通知：发送给施工方
- 逾期预警：发送给巡查员
- 待巡查提醒：发送给对应组织的巡查人员

### 7.6 审核权限控制

- 审核员只能审核**本组织巡查员发起**的整改记录
- 同一人不能既提交整改又审核同一条违规
- 权限与备案模块的组织归属字段联动
