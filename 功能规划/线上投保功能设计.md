## 设计范围

- 本文档仅讨论“备案申请 > 匹配保单”页面的前端交互改版。
- 本文档不确定保司最终接入方式，不写死外链、H5 内嵌、平台统一 API 的实现路径。
- 本文档不新增线下保单独立审核后台，不改变“保单随备案材料进入审核链路”的现状。
- 原型单独输出为 HTML 文件，类型为“文案可编辑、无标注”。

## 背景与目标

### 现状

- 当前备案申请的“匹配保单”页面只有线下上传窗口，用户对“上传保单”路径感知最强。
- 模块文档虽然已预留“线上投保/上传保单二选一”的能力描述，但实际关键交互流程未形成统一方案。
- 已有线上投保相关讨论主要集中在技术接入层，对“页面如何突出线上、弱化线下”缺少前端设计口径。

### 业务目标

来源：基于 [小散业务产品规划.md](../小散业务产品规划.md)、[小散业务产品上下文.md](../小散业务产品上下文.md)、[备案管理模块.md](../模块功能描述/备案管理模块.md)、[小散安责（上海）阶段性工作会-20260401_原文.docx](../事实资料池/其他会议资料/小散安责（上海）阶段性工作会-20260401_原文.docx)

- 让用户在“匹配保单”页第一眼先看到线上投保，而不是先看到上传框。
- 在不堵死线下上传入口的前提下，提高线上路径的优先级和可见度。
- 明确表达“线上更便捷”，但避免形成“平台强制指定购买渠道”的风险。
- 保留线下上传作为备案环节中的合法可用路径，避免投诉和合规争议。

### 已确认口径

来源：基于本轮用户确认

- 线上投保不需要用户额外填写其他资料，只需要将备案已填写资料传给保司。
- 当前仍处于方案讨论阶段，待保司合作方式谈妥后才进入开发。
- 线下保单当前仍由街镇审核人员在备案审核资料时一并审核，未确认单独的平台审核机制。

## 功能概述

本次设计的核心不是新增一条技术链路，而是重构“匹配保单”页面的入口层级与文案引导。页面需要把“线上投保”从一个未落地能力，调整为当前页面中的主推荐路径；同时保留“已有保单，直接上传”的线下入口，确保在保司方案未最终确定、用户已在线下投保、或业务不适合强推线上时，备案流程仍可继续。

## 业务流程

### 流程图

```text
用户完成基础信息/工程信息/安全信息
  -> 进入“匹配保单”
  -> 查看保单匹配说明
  -> 平台优先展示线上投保
      -> 用户选择线上投保
          -> 平台将备案已填资料传给保司
          -> 用户完成后续投保
          -> 返回备案继续提交
      -> 用户选择已有保单
          -> 进入上传保单
          -> 上传保单文件
          -> 随备案材料进入审核链路
  -> 提交备案申请
```

### 业务规则

触发条件
用户完成前三个 Tab 后进入“匹配保单”页。

处理逻辑
1. 页面默认突出线上投保，但必须保留线下上传入口。
2. 线上投保说明以“复用备案资料、减少重复填写”为核心卖点。
3. 线下上传入口不能被隐藏到难以发现，也不能仅在异常情况下出现。
4. 线上和线下都属于当前页面允许的合法路径，提交时不对路径本身做价值判断。

边界情况
- ⚠️ 待确认：若后续保司只支持外链跳转，页面状态文案是否需要区分“跳转中”与“返回继续备案”。
- ⚠️ 待确认：若某些辖区暂时未开通线上投保，是否按配置显示“推荐线上”或降级为并列展示。

## 方案对比

### 方案 A：同屏主次分明

来源：基于现有备案页结构、用户对“线下不能堵死”的确认

#### 方案说明

页面同时展示“线上投保”和“上传保单”两条路径。
线上投保放在主卡位，作为主推荐入口；上传保单放在下方副卡位，弱化但始终可见。

#### 页面结构

- 顶部保留压缩版政策说明卡片。
- 政策说明下方直接展示“推荐线上投保”主卡。
- 主卡下方展示“已有有效保单”副卡。
- 底部主按钮按当前选择路径变化。

#### 优点

- 风险最低，容易被业务、合规和审核端接受。
- 用户一眼能看到两种方式，不会误解为平台禁止线下。
- 适合当前保司接入方式未定的阶段。

#### 不足

- 线下入口仍持续可见，容易让用户沿用旧习惯直接上传保单。
- 只能做到“线上更显眼”，不一定能形成明显导流。

### 方案 B：默认线上，线下可切换

来源：基于本轮讨论结论，结合会议中“要控制投保流程、突出平台承接”的方向

#### 方案说明

页面默认只展开线上投保主流程，同时保留明确的线下切换入口，如“已有保单？直接上传”。
用户点击后切换为上传保单态，且需保留“改为线上投保”的回切入口。

#### 页面结构

- 顶部保留压缩版政策说明卡片。
- 页面默认展示“线上投保”主流程卡。
- 主流程卡下方展示线下切换入口。
- 切换后展示上传区域，并保留返回线上入口。
- 底部主按钮仅服务当前路径。

#### 优点

- 比方案 A 更能体现“线上优先”的产品态度。
- 又没有把线下入口隐藏成异常兜底，风险可控。
- 与“线上复用备案资料、减少重复填写”的业务事实更匹配。

#### 不足

- 对入口文案、回切逻辑、默认态表达要求更高。
- 若线下入口过弱，容易引发“被引导得过头”的感受。

### 方案 C：线上主流程，线下极弱化/后置兜底

来源：基于本轮讨论中对极致导流方案的延伸分析

#### 方案说明

页面几乎只展示线上投保，线下上传要么后置，要么仅在异常场景下露出。

#### 预期收益

- 平台对投保入口的控制力最强。
- 线上路径的视觉占比和导流力度最高。

#### 顾虑与风险

来源：基于本轮用户确认、[备案管理模块.md](../模块功能描述/备案管理模块.md) 中“上传保单需要审核人员人工核验保单的真实性和有效性”的现状描述、[会议原文](../事实资料池/其他会议资料/小散安责（上海）阶段性工作会-20260401_原文.docx) 中对投保流程控制的讨论

- 合规风险：
线下保单仍是备案环节中的现实合规路径。若前端将线下入口极度后置，容易被理解为平台变相限制用户提交已生效保单。

- 投诉风险：
用户已经在线下投保，却找不到明确上传入口，容易投诉“平台强推线上购买”。

- 审核主体风险：
若继续沿用“平台单独审核线下保单并卡 1~3 天”的思路，会引出“谁来审核、依据是什么、平台审核是否合规”的新问题。当前项目文档尚未定义独立的平台保单审核角色。

- 方案落地风险：
当前保司接入方式、收款责任边界、线上流程的技术形态都未定，前端若先做成“几乎只给线上”，会把后续合作的不确定性提前暴露给用户。

#### 结论

现阶段暂不采用方案 C。

原因
1. 该方案需要更清晰的合规边界和审核职责定义。
2. 该方案容易引发用户对“线下保单为何不能直接上传”的质疑。
3. 当前项目更适合先通过前端层级和流程便利性做引导，而不是通过封堵线下入口做导流。

## 推荐方案

来源：基于本轮讨论与用户确认

推荐采用方案 B，方案 A 作为保守备选，方案 C 作为已讨论但暂不采用方案保留在文档中。

推荐理由
1. 能清楚表达“线上优先”，但不堵死线下。
2. 与“线上无需重复填写资料”的业务事实最匹配。
3. 不依赖保司最终接入方式，适合当前阶段先完成前端方案确定。
4. 避免提前引入“平台是否单独审核线下保单”的争议。

## 线下路径软卡点设计

来源：基于本轮用户新增要求、[备案管理模块.md](../模块功能描述/备案管理模块.md) 中“上传保单需要审核人员人工核验保单的真实性和有效性”的现状描述、以及本轮关于合规与投诉风险的讨论结论

设计原则
1. 线下路径可以增加“软卡点”，但不能增加“硬拦截”。
2. 软卡点的目的应是再次提醒“线上更便捷”，而不是制造“线下不可走”的感受。
3. 所有软卡点尽量合并在一次弹窗内完成，不叠加多层弹窗。
4. 不写死“1~3 天”等具体时长，仅表述“可能影响备案审核时效”。
5. 不引入“需先尝试线上失败才能上传”的逻辑。

### 卡点 1：⚠️ 提示 + 二次确认

触发时机
用户点击“已有保单？直接上传”或“上传保单”入口时触发，不直接切换到上传态。

弹窗建议文案
线上投保
- 快捷出单
- 保单免审核
- 大平台有保障

上传保单
- 需人工核验真实性
- 可能影响备案时效

[ 仍要上传保单 ]
[ 改为线上投保 ]

效果说明
1. 能拦住一部分“习惯性直接上传”的用户，给线上路径一次明确回拉机会。
2. 能把“线上更便捷”的原因讲清楚，降低用户对平台弱化线下的敏感度。
3. 属于可解释的前端确认动作，合规风险低于隐藏入口或拖慢流程。

### 卡点 2：原因采集

触发时机
与二次确认弹窗合并呈现，用户决定继续走线下时顺手完成一次原因选择。

建议采集项
- 已线下购买保单
- 暂不选择线上投保
- 线下投保更优惠
- 其他

规则说明
原因采集仅保留单选，不增加长文本必填项。
默认不选中任何原因，用户需主动选择后才可继续上传保单。

效果说明
1. 为后续判断线下分流原因提供数据依据，便于和保司、业务继续讨论。
2. 帮助后续验证“是用户习惯问题，还是线上能力覆盖问题”。
3. 在不明显增加操作成本的前提下，形成一次用户路径留痕。

### 卡点 3：线上/线下路径轻对比

触发时机
不单独再加一层弹窗，而是与“⚠️ 提示 + 二次确认”合并在同一弹窗中展示。

展示原则
- 采用轻量说明式对比，不做营销海报式夸张对比。
- 线上强调“复用备案资料、无需重复填写、完成后继续备案”。
- 线下强调“适用于已完成投保、保单需人工核验”。
- 不出现“线下慢很多”“线下不推荐”等过强措辞。

效果说明
1. 比单句提示更容易让用户理解两条路径的真实差异。
2. 可以增强“改为线上投保”按钮的转化力。
3. 通过“说明差异”而非“攻击线下”的方式导流，更适合当前合规边界。

### 软卡点整体结论

推荐在方案 B 中采用“单层弹窗合并三项信息”的方式实现软卡点。

建议实现顺序
1. 先上线“⚠️ 提示 + 二次确认”。
2. 同时加上轻量原因采集。
3. 将线上/线下差异说明融入同一弹窗，不再额外叠加第二层比较页。

不建议采用
- 隐藏线下入口
- 强制先尝试线上失败后才能上传
- 人为设置未确认依据的审核时长承诺
- 通过增加过多字段和材料要求来故意提高线下操作成本

## 功能点描述

### 匹配保单页改版

入口路径
施工端 -> 备案申请 -> 匹配保单

页面名称
“匹配保单”页面

页面功能

1. 政策说明摘要区
- 将现有大段政策文案压缩为摘要卡片。
- 保留政策必要性，但不再占据页面主体视觉中心。

2. 线上投保主入口
- 作为页面主推荐区出现。
- 说明线上投保可复用备案资料，无需重复填写。
- 入口命名避免写死为某种技术形态。

3. 线下上传入口
- 作为合法保底路径保留。
- 文案强调“已有保单，直接上传”，而非与线上完全并列竞争。
- 用户点击线下入口时，先弹出单层软卡点弹窗，再决定是否进入上传态。
- 软卡点弹窗内同时承载风险提示、线上/线下差异说明和原因采集。

4. 底部主按钮
- 按当前路径变化文案。
- 同一状态只保留一个主操作按钮。

业务规则
- 不论方案 A 还是方案 B，线下上传入口都必须可达。
- 不允许仅在“线上失败”后才出现线下入口。
- 页面引导重点是“线上更便捷”，不是“线下不可用”。
- 当前不新增线下保单平台审核角色。
- 线下软卡点仅允许采用“提示、对比、原因采集”这类轻确认方式，不允许形成事实上的禁止上传。
- 线下风险提示可使用“可能影响备案审核时效”，但当前不写死具体时长。
- 原因采集建议为单选，且与二次确认合并在同一弹窗中完成。
