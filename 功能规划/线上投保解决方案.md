## 背景说明

### 现状
- 已上线：上传保单（线下投保）+ OCR识别
- 未上线：线上投保（保险公司入驻平台，用户直接选险购买）

### 需求
- 保险公司入驻平台，用户可直接选择保险公司投保付款
- 投保流程：选保险公司 → 填写投保信息 → 支付 → 保单数据自动回传平台
- 保费计算：等待保险公司方案，**大概率是动态报价**（基于工程信息实时计算）
- 保单关联：**同一工程仅关联1张有效保单**

## 3种技术方案对比

### 总览表

| 对比维度 | 方案A：外链跳转 | 方案B：H5内嵌（推荐） | 方案C：平台统一API |
|------|---------------|---------------------|------------------|
| **技术原理** | 跳转到保险公司外部投保页面 | 平台内打开保险公司H5页 | 平台直接调用保险公司API完成报价、投保、出单 |
| **平台开发量** | 低 | 中 | 高 |
| **预估周期** | 1-2周 | 4-6周 | 2-3个月 |
| **保险公司配合要求** | 低，只需提供投保链接 | 中，需提供H5并对接JS Bridge/回调 | 高，需开放标准API并配合联调 |
| **用户体验** | 一般，跳出平台 | 较好，留在平台内 | 最佳，全程平台内完成 |
| **保单回传能力** | 弱，依赖Webhook或补录 | 强，可自动回传 | 强，可自动回传 |
| **动态保费支持** | 保险公司页面自行计算 | H5页实时试算 | 平台直接调用报价API |
| **支付链路控制力** | 低 | 中 | 高 |
| **平台数据掌控度** | 低 | 中 | 高 |
| **异常单处理难度** | 高 | 中 | 中 |
| **后续扩展能力** | 低 | 中高 | 高 |
| **主要风险** | 回跳丢失、状态不一致、转化率低 | 多家保司适配成本不一致 | 技术复杂、合规要求高、周期长 |
| **适合阶段** | 快速试点 | 首期正式上线 | 长期平台化建设 |

### 决策维度说明

在与保险公司确认方案前，建议优先围绕以下 5 个维度评估：

| 维度 | 关注问题 | 对方案选择的影响 |
|------|---------|----------------|
| **保险公司配合深度** | 是否愿意提供H5或API？是否有成熟技术团队？ | 配合度低时更偏向方案A；配合度中等可落方案B；深度合作才适合方案C |
| **上线时效** | 是否需要快速形成可演示、可试点能力？ | 时间越紧，越应优先考虑方案A或方案B |
| **用户体验要求** | 是否希望投保全流程尽量不离开备案平台？ | 对体验要求越高，越倾向方案B/C |
| **平台运营诉求** | 是否希望沉淀订单、支付、转化等经营数据？ | 对平台掌控要求越高，越倾向方案C |
| **长期战略定位** | 线上投保是单点功能，还是未来要做保险能力平台？ | 只是先补齐流程可用方案B；要做能力中台则应规划方案C |

### 方案A：外链跳转模式

**流程**
```
用户选择保险公司 → 点击"立即投保" → 跳转至保险公司外部投保页
→ 用户在保险公司页面完成投保+支付 → 返回平台
  ├─ 保险公司支持Webhook：自动推送保单数据到平台
  └─ 保险公司不支持：用户手动确认或上传保单（降级）
```

**平台需做**
- 提供URL参数规范（传入工程信息）
- 可选：提供Webhook接收端点接收保单数据

**保险公司需做**
- 提供可访问的投保页面URL
- 可选：对接平台Webhook推送保单数据

**优点**
- 接入门槛最低，保险公司无需改造系统
- 平台开发量最小，1-2周可上线

**缺点**
- 用户体验割裂（离开平台）
- 若无Webhook，保单数据需用户手动确认
- 无法实时同步保单状态

**动态保费处理**
- 保险公司页面自行计算保费，平台不参与
- 列表展示保险公司提供的"参考价"或"¥xx起"

---

### 方案B：H5内嵌页模式（推荐）

**流程**
```
用户选择保险公司 → 平台打开内嵌WebView加载保险公司H5页
→ H5页携带工程信息，用户在内嵌页完成投保+支付
→ 支付成功后，H5调用JS Bridge通知平台
→ 保险公司服务端回调API推送保单数据
→ 平台自动关联保单
```

**平台需做**
- 定义H5页面参数规范
- 定义JS Bridge通信协议
- 提供保单回调API
- WebView容器开发

**保险公司需做**
- 提供适配移动端的H5投保页
- 接入平台JS Bridge协议（支付成功通知）
- 服务端对接保单回调API

**优点**
- 用户体验好，不离开平台
- 保单数据自动回传，闭环完整
- 支持动态保费（H5页实时试算）
- 开发成本适中

**缺点**
- 需定义接口标准
- 各家保险公司需做适配开发

**动态保费处理**
- H5页进入后，基于传入的工程信息实时试算保费
- 展示最终支付金额，用户确认后支付

---

### 方案C：平台统一投保API模式

**流程**
```
用户选择保险公司和险种 → 平台调用保险公司报价API获取保费
→ 展示保费信息，用户确认 → 平台发起投保请求（调用承保API）
→ 用户在平台内完成支付 → 保险公司出具保单，API返回保单信息
→ 平台自动关联保单
```

**平台需做**
- 建设保险公司API网关（聚合不同保险公司的API差异）
- 统一支付收款能力（需支付资质）
- 完整的投保UI流程（选险、报价、支付）

**保险公司需做**
- 开放标准化API（报价、承保、查询）
- 满足金融监管合规要求（销售资质、系统接入合规）

**优点**
- 体验最佳，全程在平台内完成
- 数据完全受控
- 支持后续理赔服务延伸

**缺点**
- 开发量最大
- 需要支付资质
- 保险公司API对接合规要求高
- 上线周期最长（2-3个月）

**动态保费处理**
- 平台实时调用保险公司报价API获取保费
- 支持多险种对比、价格锁定期等功能

## 关键问题说明

### 1. 保费计算方式（动态报价）

**结论**：首期按**动态报价**设计

| 场景 | 处理方式 |
|------|---------|
| 保险公司列表页 | 展示"¥xx起"或"参考价"（不作为最终价） |
| 投保确认页 | 基于工程信息（类型、地址、周期、规模）实时试算 |
| 不支持实时试算的保险公司 | 展示"价格待确认"，支付前提示"以实际报价为准" |

### 2. 同一工程多保单问题

**结论**：**一期不支持**，一个备案工程仅关联1张有效主保单

| 场景 | 规则 |
|------|------|
| 支付前切换保险公司 | 允许重新发起投保，以最后一次成功为准 |
| 已出单后更换保险公司 | 需先作废原保单，再重新投保 |
| 多险种需求 | 一期不支持主险+附加险模式，如需联合承保，后续扩展 |

## 推荐建议

| 场景 | 推荐方案 |
|------|---------|
| 快速上线、试探市场 | 方案A（外链跳转） |
| **首期正式接入（推荐）** | **方案B（H5内嵌）** |
| 深度合作、战略级投入 | 方案C（平台统一API） |

### 推荐意见

**当前推荐方案：方案B（H5内嵌）**

推荐理由：
- 能满足“保险公司入驻平台、用户在平台内完成线上投保”的核心诉求
- 比方案A更完整地承接投保结果、保单回传和状态闭环
- 比方案C更现实，不要求首期就完成统一API和支付中台建设
- 适合当前“合作模式未完全确定、但要尽快推进线上投保”的阶段

**不建议首期直接上方案C**：
- 当前赔付方式、报价规则、支付归属都还未完全确定
- 如果现在直接按方案C推进，容易因为保险公司接口和合规条件不成熟导致项目周期失控
- 方案C更适合作为二期升级目标，而不是首期落地方案

**方案A的定位**：
- 可以作为保险公司技术配合不足时的兜底方案
- 适合先跑通商务合作和基础投保闭环
- 但不建议作为长期主方案

### 决策建议

建议按“两步走”推进：

1. **先决策首期上线方案**
   - 建议首期以方案B为目标方案
   - 若某家保险公司短期无法提供H5，仅能提供投保链接，则允许该保司临时按方案A接入

2. **再预留二期升级路径**
   - 首期先统一工程信息传参规范、保单回调规范、保单绑定规则
   - 等保险公司合作模式、支付规则、报价规则稳定后，再评估是否升级到方案C

建议的实际落地路径：

| 阶段 | 建议动作 | 目标 |
|------|---------|------|
| **第一阶段** | 输出统一接入标准（工程参数、回调字段、鉴权方式） | 让不同保险公司至少能按A/B方案接入 |
| **第二阶段** | 首家保险公司按方案B接入上线 | 形成真实闭环样板 |
| **第三阶段** | 补充后台配置能力（辖区开关、是否强制、展示哪些保司） | 支撑按街道/社区差异化投保 |
| **第四阶段** | 评估是否建设统一API中台 | 决定是否升级方案C |

### 当前阶段是否需要原型

当前阶段**不建议先画详细原型**，原因是：
- 现在的核心问题是方案选型，不是页面交互细化
- 保险公司接入方式、支付链路、动态报价规则都还未最终确定
- 如果先画原型，后续方案变化时大概率需要整体重画

当前阶段建议保留：
- 方案流程图
- 方案对比表
- 接口和回传规则说明

待首期方案明确后，再补页面原型会更高效。
